# Anatomy of a Thinking Discipline

A thinking discipline has two anatomies: what it IS (the spec) and what it PRODUCES (the output). Both are universal — every discipline follows both structures regardless of which cognitive operation it formalizes.

---

## Spec Anatomy — What a Discipline IS

From `what_are_they.md`: every discipline spec contains:

1. **Definition / Philosophy** — what cognitive operation this formalizes, what its transform is (input state → output state), what it is NOT
2. **Structural Components** — the discipline-specific internal parts (anchors for sensemaking, mechanisms for innovation, dimensions for critique, depth hierarchy for comprehend, etc.)
3. **Process Model** — the sequence of phases or steps the discipline follows
4. **Failure Modes** — predictable ways the discipline fails, with recognition signals and corrective actions
5. **Coverage Strategy** — how you know the discipline's work is sufficient, when to stop

The spec anatomy describes the discipline as a formalization — its theory, its structure, its process, and its quality guarantees.

---

## Output Anatomy — What a Discipline PRODUCES

Every discipline's runtime output has four layers:

| Layer | What it is | Discipline-specific? |
|---|---|---|
| **Transform** | The core cognitive output — the discipline's reason for existing | Yes — unique per discipline |
| **Progression** | Versioned snapshots showing how the work evolved | Yes — unique version names per discipline |
| **Telemetry** | Quantitative measurements about the run's quality | No — generic structure, discipline-specific metrics |
| **Frontier** | Questions the discipline raised but didn't answer | No — generic structure, discipline-specific questions |

### Transform

The primary output. What the discipline exists to produce. Each discipline's transform is unique:

- Sensemaking → stable conceptual structure (SV6)
- Innovation → tested novel ideas (survivors)
- Critique → fitness landscape with positioned candidates and verdicts
- Comprehend → internal working model with predictive power (CV5)
- Exploration → structural map with confidence levels
- Decomposition → question tree with pieces, interfaces, and dependency ordering
- Wayfinding → steering directive (one of six moves with parameters)

### Progression

Versioned snapshots that show how the work evolved from initial state to final output. Each discipline has its own version naming:

- Sensemaking: SV1 → SV6 (Sense Versions)
- Comprehend: CV1 → CV5 (Comprehension Versions)
- Innovation: Seed → Generate → Test (phase outputs)
- Critique: Phase 0 → Phase 4 (dimension construction through convergence)
- Exploration: Cycle 1 → Cycle N (iterative scan-probe cycles)
- Decomposition: Step 1 → Step 7 (perceive through evaluate)
- Wayfinding: Read → Integrate → Ask → Produce (checkpoint cycle)

Progression serves two purposes: it makes the discipline's reasoning TRACEABLE (how did you get here?) and it enables CROSS-SESSION RESUME (pick up from the last saved version).

### Telemetry

Quantitative measurements about the run's quality. Telemetry is NOT evaluation — it's reporting. The discipline reports numbers about what it did. The routing decision (proceed, flag, re-run) is made by inquiry based on these numbers, not by the discipline itself.

Telemetry is generic in structure but discipline-specific in content:

- Sensemaking: perspective count, new anchor types per perspective, ambiguity resolution ratio, SV delta
- Innovation: generators applied, framers applied, convergence signal, survivors tested
- Critique: dimensions evaluated, adversarial strength, landscape stability, clean SURVIVE existence
- Comprehend: prediction accuracy at target depth, adversarial cases attempted, confidence map
- Exploration: frontier state, discovery rate, convergence criteria status
- Decomposition: independence, completeness, reassembly (self-evaluation dimensions)
- Wayfinding: checkpoint frequency, move consistency, layer conflict detection

Telemetry is what makes automation possible — inquiry reads telemetry and routes without human judgment at every transition.

### Frontier

Questions the discipline raised but didn't answer — coverage gaps expressed as questions. Frontier questions are the most valuable single output for directing NEXT STEPS: they tell you exactly what to investigate further.

Frontier is generic in structure but discipline-specific in content:

- Sensemaking: perspectives not checked, ambiguities flagged OPEN, anchors that need external validation
- Innovation: mechanisms not applied, seeds not explored, convergence not tested
- Critique: dimensions not evaluated, regions of landscape unexplored, candidates not tested
- Comprehend: depth levels not reached, components not traced, predictions not tested
- Exploration: regions not scanned, signals deferred, gaps unbounded
- Decomposition: pieces not validated, interfaces not specified, dependencies not ordered
- Wayfinding: layers not read, gates not assessed, reconsideration flags pending

Frontier questions grow as the discipline goes deeper — deeper understanding reveals more structure to question. A growing frontier is a signal of depth, not a signal of failure.

## The Relationship

The spec anatomy and output anatomy are complementary:

- The **spec** says WHAT the discipline is and HOW it works
- The **output** says WHAT the discipline produced and HOW WELL it worked

The spec is written once (the discipline definition). The output is produced every time the discipline runs. The spec is for discipline builders. The output is for discipline users and for inquiry (the loop runner).

---

## Universal Standards

When building a new discipline:

**The spec must contain:** definition, structural components, process model, failure modes, coverage strategy.

**The output must contain:** transform (core cognitive work), progression (versioned evolution), telemetry (operational metrics), frontier (unanswered questions).

Both are required for a discipline to be complete. A discipline with a spec but no output standard produces inconsistent outputs. A discipline with output structure but no spec is a procedure without theory.
