# Critique: Is GATE Just Critique in Disguise?

## Verdict: KILL separate GATE protocol. REFINE into Gate Mode of Critique.

### The user was right

GATE and Critique have the same structural operation: extract criteria → evaluate → produce verdict. Creating a separate "GATE protocol" with flat PASS/FLAG/FAIL is both redundant (duplicates Critique) and a downgrade (loses dimensional information).

### The resolution

Critique gains a second operational mode:

**Full Mode** (existing):
- When: between iterations, end of SIC cycle
- Weight: heavy — adversarial prosecution/defense, landscape, convergence
- Output: SURVIVE / REFINE / KILL per candidate, across dimensions
- Purpose: which solution is best?

**Gate Mode** (new):
- When: between disciplines within an iteration
- Weight: light — criteria check only, no adversarial, no landscape
- Output: PASS / FLAG / FAIL per quality dimension (NOT flat — per dimension)
- Purpose: is this output minimally viable to proceed?
- Criteria: pre-extracted from the discipline's own spec

### Why this is better than separate GATE

1. No redundant protocol — reuses Critique's machinery
2. Dimensional verdicts propagate forward — "sensemaking was strong on ambiguity but thin on diversity" informs the next discipline
3. Gate Mode criteria already exist in discipline specs — no invention needed
4. Strictness still calibrated by SCOPE (loose for exploratory, strict for critical)

### Revised automation flow

```
CONFIGURE → discipline 1 → Critique[GATE] ✓ → discipline 2 → Critique[GATE] ✓ → discipline 3 → Critique[FULL] → STEER → [human]
```
