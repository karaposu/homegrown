### Probe 2 (R4) — /intuit's placement in the taxonomy

/intuit has characteristics that don't cleanly fit Core / Boundary / Situational:

- **Not pure Core:** Core disciplines are pipeline-sequential (E → S → D → I → C). /intuit is multi-location — invokable at multiple points.
- **Not pure Boundary:** Boundary disciplines operate between cycles; /intuit operates WITHIN disciplines and between them.
- **Not pure Situational:** Situational disciplines are invoked as needed for specific purposes; /intuit operates as background substrate that other disciplines draw from.

**Three options:**
1. Force into existing category (Situational) — imperfect fit but minimal architectural change
2. Create a new category ("Substrate" or "Cross-cutting") — adds taxonomy complexity
3. Relax the category definitions — Core disciplines can be multi-location IF they satisfy other Core criteria (rigor, spec quality, primitive profile)

**Depth finding:** /intuit FITS BEST as a fourth category: "Cross-cutting" or "Substrate" disciplines — those invocable across the pipeline, producing inputs to other disciplines. This is a taxonomy refinement, not a reorganization of existing disciplines.

**Alternative framing:** /intuit IS Core (same tier as E/S/D/I/C) but with different invocation rules. The Core category gets a sub-classification: "pipeline-sequential" vs "cross-cutting."

Both framings preserve the existing disciplines. The question is labeling.

### Probe 4 (R8) — Pipeline shape re-evaluation

Current pipeline: `E → S → D → I → C`

**Questions:**

**Answers:**

- **Risk of shallower exploration:** pre-seeded attention could narrow the scan. Mitigation: /explore's "jump scan" principle (Failure Mode 3: False Confidence) explicitly requires exploring regions different from prior attention. If /explore respects this, narrow-seeding doesn't damage completeness.

**Depth finding:** the pipeline shape changes from sequential linear to "intuition-framed sequential." The CHANGE is minor at the discipline-level (add /intuit before /explore) but SIGNIFICANT at the practice-level (every inquiry starts with recognition of what's prior). This doesn't reorganize disciplines; it adds one at the start.

### Probe 5 (R3) — Situational discipline coherence

## Cycle 4 — Jump Scans

Deliberate scans in different directions.

### Jump 1 — "What does a COMPLETELY DIFFERENT discipline architecture look like?"

If we re-designed from scratch given the new understanding, would we produce the same set?

Hypothetical reset: organize by primitive-type consumption.
- Simulation-heavy disciplines: Innovate, Decompose (piece-interaction), Sensemaking (perspective run)
- Evaluation-heavy disciplines: Critique, Wayfinding (direction selection)
- Intuition-similarity-heavy disciplines: Intuit, parts of Explore
- Metacognition-heavy disciplines: Reflect, parts of Critique

Insight: this grouping cuts across the existing Core/Boundary/Situational axis. Each existing discipline is a BUNDLE of primitive usage; reorganizing by primitive type would disaggregate working bundles.

**Finding:** the existing disciplines are PRIMITIVE-COMPOSITES that happen to map onto useful units of cognitive work. Each is a stable combination of primitives that humans (and LLMs) actually use together. Disaggregating by primitive type would produce disciplines that don't match actual cognitive practice. **Conservation verdict reinforced.**

### Jump 2 — "What about the Comprehension/Foundation disciplines (arch-* commands)?"

These are lifecycle-specific (understanding a codebase) rather than cognitive primitives. They use the Core disciplines internally. Orthogonal to this audit's scope. **Confirmed out of scope.**

### Jump 3 — "Are there genuinely NEW cognitive operations the primitive typology surfaces?"

Walking through each admitted primitive:
- Working Memory → not a discipline; a buffer; no discipline needed
- Attention-pointer → not a discipline; an operation; no discipline needed
- Focus-deep → not a discipline; an operation; no discipline needed
- Intuition-similarity → `/intuit` (discipline-level application)
- Inhibition → operates within disciplines (critique's KILL; exploration's deferred signals)

**Finding:** no primitive surfaces a genuinely new discipline-level operation. Every primitive is either implemented within an existing discipline or within /intuit. The primitives are atoms; disciplines are compounds. **Conservation verdict reinforced.**

### Jump 4 — "What if the pipeline itself is wrong?"

Is E → S → D → I → C the right cognitive sequence?

Alternative sequences considered:
- S → E → D → I → C: sensemaking first. Requires something to sense-make over — but that something comes from exploration. Backward.
- E → D → S → I → C: decompose before sense-making. But decomposition requires clarified whole — which is sensemaking output. Backward.
- E → S → I → C → D: innovate before decompose. Innovation without partitioned problem produces undirected candidates. Weaker.
- Parallel E+S with later D: exploration and sensemaking could overlap. But sensemaking operates on explored territory; temporal dependence is real.

**Finding:** the sequence is structurally right. Dependencies force the order. Alternatives produce worse or equivalent orderings. **Conservation verdict reinforced on pipeline.**

### Jump 5 — "Does the new understanding reveal that boundary disciplines need restructuring?"

R (reflect-backward) and N (navigate-forward) are the two temporal directions at cycle boundaries.

With the three-layer architecture, a new temporal dimension exists: L2 retrospective (T2+ → T4). Does THIS surface a new boundary discipline?

Candidate: "L2 boundary discipline" — operates between far-apart cycles (T2+ horizon), observing how completed findings fare over time.

But /intuit's Phase D already includes Baldwin seed generation (patterns across many inquiries over time). That IS the L2-boundary work.

**Finding:** no new boundary discipline needed. The L2 scope is owned by /intuit's calibration layer. **Conservation verdict reinforced.**

### Jump 6 — "What about the /MVL+ meta-discipline?"

/MVL+ orchestrates the core pipeline. It's a meta-discipline — operates ABOVE disciplines. Does the new understanding change /MVL+?

Changes that may be warranted:
- /MVL+ auto-invokes /intuit at inquiry creation (pipeline-early opt-in → default-on after calibration)
- /MVL+ documents the typed primitive set as an available concept for inquiries
- /MVL+ handles the IMPACTED_BY relationship (from the inquiry chain)

These are operational refinements to /MVL+, not restructurings. **Keep /MVL+ as-is with minor refinements when /intuit ships.**
