## 2. Fitness Landscape

### Viable Region

The viable region contains approaches that:

- keep MVL/MVL+ as the reasoning engine;
- use a protocol or template only to shape diagnostic MVL+;
- store diagnostic evidence as findings;
- use git branches for real fundamentals variants;
- require evaluation before branch selection;
- can start manually.

### Dead Region

The dead region contains approaches that:

- create a new discipline before there is evidence for one;
- rely on git history alone as semantic maintenance memory;
- rely on Markdown docs alone for executable variants;
- automate branch selection before criteria exist.

### Boundary Region

Boundary candidates are useful but incomplete:

- diagnostics folder without lifecycle state;
- protocol without storage convention;
- git branch experiments without evaluation corpus;
- lifecycle index before enough records exist.

### Unexplored Region

The main unexplored region is a future branch-selection fitness function. It is intentionally left open because the project needs real branch experiments before it can define reliable selection criteria.

## 3. Candidate Verdicts

### A. Full `loop-diagnose` Discipline Now

**Prosecution:** This violates atomic-loop coherence. The current evidence shows diagnostic comparison can be performed by MVL+ itself.

**Defense:** Failure attribution may eventually develop its own stable concepts and failure modes.

**Collision:** Defense is plausible later, but not now. A full discipline would harden an operation before enough examples exist.

**Verdict:** REFINE / DEFER.

Constructive output: revisit discipline status after 5 to 10 diagnostic MVL+ findings show stable repeated internal structure that is not merely ordinary MVL+.

### B. `loop-diagnose` Protocol That Runs MVL+

**Prosecution:** A protocol still adds another named thing and may create confusion if written poorly.

**Defense:** The protocol preserves MVL+ as the thinker while standardizing diagnostic inputs and outputs.

**Collision:** Defense wins if the protocol is explicitly framed as a recipe, not a discipline.

**Verdict:** SURVIVE.

Constructive output: implement only as an input/output protocol for diagnostic MVL+.

### C. Diagnostics Folder as Ledger

**Prosecution:** A folder of findings can become hard to scan if it lacks lifecycle state.

**Defense:** The pattern already exists, keeps evidence complete, and uses the normal MVL+ artifact lifecycle.

**Collision:** Defense wins for near-term storage. Prosecution requires adding lightweight metadata or an index as volume grows.

**Verdict:** SURVIVE WITH CAVEAT.

Constructive output: use `devdocs/inquiries/diagnostics/` now; add index/frontmatter after three diagnostic findings or one active branch experiment.

### D. Git-Only Self-Maintenance

**Prosecution:** Git records file changes, not full causal diagnosis, risk class, evaluation gate, or output-comparison rationale.

**Defense:** Git is already present and excellent for branches, diffs, merges, and rollback.

**Collision:** Defense proves git is necessary, not sufficient. It cannot replace semantic maintenance records.

**Verdict:** KILL as complete approach.

Extracted seed: use git as the variant substrate and link branches to diagnostic/evaluation docs.

### E. Markdown-Only Self-Maintenance

**Prosecution:** Docs cannot isolate executable variants or safely compare branch-specific runtime behavior.

**Defense:** Findings are excellent for semantic evidence and are easy for agents to read.

**Collision:** Defense proves docs are necessary, not sufficient. Source variants need git.

**Verdict:** KILL as complete approach.

Extracted seed: keep findings as evidence records tracked by git.

### F. Git-Backed Maintenance Experiments

**Prosecution:** This can become complex quickly. It needs evaluation cases, branch naming, branch lifecycle, and comparison rules.

**Defense:** It is the first architecture that makes the user's evolutionary-branch idea concrete and testable.

**Collision:** Defense wins as next-stage architecture, not as the very first manual step.

**Verdict:** SURVIVE AS NEXT STAGE.

Constructive output: define branch experiments after diagnostic MVL+ has produced maintenance candidates that actually require source changes.

### G. Distributed Ledger Plus Small Index

**Prosecution:** An index can become another maintenance burden.

**Defense:** The index can stay minimal and only track lifecycle state; evidence remains in findings.

**Collision:** Defense wins if the index is gated by need.

**Verdict:** SURVIVE.

Constructive output: start with diagnostic finding frontmatter; add an index only after three diagnostics or one active branch.

### H. Evolutionary Meta-Loop Now

**Prosecution:** The selection criteria are not ready. Automating branch creation and killing would multiply uncertainty.

**Defense:** It points toward the user's most ambitious and coherent future direction.

**Collision:** Defense wins conceptually, prosecution wins on timing.

**Verdict:** REFINE / DEFER.

Constructive output: design after branch experiments produce enough comparison records.

### I. Staged Hybrid Architecture

**Prosecution:** It has multiple layers, so it could become too procedural if over-applied.

**Defense:** Each layer has a distinct job and the staging prevents premature complexity. It answers the user's correction directly.

**Collision:** Defense survives. The prosecution is handled by gates: no index until volume needs it, no branch until source changes need it, no automation until evaluation exists.

**Verdict:** SURVIVE.

Constructive output: use this as the corrected architecture.

## 4. Assembly Check

The strongest assembly is Candidate I:

```text
diagnostic MVL+
+ loop_diagnose protocol/template
+ devdocs/inquiries/diagnostics/ records
+ optional lifecycle metadata/index
+ git-backed branch experiments
+ evaluation MVL+ for baseline-vs-candidate comparison
+ later meta-loop branch selection
```

This assembly survives because it resolves the main false dichotomies:

- It is not `loop-diagnose` versus MVL+. `loop-diagnose` is MVL+ used diagnostically.
- It is not git versus ledger. Git stores variants; findings store evidence.
- It is not one ledger file versus no ledger. Diagnostic findings can be ledger records, with a small lifecycle index only when useful.
- It is not branch evolution now versus never. Branch evolution is valid, but selection criteria must come first.

## 7. Final Critical Judgment

The user is right on the main correction: `loop-diagnose` should not be an individual discipline at this stage. It should be a diagnostic MVL+ use case with a protocol/template.

The user is also right that git branches are a powerful self-maintenance substrate. But git is not enough by itself, because it does not store the full semantic evidence needed for selection.

The best architecture is git-backed evolution with Markdown-native diagnostic and evaluation findings.
