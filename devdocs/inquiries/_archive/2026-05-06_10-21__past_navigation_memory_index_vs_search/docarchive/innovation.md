# Innovation: PastNavigationMemoryFile Index Versus Search

## User Input

If Homegrown can find `PastNavigationMemoryFile` candidates by searching known filenames and regex-searchable patterns, does it still need a maintained index?

## Seed

Seed type: dissatisfaction + inversion.

The dissatisfaction: a maintained `PastNavigationMemoryFile Index` may be stale bookkeeping if the filesystem is already searchable.

The inversion: instead of making writers remember to register route-memory files, make Navigation intake derive candidate memory files from the active tree when it needs them.

## Mechanism Outputs

### 1. Lens Shifting

Generic variation:

- Treat the index as a convenience list.

Focused variation:

- Treat the filesystem as the source of truth and the search pattern as the explicit artifact.

Contrarian variation:

- Treat search as unsafe unless it produces a durable report.

Surviving output:

The v1 frame should be search-contract-first. Durable reports are output mode, not the baseline mechanism.

### 2. Combination

Generic variation:

- Combine search with Route Memory Review `Sources Read`.

Focused variation:

- Combine deterministic search, candidate filtering, and optional generated report:

```text
search patterns
  -> candidate list
  -> included/excluded/needs-read statuses
  -> optional past_navigation_memory_candidates.md
  -> Route Memory Review
```

Contrarian variation:

- Combine maintained index and search by requiring both to agree before review proceeds.

Surviving output:

The search/report/review pipeline survives. Requiring index-search agreement is killed because it creates two discovery authorities and a new reconciliation burden.

### 3. Inversion

Generic variation:

- Instead of "create an index so we do not search," search so we do not create an index.

Focused variation:

- Instead of registering files at creation time, validate route-memory presence at consumption time.

Contrarian variation:

- Never save search results; always regenerate.

Surviving output:

Consumption-time discovery is stronger for v1 because it cannot go stale before use. But durable search results should be saved when another session needs to trust what was scanned.

### 4. Constraint Manipulation

Generic variation:

- Add constraint: search only active docs.

Focused variation:

- Add constraints:
  - exclude archives by default;
  - search `devdocs/`, not `homegrown/`, unless a source explicitly points elsewhere;
  - route-memory discovery returns candidates, not truth;
  - no global mutable registry in v1.

Contrarian variation:

- Remove all constraints and grep the whole repository for `navigation`.

Surviving output:

The constrained search survives. Whole-repo loose grep is killed because it overmatches specs.

### 5. Absence Recognition

Generic variation:

- Missing object: search command.

Focused variation:

- Missing protocol section: `PastNavigationMemoryFile Discovery Search`.

Contrarian variation:

- Missing object: generated global index regenerated by a script.

Surviving output:

The missing protocol section is the immediate need. Generated global index is deferred until tooling is justified.

### 6. Domain Transfer

Generic variation:

- Transfer from compiler/build systems: derived artifacts are regenerated, not hand-maintained.

Focused variation:

- Transfer from test discovery: tests are found by naming convention at run time, not usually registered manually in a global list.

Contrarian variation:

- Transfer from package registries: everything must be registered before consumers can find it.

Surviving output:

Test-discovery/build-artifact logic fits better than package-registry logic. Route-memory candidate files are local repository artifacts with predictable names, so discovery can run at consumption time.

### 7. Extrapolation

Generic variation:

- As artifact count grows, search may return many candidates.

Focused variation:

- If route-memory artifacts grow large, a generated report can summarize the search result for one review run.

Contrarian variation:

- If artifact names drift, search will miss files and a maintained index may become necessary.

Surviving output:

Search-first remains v1. Add explicit revival triggers for index or generated global cache when artifact count, naming drift, or search time creates real pain.

## Candidate Designs

### Candidate A - Maintained Global Index

Shape:

```text
devdocs/navigation_context/past_navigation_memory_file_index.md
```

Strength:

- Easy to browse.
- Feels explicit.

Weakness:

- Duplicates derivable filesystem state.
- Requires every writer to update it.
- Can silently become stale.

Test:

- Novelty: low-medium.
- Scrutiny survival: weak under the user's search objection.
- Fertility: medium.
- Actionability: medium.
- Mechanism independence: weak.

Verdict: kill as v1 default; preserve as deferred option.

Shape:


Strength:

- Cheap.

Weakness:

Test:

- Novelty: low.

- Mechanism independence: weak.

## Assembly Check

Strongest assembly:

```text
PastNavigationMemoryFile Discovery Search
  + scoped active-doc patterns

  + optional run-local candidate report
```

## Proposed V1 Shape

### Protocol Name

```text
PastNavigationMemoryFile Discovery Search
```

### Default Search Scope

```text
Search active devdocs.
Exclude docarchive/, archive/, and _archive/ by default.
Do not search homegrown/ for route-memory candidates unless the user or source explicitly names a Homegrown artifact as route memory.
```

### Default Candidate Patterns

```text
devdocs/navigation/**/*.md
**/navigation.md
**/_frontier.md
**/route_memory_review.md
**/prior_map_overlay.md
**/sync_brief.md
```

### Candidate Statuses

```text
candidate
excluded
needs_read
review_consumed
```

Meanings:

- `candidate`: likely contains past Navigation memory.
- `excluded`: matched mechanically but is not route memory for a named reason.
- `needs_read`: may contain route memory but must be read before classification.
- `review_consumed`: already covered by a current applicable Route Memory Review.

### Output Modes

```text
inline_status
route_memory_review_sources
save_candidate_report
```

Use `inline_status` for small same-session checks.

Use `route_memory_review_sources` when Route Memory Review runs and can record what it searched/read directly.

Use `save_candidate_report` when the search result must survive beyond the current context, when the scan is broad, or when excluded matches matter for audit.

### Maintained Index Revival Triggers

Revive a maintained or generated global index only if one of these happens:

- three or more route-memory files are missed by the documented search;
- route-memory artifacts start using nonstandard names that cannot be normalized;
- a future runner cannot execute repository search and needs a precomputed list;
- users repeatedly need a browseable cross-run catalog, not just per-run candidate reports;
- search cost becomes material after at least ten durable route-memory artifacts exist.
