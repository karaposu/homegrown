# Innovation: Navigation Context Intake Replacement Or Warm-Up Folder

## User Input

Decide whether to remove `navigation_context_intake.md`, make it use warm-up v1/v2/v3, or create `homegrown/protocols/navigation/` and put v1/v2/v3 there.

## Seed

The system needs one clear Navigator warm-up authority without losing the old protocol's safety invariants.

## Mechanism 1 - Lens Shifting

### Generic

View `navigation_context_intake.md` as a compatibility wrapper, not a protocol.

### Focused

The main question becomes: "What must still be true around warm-up execution?" not "Which file contains all instructions?"

### Contrarian

If a file is too heavy to use, preserving it unchanged is worse than deletion because it silently steers future agents into the wrong workflow.

## Mechanism 2 - Combination

### Generic

Combine:

```text
Navigation discipline ownership + v1/v2/v3 command routine + tiny protocol invariants
```

### Focused

Resulting structure:

```text
homegrown/navigation/warmup/
  navigator-warmup1.md
  navigator-warmup2.md
  navigator-warmup3.md

homegrown/protocols/navigation_context_intake.md
  tiny wrapper pointing to the warm-up routine
```

### Contrarian

Keep root files temporarily as aliases until references are updated, then remove them.

## Mechanism 3 - Inversion

### Generic

Instead of asking where to put warm-up under protocols, ask: "What should protocols NOT own?"

### Focused

Protocols should not own detailed discipline-support execution when the support routine has a clear discipline consumer.

### Contrarian

If warm-up eventually becomes useful outside Navigation, extract only the cross-cutting invariant wrapper to protocols, not the detailed files.

## Mechanism 4 - Constraint Manipulation

### Generic

Add a constraint: no two files may define the detailed warm-up procedure.

### Focused

Add a constraint: `navigation_context_intake.md` must stay under 100 lines if retained.

### Contrarian

Add a constraint: do not move files until v3 is cleaned, because otherwise the canonical folder would contain an unfinished command.

## Mechanism 5 - Absence Recognition

### Generic

Missing artifact: a tiny `README.md` or `index.md` in the warm-up folder that states run order and session boundary.

### Focused

Missing transition artifact: a "superseded by warm-up routine" note inside `navigation_context_intake.md`.

### Contrarian

Missing check: a root-level file move can break user's current habit. The move should happen with clear path announcement, not silently.

## Mechanism 6 - Domain Transfer

### Generic

From software package structure: place helper routines under the module that consumes them.

### Focused

Navigation consumes warm-up, so warm-up belongs under `homegrown/navigation/`, the same way module-local utilities live near the module.

### Contrarian

From API deprecation: old names should forward to new names for one transition period rather than disappear immediately.

## Mechanism 7 - Extrapolation

### Generic

If Homegrown adds more Navigation support files, `homegrown/protocols/navigation/` will become a second Navigation namespace and confuse readers.

### Focused

If warm-up grows, `homegrown/navigation/warmup/` scales naturally:

```text
warmup/
  navigator-warmup1.md
  navigator-warmup2.md
  navigator-warmup3.md
  README.md
  examples/
```

### Contrarian

If future runners need machine-readable warm-up contracts, add a small protocol file later. Do not organize today's human-run commands around a future runner that does not exist yet.

## Assembly Check

Best assembly:

```text
1. Finish cleaning navigator-warmup3.
2. Move v1/v2/v3 into homegrown/navigation/warmup/.
3. Add a short warmup README or index in that folder.
4. Rewrite navigation_context_intake.md as a tiny wrapper:
   - warm-up runs once per session;
   - execute v1 -> v2 -> v3;
   - Navigation consumes outputs;
   - rerun only on boundary/staleness/missing-context failure;
   - review after first real use.
5. After one real use, decide whether the wrapper should be renamed or removed.
```

This assembly avoids duplicate authority and avoids the confusing `homegrown/protocols/navigation/` path.
