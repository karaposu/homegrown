# Innovation: Protocol Priority Top 5

## User Input

`devdocs/inquiries/2026-04-28_07-35__protocol_priority_top_5/_branch.md`

## Seed

Homegrown needs the next most useful protocols based on current goals and state. The core tension is between immediate self-maintenance evidence, quality enforcement, git-backed branch evolution, and future meta-loop autonomy.

## Direction / Valuation

The useful ranking should help the user decide what to work on next. It should avoid naming glamorous future protocols ahead of the current evidence substrate.

## Mechanism Outputs

### 1. Lens Shifting

#### Generic - Rank by current workflow pain

If the frame is current pain, the top protocol is `loop_diagnose`. The user already reruns weak MVL+ findings with corrections; the missing piece is a repeatable way to learn from that correction chain.

**Status:** Survives.

#### Focused - Rank by autonomy ladder prerequisites

If the frame is autonomy prerequisites, the top protocol is quality awareness: structural checks, telemetry, and retrospective outcome review. A system that cannot evaluate its own output cannot safely self-maintain.

**Status:** Survives as enforcement + review priorities.

#### Contrarian - Rank by future meta-loop architecture

If the frame is the future whirl/meta-loop, the top protocol would be meaningful traversal. It tells a runner whether it is thinking or spinning.

**Status:** Refine. Strategically important, but it needs traces and outcomes first.

### 2. Combination

#### Generic - Combine diagnosis and branch experiment

`loop_diagnose` plus `maintenance_branch_experiment` creates a full path from correction evidence to candidate source variant.

**Status:** Strong survivor.

#### Focused - Combine structural checks and telemetry

A single priority cluster can cover both deterministic artifact checks and standardized verdict telemetry. They are different mechanisms, but they share the same purpose: make outputs routeable and checkable.

**Status:** Survives with implementation split.

#### Contrarian - Combine Navigation selection and retrospective review

Selection records could be treated as a subtype of retrospective outcome review: pick an option, later ask whether it was useful.

**Status:** Partial. It is conceptually clean, but keeping Navigation selection separate makes meta-loop data easier to collect.

### 3. Inversion

#### Generic - What if no new protocol is needed?

Maybe the answer should be "do not add protocols; use MVL+ manually." This protects against over-protocolization.

**Status:** Killed as complete answer. Manual MVL+ is enough for one-off reasoning, but the current goal is improving fundamentals over time. That needs repeatable evidence contracts.

#### Focused - What if git replaces the ledger?

Git could be the self-maintenance substrate. Branches, commits, diffs, and merges already record changes.

**Status:** Killed as complete answer. Git records executable state, not semantic diagnosis, evaluation criteria, or branch superiority.

#### Contrarian - What if Navigation should be first?

Navigation could show all next moves and help choose what protocol to build.

**Status:** Refined. Navigation maps options, but it does not create the maintenance evidence needed to judge future changes.

### 4. Constraint Manipulation

#### Generic - Add "must be buildable in one small pass"

Under this constraint, `loop_diagnose`, structural check protocol, and navigation selection record rise. Full meaningful traversal falls because it is still fuzzy.

**Status:** Survives.

#### Focused - Add "must unlock multiple future capabilities"

Under this constraint, structural/telemetry contract rises because it unlocks Resume, Primitive RC, routeable discipline outputs, and future quality gates.

**Status:** Strong survivor.

#### Contrarian - Add "must not require code"

Under this constraint, `loop_diagnose`, branch experiment template, retrospective review, and selection record can start as Markdown protocols. Structural checker weakens because real value requires tooling.

**Status:** Refined. The protocol can start as Markdown, but the missing tool must eventually be implemented.

### 5. Absence Recognition

#### Generic - Missing failure-to-improvement bridge

The system lacks a protocol that transforms a known bad run plus correction into a maintenance candidate.

**Status:** Strong survivor: `loop_diagnose`.

#### Focused - Missing experiment lifecycle

The system lacks a protocol for candidate branches: why branch exists, what changed, what evaluation proves, and when to merge/revert.

**Status:** Strong survivor: `maintenance_branch_experiment`.

#### Contrarian - Missing protocol inventory hygiene

The protocol folder itself has active/dormant/legacy ambiguity. `resume.md` and `unknown.md` show this.

**Status:** Useful but not top five. It is cleanup, not the main learning engine.

### 6. Domain Transfer

#### Generic - Transfer from scientific method

Diagnosis is hypothesis generation; branch experiment is intervention; retrospective review is outcome measurement. The top protocols should mirror this scientific cycle.

**Status:** Strong survivor.

#### Focused - Transfer from CI/CD

Structural checks and telemetry are like CI gates. A pipeline should not rely only on human eyeballing when required sections or verdicts are missing.

**Status:** Strong survivor.

#### Contrarian - Transfer from product analytics

Navigation selection records are like event logs. Before building recommendation/selection algorithms, collect events: options shown, choice made, reason, outcome.

**Status:** Survives as top-five candidate.

### 7. Extrapolation

#### Generic - If inquiry volume grows

Manual memory will fail. The system will need diagnostic records, branch links, structural checks, and retrospective decisions.

**Status:** Survives.

#### Focused - If meta-loop starts being used

The system will need selection traces and meaningful traversal signals. But selection traces come first because they are the data meaningful traversal learns from.

**Status:** Survives: selection record outranks formal meaningful traversal now.

#### Contrarian - If autonomy accelerates

Telemetry-aware Resume and autonomous selector become important. But both depend on standardized verdicts and outcome data.

**Status:** Deferred.

## Tested Survivors

### Survivor A - `loop_diagnose`

**Claim:** The most useful next protocol is a lightweight diagnostic protocol/template for running MVL+ on correction chains.

**Strongest objection:** It is just MVL+ with a fancy name.

**Response:** The thinking remains MVL+, but the protocol adds the missing input/output contract. That is exactly what a protocol should do here.

**Verdict:** SURVIVE.

### Survivor B - `maintenance_branch_experiment`

**Claim:** The second most useful protocol defines git-backed candidate fundamentals experiments.

**Strongest objection:** This might be too early before diagnostics are mature.

**Response:** It should be built only when the first diagnostic finding motivates an actual source change. But as a priority, it is next because it prevents arbitrary source edits.

**Verdict:** SURVIVE.

### Survivor C - `structural_integrity_and_telemetry_contract`

**Claim:** The third most useful priority is an enforcement substrate: required artifact checks plus standardized verdict telemetry.

**Strongest objection:** This combines two different things.

**Response:** Implementation can split them. Ranking them together is justified because both are Primitive RC substrate and both unlock routing.

**Verdict:** SURVIVE / REFINE.

### Survivor D - `retrospective_outcome_review`

**Claim:** The fourth most useful protocol closes the loop after later evidence appears.

**Strongest objection:** It cannot run until enough later evidence exists.

**Response:** True, but specifying it early prevents branch experiments and source patches from becoming "done" without outcome closure.

**Verdict:** SURVIVE.

### Survivor E - `navigation_selection_record`

**Claim:** The fifth most useful protocol records Navigation choices and outcomes.

**Strongest objection:** Meaningful traversal seems more important for meta-loop.

**Response:** Meaningful traversal needs selection/outcome traces. Selection record is the cheaper precursor.

**Verdict:** SURVIVE.

## Killed or Deferred Candidates

### Immediate runtime Resume wire-up

Killed because Resume is blocked by telemetry standardization and basic inline resume already exists.

### Full meaningful traversal formula

Deferred because it needs trace and outcome data. A qualitative placeholder can remain in meta-loop v1.

### Protocol inventory hygiene as top-five

Refined downward. It is useful cleanup, especially for `unknown.md` and dormant `resume.md`, but it does not directly create a learning loop.

### Autonomous selector

Deferred until Navigation maps, human selections, and outcomes exist.

### New full cognitive discipline

Killed for current stage. MVL/MVL+ should remain the atomic reasoning method.

## Assembly Check

The strongest assembly is a Level 1 self-maintenance protocol stack:

```text
1. loop_diagnose
   Finds what went wrong in correction chains.

2. maintenance_branch_experiment
   Turns evidence-backed candidates into controlled git variants.

3. structural_integrity_and_telemetry_contract
   Gives outputs/specs basic machine-checkable quality signals.

4. retrospective_outcome_review
   Decides retain/revert/refine based on later evidence.

5. navigation_selection_record
   Starts collecting selection data for meta-loop and future autonomy.
```

This assembly is better than a meta-loop-first ranking because it creates the evidence substrate that meta-loop will later need.
