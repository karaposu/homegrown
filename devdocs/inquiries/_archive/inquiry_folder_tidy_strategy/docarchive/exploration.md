## Possibility Scan

Candidate organization approaches:

1. **Datetime-prefix folder names**  
   Example: `2026-04-27_finding_lineage_metadata`.

2. **Date bucket folders**  
   Example: `devdocs/inquiries/2026-04/slug/`.

3. **Status bucket folders**  
   Example: `devdocs/inquiries/active/slug/`, `complete/slug/`, `archive/slug/`.

4. **Physical archive folder inside inquiries**  
   Example: move old folders to `devdocs/inquiries/archive/slug/`.

5. **Status-only archive**  
   Keep folders in place; add `ARCHIVED` or `COLD` status in `_state.md`; optionally hide from default views later.

6. **README/index file**  
   Keep folders stable; add `devdocs/inquiries/README.md` listing recent, active, complete, superseded, archived.

7. **Manual lightweight index without tool**  
   Keep a hand-edited `README.md` with only "Current / Recent / Archived" pointers.

8. **Small helper command later**  
   Keep folders stable now; later add a helper if manual indexing becomes annoying.

9. **No structural change, only naming discipline going forward**  
   Keep old folders as-is; new folders use a clearer slug pattern.

10. **Hybrid: stable folders + manual index + optional archive status**  
    Avoid renames and moves; maintain a small index and mark stale items by status.

## Probes

### Probe A: Datetime Prefixes

Datetime prefixes solve one thing well: lexical sorting by creation date.

They create three costs:

- They lengthen every folder name.
- They make topic scanning harder because every name starts with a date.
- Retrofitting them would break or require rewriting path references.

They are more acceptable for future-only creation than retroactive renaming, but even future-only prefixes split conventions between old and new folders.

**Confidence:** Confirmed enough for decision.

### Probe B: Physical Archive Folder

A physical archive folder reduces visible clutter in the root folder.

It also changes paths. Moving `devdocs/inquiries/x/` to `devdocs/inquiries/archive/x/` breaks every existing path reference unless all references are rewritten.

Physical archive also hides historical lineage from simple path assumptions. Existing protocols assume inquiries live directly under `devdocs/inquiries/<name>/`.

**Confidence:** Confirmed enough for decision.

### Probe C: Status/Index Approach

An index or status view solves the user's actual problem without changing folder identity.

The simplest version does not require a script:

- Add `devdocs/inquiries/README.md`.
- Maintain sections such as `Current`, `Recent`, `Reference`, `Archived/Superseded`.
- Link to inquiry folders or findings.
- Keep all folders at stable paths.

Later, if this becomes tedious, a helper script can generate the index from `_state.md`.

**Confidence:** High.

### Probe D: Existing `devdocs/archive/`

Because `devdocs/archive/` already has discipline/material categories, using it for inquiry folders would overload the term "archive." It should not become the inquiry archive unless the archive taxonomy is redesigned.

If an inquiry archive exists, it should be inside `devdocs/inquiries/` as a view/status concept first, not as moved folders.

**Confidence:** High.
