# Critique: Inquiry Folder Tidy Strategy

## User Input

`devdocs/inquiries/inquiry_folder_tidy_strategy/_branch.md`

## Phase 0: Dimensions With Weights

| Dimension | Weight | Success Criteria |
|---|---:|---|
| Path stability | Critical | Existing references and runner assumptions remain valid. |
| Human navigability | Critical | User can quickly see current, recent, stale, and important inquiries. |
| Simplicity | Critical | First step is small and does not require a tooling project. |
| Scalability | High | Convention still works as inquiry count grows. |
| Metadata integrity | High | Missing status/state is surfaced or fixed. |
| Migration cost | High | Existing 59 folders do not require broad path rewrites. |
| Reversibility | Medium | If the convention feels wrong, it can be changed without damage. |
| Archive clarity | Medium | Archive semantics are clear and do not conflict with existing `devdocs/archive/`. |

## Phase 1: Fitness Landscape

### Viable Region

- Stable flat folders.
- Topic-first slug names.
- README/index as the human-facing view.
- Status hygiene pass.
- Archive as semantic section/status, not a physical move.

### Boundary Region

- Script-generated index: useful later, but probably more than the first step needs.
- Future-only date prefixes: possible, but creates a split convention.
- `ARCHIVED` status immediately: reasonable if the status enum is ready, but a README section may be enough first.

### Dead Region

- Retroactive datetime prefixes.
- Moving old inquiries into `devdocs/inquiries/archive/`.
- Status bucket subfolders such as `active/`, `complete/`, `archive/`.
- Reusing `devdocs/archive/` as the inquiry archive.

## Phase 2: Candidate Verdicts

### Candidate A: Retroactive Datetime Prefixes

**Prosecution:** This changes every folder path. Existing `refines`, `RELATED`, body references, and human memory would need migration. It also makes topic scanning worse because names begin with dates rather than concepts.

**Defense:** It gives chronological sorting in plain `ls` with no tools or index.

**Collision:** Defense solves only one display mode; prosecution shows structural damage to canonical identifiers.

**Verdict:** KILL.

**Seed extracted:** Dates belong in metadata/index rows, not in canonical folder identity.

### Candidate B: Future-Only Datetime Prefixes

**Prosecution:** It avoids retroactive rewrites but creates a two-era convention: old topic slugs and new date-prefixed slugs. Topic scanning still degrades for new folders.

**Defense:** It gives chronological sorting for future work without breaking old links.

**Collision:** Defense has merit, but the split convention is not worth it while an index can show dates cleanly.

**Verdict:** REFINE to "topic-first slugs, dates in README/index."

### Candidate C: Physical Inquiry Archive Folder

**Prosecution:** Moving folders changes paths and violates the flat `devdocs/inquiries/<slug>/` assumption. It also conflicts conceptually with the existing `devdocs/archive/`, which is not an inquiry archive.

**Defense:** It quickly reduces visible clutter in the root folder.

**Collision:** The clutter reduction is real but too expensive. Moving canonical objects is the wrong first lever.

**Verdict:** KILL as v1.

**Seed extracted:** Archive should first mean "lower visibility," not "new path."

### Candidate D: Status Bucket Subfolders

**Prosecution:** Every status change becomes a folder move. An inquiry moving from active to complete would change its path, causing repeated reference churn.

**Defense:** It gives immediate visual grouping in the filesystem.

**Collision:** This optimizes folder browsing at the cost of persistent identity. That is a poor trade for a linked knowledge system.

**Verdict:** KILL.

**Seed extracted:** Status should be metadata, not path topology.

### Candidate E: Manual README Index

**Prosecution:** It can go stale if the user forgets to update it.

**Defense:** It is the smallest useful visibility layer. It preserves paths, can be edited immediately, and can later be generated by a tool if manual upkeep fails.

**Collision:** Defense wins for v1. Staleness risk is real but reversible and observable.

**Verdict:** SURVIVE.

**Constructive output:** Create `devdocs/inquiries/README.md` with a clear note that it is a navigation view, not the source of truth.

### Candidate F: README + Status Hygiene

**Prosecution:** Requires a small cleanup pass over incomplete metadata.

**Defense:** The cleanup is exactly the missing integrity layer. Without it, any view is uncertain; with it, the user gains both readability and better system state.

**Collision:** Defense wins. The cleanup is bounded: current scan found 4 missing `_state.md` and 2 missing `## Status`.

**Verdict:** SURVIVE as strongest candidate.

**Constructive output:** Add README sections plus a `Needs Hygiene` section until missing state/status folders are resolved.

### Candidate G: Script-Generated Index Now

**Prosecution:** It turns a tidiness decision into a tooling task. The prior bad finding leaned too far in this direction.

**Defense:** It prevents README staleness and scales better.

**Collision:** Defense is valid later, not now. Manual convention should exist before automation encodes it.

**Verdict:** REFINE / DEFER.

**Constructive output:** Add automation only after manual README updates become repetitive or unreliable.

### Candidate H: Semantic Archive Section / Status

**Prosecution:** If only a README section, archive may feel less "real" than moving folders. If implemented as a status too early, it expands the status vocabulary.

**Defense:** It solves the real goal: reduce attention to old material while preserving canonical paths. It can start as a README section and later become formal `ARCHIVED` status.

**Collision:** Defense wins. The semantic archive can evolve without path churn.

**Verdict:** SURVIVE.

**Constructive output:** Start with an `Archived / Cold` README section; formalize `ARCHIVED` status only if the section becomes regularly used.

## Phase 3.5: Assembly Check

### Assembly Candidate: View-Based Tidiness

The best assembly:

1. Keep all inquiry folders directly under `devdocs/inquiries/<slug>/`.
2. Keep folder slugs topic-first; do not add datetime prefixes by default.
3. Add `devdocs/inquiries/README.md` as the tidy navigation surface.
4. Use README sections:
   - `Current`
   - `Recent Findings`
   - `Important Reference`
   - `Superseded`
   - `Archived / Cold`
   - `Needs Hygiene`
5. Fix or list missing `_state.md` / missing `## Status`.
6. Treat archive as semantic first; no physical archive folder yet.
7. Revisit automation after the manual index proves useful but annoying.

**Prosecution:** A manual README can go stale and does not make `ls devdocs/inquiries` itself shorter.

**Defense:** It solves the higher-value problem: finding what matters without breaking paths. It is reversible, small, and directly actionable. If it becomes stale, that is evidence for a generator script.

**Collision:** Defense wins decisively for the user's "non-complex alternatives" criterion.

**Verdict:** SURVIVE.
