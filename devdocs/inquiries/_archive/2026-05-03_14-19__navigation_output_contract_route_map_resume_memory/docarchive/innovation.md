# Innovation: Navigation Output Contract, Route Map, And Resume Memory

## User Input

Evaluate whether current Navigation supports a comprehensive structured route map with movement information, blocked paths, route purpose, and reusable Markdown maps for continuation.

## Seed

Navigation already has a map format, but it needs route-level state so saved maps can become continuation memory.

## Mechanism 1 - Lens Shifting

### Generic

View the Navigation output not as "next options" but as a **route map**.

### Focused

Every item should answer:

```text
What route is this?
Why does it matter?
Where does it move the project?
Can it be taken now?
What opens if it becomes possible?
What should future warm-up remember?
```

### Contrarian

Do not make Navigation select a route. Route awareness should make the map richer, not turn it into a plan.

## Mechanism 2 - Combination

### Generic

Combine Navigation map + reachability gates + warm-up current frontier.

### Focused

Navigation writes route maps. Warm-up reads route maps. Future Navigation then starts from remembered route state.

### Contrarian

The most valuable output may be not the highest-confidence route, but a low-confidence blocked route whose unblock check opens several future paths.

## Mechanism 3 - Inversion

### Generic

Instead of excluding blocked paths, include them as blocked routes.

### Focused

`UNBLOCK` should not replace blocked routes. It should point to the check or work that opens them.

### Contrarian

An Excluded section should not be where important blocked paths go. Excluded should mean structurally inapplicable, not currently unreachable.

## Mechanism 4 - Constraint Manipulation

### Generic

Add a constraint: every route must have a purpose.

### Focused

Add a constraint: every route must have a status.

Allowed statuses:

```text
open
blocked
deferred
active
done
stale
```

### Contrarian

Add a constraint: Navigation cannot assign owners, dates, or final sequence. That prevents route maps from becoming plans.

## Mechanism 5 - Absence Recognition

### Generic

Missing fields:

- Purpose;
- Movement;
- Status;
- Blocked by;
- Unlocks;
- Continuation note.

### Focused

Missing warm-up source:

```text
previous navigation maps
```

### Contrarian

Missing telemetry:

```text
route field completeness
blocked routes represented
previous-map continuity checked
```

## Mechanism 6 - Domain Transfer

### Generic

From road maps: blocked roads are still on the map. They are marked closed, not erased.

### Focused

From project trackers: status labels let future readers understand what happened without reconstructing the whole conversation.

### Contrarian

From dependency graphs: a blocked node matters because it may unlock many downstream nodes. Navigation should expose those unlock relationships.

## Mechanism 7 - Extrapolation

### Generic

If Navigation maps accumulate without status fields, future warm-up will see many old options but not know which are still live.

### Focused

If Navigation maps accumulate with route status and continuation notes, warm-up can reconstruct movement history and current frontier with much less user input.

### Contrarian

If route maps become too detailed, Navigation may become planning. The refinement must stop at descriptive route state.

## Survivor Candidates

### Candidate A - Keep Current Navigation Contract As-Is

Test:

- **Novelty:** none.
- **Scrutiny survival:** weak. It lacks route-state fields and prior-map warm-up rules.
- **Fertility:** low.
- **Actionability:** high but insufficient.
- **Mechanism independence:** weak.

Verdict: **KILL.**

### Candidate B - Add Route Fields To Navigation Items

Proposed fields:

```text
Purpose
Movement
Status
Blocked by
Unlocks
Continuation note
```

Test:

- **Novelty:** medium.
- **Scrutiny survival:** strong if Navigation remains non-selective.
- **Fertility:** high. It makes maps reusable.
- **Actionability:** high. Markdown format patch.
- **Mechanism independence:** strong; generated by lens shifting, absence recognition, domain transfer, and extrapolation.

Verdict: **SURVIVE.**

### Candidate C - List Blocked Routes In Main Map

Test:

- **Novelty:** medium.
- **Scrutiny survival:** strong. Comprehensive maps must include currently unreachable routes.
- **Fertility:** high. It exposes unblocking work and downstream routes.
- **Actionability:** high.
- **Mechanism independence:** strong; generated by inversion, domain transfer, and constraint manipulation.

Verdict: **SURVIVE.**

### Candidate D - Make Prior Navigation Maps Warm-Up Sources

Test:

- **Novelty:** high for Homegrown.
- **Scrutiny survival:** strong. Previous maps contain route-space memory that findings do not.
- **Fertility:** high. It can reduce user context steering.
- **Actionability:** high. Patch warm-up read rules.
- **Mechanism independence:** strong; generated by combination, absence recognition, and extrapolation.

Verdict: **SURVIVE.**

### Candidate E - Build A Separate Route Graph Engine

Test:

- **Novelty:** high.
- **Scrutiny survival:** weak now. It is overbuilt before Markdown maps have been tried.
- **Fertility:** high later.
- **Actionability:** low now.
- **Mechanism independence:** weak.

Verdict: **DEFER.**

### Candidate F - Patch 15/16 Taxonomy Mismatch

Test:

- **Novelty:** none.
- **Scrutiny survival:** strong. It is a correctness cleanup.
- **Fertility:** medium. It prevents confusion.
- **Actionability:** high.
- **Mechanism independence:** moderate.

Verdict: **SURVIVE.**

## Assembly Check

Best assembly:

```text
Navigation output contract:
  - keep Content / Process / Context grouping
  - keep type taxonomy and confidence
  - add route fields to each item
  - require blocked routes in main map
  - keep Excluded only for structurally inapplicable types
  - add route telemetry

Warm-up:
  - read prior navigation.md files in inquiry folders
  - read devdocs/navigation/*.md
  - summarize open/blocked/active/done/stale routes in Current Frontier
  - use prior maps as movement evidence, not as binding plans

Do not:
  - make Navigation select automatically
  - build a route graph engine yet
```
