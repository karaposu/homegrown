# Innovation: Next Load-Bearing Self-Improvement Loop

## User Input

Generate and test candidate next capabilities after navigation and materialization for a Homegrown system that must gradually become load-bearing for its own improvement without exceeding comprehension limits.

## Seed

The system should eventually start doing the user's breakthrough-finding job, but it must not become too complex to carry. The next piece should make future building more feasible and create a feedback loop that improves over time.

Valuation signal:

- The user values compounding capability, not feature count.
- The user is sensitive to complexity and false autonomy.
- The next step must improve the builder, not merely the built object.

## Mechanism Outputs

### 1. Lens Shifting

Generic variation:

- Instead of asking "what feature is next?", ask "what feedback would make every future feature more learnable?"

Focused variation:

- Under the lens of Homegrown as a training system, the next capability is not multihead throughput but outcome capture: the data stream that trains future protocol revisions.

Contrarian variation:

- The next breakthrough may be intentionally boring: a disciplined after-use review record, because reliable compounding often starts as mundane accounting.

Test:

- Novelty: medium; retrospectives are known, but positioning them as Homegrown's training-data substrate is strong.
- Scrutiny survival: strong; it directly addresses complexity and evidence gaps.
- Fertility: high; it feeds navigation, branching, materialization, reflect, and comprehension.
- Actionability: high; can start as Markdown protocol.
- Mechanism independence: appears again through absence recognition, domain transfer, and extrapolation.

Candidate extracted:

- C1: Outcome Review / Retrospective Calibration Protocol.

### 2. Combination

Generic variation:

- Combine navigation map + branch protocol + materialization trace + retrospective review.

Focused variation:

- A completed inquiry or artifact produces a finding/trace. After use, outcome review produces maintenance candidates. Navigation enumerates what to do with those candidates. Branch protocol creates child inquiries. Materialization implements selected fixes.

Contrarian variation:

- Combine "human frustration moments" with protocol trace records: the user's corrections become first-class training examples, not transient chat comments.

Test:

- Novelty: high in this local system; it combines pieces already present but not yet joined.
- Scrutiny survival: strong; each component has a clear role.
- Fertility: high; creates a closed loop.
- Actionability: high; requires protocol edits and a backlog/index.
- Mechanism independence: high; also emerges via domain transfer and extrapolation.

Candidate extracted:

- C2: Closed Builder Loop: `finding/trace -> outcome review -> maintenance backlog -> navigation/branch/materialization -> new trace`.

### 3. Inversion

Generic variation:

- Invert "the system should think more" into "the system should remember the consequences of thinking better."

Focused variation:

- Invert "Comprehend should manage navigation" into "Navigation should be fed by a better memory; comprehension can summarize that memory later."

Contrarian variation:

- Invert "multihead creates intelligence" into "multihead reveals the absence of intelligence if the system cannot compare outcomes."

Test:

- Novelty: medium.
- Scrutiny survival: strong; the inversion exposes why breadth without feedback is dangerous.
- Fertility: high; reframes multihead and comprehension as downstream.
- Actionability: medium-high; suggests sequencing more than artifact shape.
- Mechanism independence: strong; matches lens shift and critique intuition.

Candidate extracted:

- C3: Feedback-before-scale sequencing rule.

### 4. Constraint Manipulation

Generic variation:

- Add constraint: every new capability must produce reusable evidence for future capability decisions.

Focused variation:

- Add constraint: every materialized artifact must have a follow-up review gate after real use.

Contrarian variation:

- Add constraint: no multihead branch set unless each branch has an explicit outcome comparison path.

Test:

- Novelty: medium.
- Scrutiny survival: strong; constraints are protective and cheap.
- Fertility: high; can become protocol policy.
- Actionability: high.
- Mechanism independence: strong.

Candidate extracted:

- C4: Evidence-producing capability rule.

### 5. Absence Recognition

Generic variation:

- What should exist but does not: a record that says whether a finding or artifact actually worked later.

Focused variation:

- Missing artifact: a maintenance backlog where outcome reviews turn into prioritized, evidence-backed improvement candidates.

Contrarian variation:

- Missing negative space: a "no improvement needed" record. Without explicit no-op outcomes, the system only learns from failures and over-corrects.

Test:

- Novelty: high locally.
- Scrutiny survival: strong; the absence is visible in the current files.
- Fertility: high; supports learning and prioritization.
- Actionability: high.
- Mechanism independence: high.

Candidate extracted:

- C5: Maintenance Backlog / Baldwin Log with positive, negative, and no-op outcomes.

### 6. Domain Transfer

Generic variation:

- From software engineering: tests and incident retrospectives turn failures into changes in code, checklists, and runbooks.

Focused variation:

- From ML training: data, not direct weight editing, changes the system. In Homegrown, outcome records are the training data; protocol/materialization edits are the weight updates.

Contrarian variation:

- From science: lab notebooks matter before complex instrumentation. The system needs observation discipline before autonomy machinery.

Test:

- Novelty: medium; the analogies are known but fit tightly.
- Scrutiny survival: strong; they explain why feedback must precede scale.
- Fertility: high; points toward schema and evaluation.
- Actionability: high.
- Mechanism independence: high.

Candidate extracted:

- C6: Lab-notebook-for-thinking / training-data substrate.

### 7. Extrapolation

Generic variation:

- If the system keeps adding protocols without outcome feedback, it will become conceptually rich but empirically weak.

Focused variation:

- If branch and navigation are used for months, the most valuable asset will be a record of which branch sources produced useful improvements and which repeatedly wasted effort.

Contrarian variation:

- If multihead arrives early, the limiting factor becomes not generation of branches but digestion of branch results.

Test:

- Novelty: medium.
- Scrutiny survival: strong; aligns with current complexity risk.
- Fertility: high; gives a future-state rationale.
- Actionability: medium-high.
- Mechanism independence: high.

Candidate extracted:

- C7: Branch/use outcome analytics before multihead scale.

## Assembly Check

The strongest assembly combines C1, C2, C4, and C5:

```text
Outcome Review Protocol
  records after-use evidence
  extracts maintenance candidates
  appends to a Baldwin/maintenance log
  gives navigation and branch protocol evidence-backed next moves
  gives materialization a follow-up gate
```

This assembly is stronger than any individual candidate because:

- C1 supplies the review operation.
- C2 supplies the loop architecture.
- C4 supplies the policy that future capabilities must generate evidence.
- C5 supplies the durable memory/index.

## Candidate Set for Critique

### Candidate B - Build Isolated Comprehend as State Manager Next

Create a comprehension layer that reads the project and manages navigation/next steps.

### Candidate C - Build Outcome Review / Retrospective Calibration Protocol Next

Create a reusable after-use protocol that records expected vs actual outcomes, mismatches, failure attribution, maintenance candidates, and routing actions.

### Candidate D - Build Full Evaluation Harness Next

Focus first on empirical scoring, tests, and benchmark infrastructure before more self-improvement protocols.

### Candidate E - Build Minimal Builder Loop Assembly Next

Create outcome review plus a maintenance backlog/Baldwin log and integrate it lightly with navigation, branch protocol, and materialization follow-up gates.
