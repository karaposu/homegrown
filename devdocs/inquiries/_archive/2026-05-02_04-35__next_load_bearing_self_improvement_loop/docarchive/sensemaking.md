# Sensemaking: Next Load-Bearing Self-Improvement Loop

## User Input

What is the most important next capability after navigation and synthesis/materialization protocol for making Homegrown improve itself over time without becoming too complex to carry or comprehend? Compare multihead, isolated comprehension, and other possible next steps.

## SV1 - Baseline Understanding

The user is asking for strategic prioritization. The system is becoming complex, so the next step must not merely add power; it must make future building more feasible. The likely question is not "what feature is exciting next?" but "what feature makes the system better at improving itself safely?"

## Phase 1 - Cognitive Anchor Extraction

### Constraints

- The system must be built gradually and in pieces.
- Each piece should be load-bearing for future building.
- Human comprehension is limited; the system must reduce dependence on the user holding all structure in mind.
- Navigation and branch protocol already exist or are emerging.
- Materialization/synthesis protocol is needed for concrete artifacts.
- Multihead/parallelism must not multiply complexity before the system can absorb results.
- The next capability should help discover future breakthroughs over time, not only solve one local issue.

### Key Insights

- A loop that finds breakthroughs over time needs feedback from reality, not just more reasoning passes.
- Navigation without outcome memory maps possibilities but cannot learn which possibilities were actually valuable.
- Branching without outcome review creates a tree but not a learning tree.
- Materialization without after-use review creates artifacts but not training data.
- Isolated comprehension can reduce cognitive load only if it reads reliable accumulated evidence, not just raw Markdown.

### Structural Points

- Existing chain:

```text
MVL/MVL+ -> finding -> navigation -> branch/materialization
```

- Missing chain:

```text
artifact or inquiry used -> outcome observed -> mismatch diagnosed -> maintenance candidate created -> navigation/branch/materialization choose next improvement
```

- The system needs a boundary operation after use, analogous to reflection after thinking and validation after implementation.

### Foundational Principles

- Disciplines judge or transform cognitive content; protocols route, persist, and govern operations.
- Load-bearing improvements are improvements that make future improvements easier, safer, or more accurate.
- Autonomy must be earned by calibrated feedback, not assumed from richer internal vocabulary.
- Scaling breadth before feedback risks amplifying weak process.

### Meaning-Nodes

- Load-bearing builder.
- Baldwin cycle.
- Outcome review.
- Retrospective calibration.
- Maintenance backlog.
- State comprehension.
- Multihead as scaling, not learning.
- Navigation as map, not choice.

## SV2 - Anchor-Informed Understanding

The strongest interpretation is that the user is asking for the next architectural feedback mechanism. Navigation and branching create optionality. Materialization creates artifacts. But the system still needs to learn from what happened after those artifacts or inquiries were used. The next piece should likely be a cross-run outcome review protocol that turns real-use outcomes into maintenance candidates and future route pressure.

## Phase 2 - Perspective Checking

### Technical / Logical Perspective

New anchors:

- A system cannot improve its own protocols reliably unless it retains comparisons between expected and actual results.
- Outcome records need stable identifiers: source inquiry, source artifact, prediction/intent, validation, actual outcome, mismatch, candidate fix.
- A selector/prioritizer needs evidence. Without evidence, it is just another critique-like judgment layer.

### Human / User Perspective

New anchors:

- The user currently supplies breakthrough detection, frustration signals, and priority judgment.
- The system should not immediately replace the user; it should first capture the user's corrections and observations as reusable training data.
- The user's limited brain is best supported by externalizing "what failed repeatedly and what should be improved next."

### Strategic / Long-Term Perspective

New anchors:

- Multihead is eventually valuable, but only after the system can compare and merge branch results.
- State comprehension is eventually valuable, but it needs durable outcome evidence to remain grounded.
- The next system capability should create compounding returns across all later work.

### Risk / Failure Perspective

New anchors:

- Adding multihead too early can create many outputs that nobody can absorb.
- Adding a "comprehend as me" controller too early can create a fluent but uncalibrated manager.
- Adding a vague self-improvement discipline could become self-referential and non-empirical.
- The safer form is a protocol with records and after-use evidence.

### Resource / Feasibility Perspective

New anchors:

- A minimal outcome review protocol is small: a template and routing rules.
- It can begin manually with one file per review or one append-only log.
- It does not require automation, multi-agent orchestration, or code harnesses to start producing value.

### Definitional / Consistency Perspective

New anchors:

- Navigation's own definition says it does not choose.
- Materialization's lifecycle already says retrospective learning happens after use.
- Protocol docs already frame protocols as Baldwin-cycle substrate.
- Therefore outcome review is not a random new idea; it is the missing operationalization of already-stabilized principles.

## SV3 - Multi-Perspective Understanding

The problem has shifted from "what feature after navigation?" to "what feedback substrate makes navigation, branching, materialization, and future comprehension learn from use?" Multihead and isolated comprehension are not wrong, but they are downstream of the feedback substrate. The next piece should capture outcomes and convert them into navigable improvement pressure.

## Phase 3 - Ambiguity Collapse

### Ambiguity: What does "self-sufficient" mean here?

**Strongest counter-interpretation:**
Self-sufficient could mean autonomous multihead execution: the system runs many branches, compares them, and decides what to build.

**Why the counter-interpretation fails (structural grounds):**
Autonomous multihead execution requires branch creation, branch selection, branch result comparison, merge policy, and outcome calibration. Branch creation exists, but selection and outcome calibration are weak. If multihead runs before those pieces, it produces more outputs without the ability to learn which outputs improved the builder.

**Confidence:** HIGH.

**Resolution:**
Self-sufficient should mean "able to accumulate evidence from its own operation and use that evidence to improve future operations," before it means "able to run many things autonomously."

**What is now fixed?**
Feedback and calibration are prerequisites for serious autonomy.

**What is no longer allowed?**
Treating multihead throughput as self-sufficiency.

**What now depends on this choice?**
Next-step priority, protocol design, and whether isolated comprehension is framed as controller or state reader.

**What changed in the conceptual model?**
The target becomes a learning loop, not a scheduling architecture.

### Ambiguity: Is the next thing a discipline or a protocol?

**Strongest counter-interpretation:**
It could be a new discipline called "comprehend" or "self-improve" that evaluates system state and decides next moves.

**Why the counter-interpretation fails (structural grounds):**
The missing operation is not primarily a cognitive transformation. It is a cross-run operational contract: capture source authority, expected effect, actual effect, mismatch, evidence, and routing. That is protocol territory. A discipline can analyze the record later, but the first missing thing is the record and routing machinery.

**Confidence:** HIGH.

**Resolution:**
The next core artifact should be a protocol, probably `outcome_review.md` or `retrospective_calibration.md`, not a new top-level thinking discipline.

**What is now fixed?**
The v1 should define inputs, record format, timing, outputs, routing actions, and failure modes.

**What is no longer allowed?**
Free-form self-improvement commentary without durable records.

**What now depends on this choice?**
Integration with materialization trace, navigation, branch inquiry, and loop diagnosis.

**What changed in the conceptual model?**
Self-improvement becomes an operational feedback channel.

### Ambiguity: Does comprehension come before outcome review?

**Strongest counter-interpretation:**
The system first needs a comprehend layer to know what exists and manage navigation.

**Why the counter-interpretation fails (structural grounds):**
Comprehension can model current state, but without outcome records it cannot know which parts actually worked. It would read findings and traces, but not calibrated effects. That makes it a map of claimed structure, not a map of learned structure.

**Confidence:** MEDIUM-HIGH.

**Resolution:**
Comprehension should become a state-model consumer of outcome records. A lightweight comprehension view can be built soon, but outcome review should define the evidence it reads.

**What is now fixed?**
Comprehension is a near-term dependent component, not the primary next load-bearing piece.

**What is no longer allowed?**
Treating "comprehend as me" as a fully autonomous manager before feedback exists.

**What now depends on this choice?**
The later state model should be designed around logs/backlogs, not only inquiry text.

**What changed in the conceptual model?**
Comprehension becomes downstream of memory quality.

## SV4 - Clarified Understanding

The next most important capability is a reusable after-use feedback protocol. It should turn actual outcomes of inquiries, branches, and materialized artifacts into durable evidence. That evidence should feed navigation, branch creation, materialization, loop diagnosis, and later state comprehension. Multihead and autonomous comprehension should be deferred or scoped until this feedback layer exists.

## Phase 4 - Degrees-of-Freedom Reduction

### Fixed Variables

- The next capability should be small and manually usable at first.
- It should be a protocol, not a free-form new discipline.
- It should run after use, not immediately after conclusion only.
- It should produce trace data that future loops can inspect.
- It should create maintenance candidates or branchable questions.

### Eliminated Options

- Full multihead as the immediate next priority.
- A broad autonomous "comprehend as user/controller" layer as immediate priority.
- More conceptual taxonomies without outcome evidence.
- Treating materialization validation as sufficient for learning; validation happens before use, while retrospective calibration happens after use.

### Remaining Viable Paths

1. Build a minimal `outcome_review.md` protocol.
2. Add a maintenance backlog/Baldwin log format.
3. Make materialization traces point to an outcome review gate.
4. Later build a state-comprehension view over findings, branches, materializations, and outcome records.
5. Later add multihead branch sets once the system can prioritize, compare, and merge outcomes.

## SV5 - Constrained Understanding

The solution space has narrowed to one core path:

```text
Use -> outcome record -> mismatch diagnosis -> maintenance candidate -> navigation/branch/materialization -> new use
```

This is the smallest loop that trains Homegrown on its own history. It does not automate everything. It creates the data and routing pressure that later automation needs.

## Phase 5 - Conceptual Stabilization

## SV6 - Stabilized Model

The next load-bearing piece after navigation and materialization is not multihead. It is a cross-run outcome review / retrospective calibration protocol.

Navigation gives the system a map of possible next moves. Branch protocol gives it a tree structure. Materialization gives it concrete artifacts. But none of those proves that the resulting move helped. The missing capability is after-use learning: record what was expected, what actually happened, what mismatch appeared, which discipline/protocol likely caused it, and what maintenance candidate should be created.

This differs from SV1 by making the priority more precise. The issue is not simply strategic sequencing; it is the absence of a general training-data loop. The system needs to convert human corrections, artifact usage, failed branches, validation misses, and repeated critique kills into durable evidence that future loops can navigate.
