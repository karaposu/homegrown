# Exploration: Next Load-Bearing Self-Improvement Loop

## User Input

The user is worried about building a system larger than human comprehension can carry. The system must be built gradually, in pieces, and the pieces must be load-bearing for the act of building itself. Recent breakthroughs include branch protocol, navigation, and materialization/synthesis protocol. The user asks what the most important next thing is after navigation and materialization: multihead, isolated comprehension, or something else.

## Mode and Entry Point

- Mode: possibility exploration, grounded in existing artifacts.
- Entry point: signal-first. The user's strongest signal is "the loop must start finding breakthroughs over time and eventually do part of my job."

## Exploration Cycles

### Cycle 1 - Broad Scan of Current Capability Regions

Scanned regions:

- MVL/MVL+ loop runners: produce inquiry folders, run E/S/D/I/C, conclude into `finding.md`.
- Navigation: enumerates possible next directions after a loop; explicitly does not choose.
- Branch inquiry protocol: creates child inquiry folders with lineage, source authority, depth policy, and path discipline.
- Materialization lifecycle: turns decisions into artifacts through desc, plan, dynamic critic, repair, implementation, validation, trace, retrospective learning.
- Reflect/checkpoint architecture: checkpoint produces structural signals; reflect integrates process quality and can feed Baldwin-cycle improvements.
- Loop diagnosis: can inspect failures in prior runs and identify discipline/protocol gaps.
- Scientific summary: warns that the system is conceptually strong but empirically weak; next scientific step is validation, stricter artifacts, and evidence.

Signals detected:

- Navigation creates a tree of possibilities but does not select or learn from outcomes.
- Branching gives topology but not priority or cross-run calibration.
- Materialization governs file changes but its retrospective learning step is not yet a general cross-run system.
- Reflect/checkpoint already points toward a Baldwin cycle, but mostly for structural checks and process quality.
- The system has many ways to produce Markdown, fewer ways to convert observed outcomes into durable future improvement pressure.

Resolution decision:

- Zoom in on the gap between "enumerate next moves" and "learn which moves improved the builder."

Probe:

- Navigation can show directions like DEVELOP, TEST, DIAGNOSE, BRANCH, MERGE, CONSOLIDATE.
- Branch protocol can create child inquiries from selected directions.
- Materialization can create artifacts from selected decisions.
- Missing bridge: a stable record of whether a selected direction or materialized artifact later solved the real problem, what failed, and which protocol/discipline should change.

Frontier state:

- The territory is partly mapped. Existing components cover thinking, branching, and implementation. The outcome-learning layer is only implied.

Confidence map update:

- Confirmed: navigation is enumeration, not selection.
- Confirmed: materialization has retrospective learning, but as a lifecycle phase, not as a reusable cross-run protocol.
- Scanned: reflect/checkpoint can support learning but is not the whole project-level improvement loop.
- Unknown: exact schema and storage location for cross-run outcome records.

### Cycle 2 - Probe Candidate "Multihead Next"

Scanned:

- Multihead/parallelism would let several branches or agents pursue different directions at once.
- The branch protocol was deliberately designed for future branch sets and multihead execution.

Signals:

- Multihead increases breadth and throughput.
- It does not by itself decide which branches are worth running.
- Without outcome feedback, multihead can multiply low-quality branches, context load, merge burden, and false confidence.

Resolution decision:

- Keep multihead as a future scaling layer, but probe whether it is load-bearing now.

Probe:

- Multihead is useful when the system already has a reliable way to create candidate branches, rank/route them, compare outcomes, and merge lessons.
- Current system has branch creation and navigation, but weak outcome comparison and prioritization.

Frontier state:

- Multihead is reachable, but gated by feedback, selection, and merge discipline.

Confidence map update:

- Confirmed: multihead is not the next core capability if the goal is learning quality over time.
- Inferred: multihead becomes valuable after outcome records and selector pressure exist.

### Cycle 3 - Probe Candidate "Isolated Comprehend Next"

Scanned:

- The user asks whether comprehend can become "me" and manage navigation.
- Existing comprehension discipline is a working-model builder for opaque artifacts.
- Navigation enumerates possible directions and explicitly leaves selection to a separate operation.

Signals:

- A state model is missing: what is the current Homegrown system, what has already been tried, what is fragile, what is load-bearing?
- Comprehension could build this model.
- But "comprehend as me" risks becoming a large controller too early.

Resolution decision:

- Treat isolated comprehension as a component, not the next top-level controller.

Probe:

- Useful near-term form: a project-state comprehension view that reads findings, maintenance backlog, outcome records, and open branches.
- Risky form: an autonomous "manager" that chooses navigation, branches, materialization, and protocol edits without calibrated feedback.

Frontier state:

- Comprehension is relevant, but it needs outcome data to avoid becoming a static map.

Confidence map update:

- Confirmed: isolated comprehension is needed as a state-model capability.
- Inferred: it should follow or co-evolve with outcome review, not replace it.

### Cycle 4 - Probe Candidate "Outcome Review / Baldwin Loop"

Scanned:

- `enes/materialization_lifecycle.md` already says retrospective learning should happen after the artifact has been used.
- `thinking_disciplines/protocols/what_is_protocol.md` frames protocols as the operational substrate for Baldwin cycles: observe, detect pattern, propose spec change, evaluate, encode.
- `devdocs/inquiries/_archive/reflect_as_primitive_rc/finding.md` shows a two-mechanism learning architecture: checkpoint catches immediate signals, reflect integrates patterns, patterns inform spec improvements.

Signals:

- This directly matches the user's neural-network analogy: the system should be trained by accumulated data from its own runs.
- It is load-bearing for the building action because every future artifact and inquiry can produce training data.
- It is smaller than full multihead or autonomous comprehension.
- It gives navigation something better to read than raw findings: prioritized maintenance candidates backed by outcome evidence.

Resolution decision:

- Zoom in. This appears to be the missing load-bearing next capability.

Probe:

- Minimal artifact: an `outcome_review` or `retrospective_calibration` protocol.
- It runs after an inquiry result, materialized artifact, branch, or protocol change has been used.
- It records expected effect, actual outcome, mismatch, failure source, maintenance candidate, and next routing action.
- It feeds Navigation, Branch Inquiry, Loop Diagnose, Reflect, and Materialization.

Frontier state:

- Strongly mapped. This is not a new grand discipline; it is a boundary protocol connecting use outcomes back into improvement.

Confidence map update:

- Confirmed: materialization already has a retrospective phase, but it needs reusable protocol form.
- Confirmed: existing docs independently point toward Baldwin-cycle feedback.
- Inferred: this should come before serious multihead.

### Jump Scan - Validation/Evaluation Territory

Jump-scanned away from architecture vocabulary into the scientific credibility problem.

Findings:

- The scientific summary says the next scientific step should be validation, stricter artifact schemas, and evaluation rather than more concepts.
- Outcome review is the operational way to accumulate validation data from real use.
- A separate evaluation harness is useful for ARC or controlled experiments, but the general Homegrown builder needs a broader feedback substrate first.

Surprise:

- The same answer appears from two directions: self-improvement architecture and scientific credibility both point to outcome capture/calibration.

## Convergence Assessment

- Frontier stability: strong. New scans keep pointing to the same gap: outcome-to-improvement feedback.
- Declining discovery rate: strong. Navigation, materialization, reflect, and scientific summary converge on the same missing layer.
- Bounded gaps: acceptable. Exact schema/storage is still open, but the next capability class is clear.

## Structural Map

### Territory Overview

The current Homegrown system has four major capability regions:

1. Thinking loop: MVL/MVL+ produces disciplined inquiry outputs.
2. Direction mapping: Navigation enumerates possible next moves.
3. Topology: Branch protocol lets follow-up questions become explicit child inquiries.
4. Artifact creation: Materialization lifecycle governs concrete file changes.

The underdeveloped region is:

5. Outcome learning: a durable way to compare intent, prediction, validation, real use, failure, and future improvement.

### Inventory

- Existing and solid enough:
  - Inquiry folders and state.
  - Conclude/resume flow.
  - Navigation as possibility enumeration.
  - Branch creation and lineage.
  - Materialization lifecycle concept.
- Existing but not generalized:
  - Retrospective learning inside materialization.
  - Reflect/checkpoint Baldwin-cycle architecture.
  - Optional improvement observations.
- Missing or weak:
  - Cross-run outcome review protocol.
  - Maintenance candidate extraction from outcomes.
  - Priority/routing pressure based on evidence.
  - Project-state comprehension over accumulated outcomes.
  - Merge/selection discipline for branch sets.

### Signal Log

- Probed: multihead as next capability.
  - Result: defer until selection and outcome feedback exist.
- Probed: isolated comprehend as next capability.
  - Result: useful as state model, risky as controller.
- Probed: outcome review/Baldwin loop.
  - Result: strongest candidate for next load-bearing capability.
- Deferred:
  - Full automated selector.
  - Multihead scheduler.
  - Branch-set merge protocol.
  - ARC-specific harness integration.

### Confidence Map

- Confirmed: Navigation does not select.
- Confirmed: Branch protocol creates topology but not learning.
- Confirmed: Materialization needs after-use retrospective learning.
- Confirmed: Scientific credibility requires validation data.
- Scanned: Comprehend can become a state-model reader.
- Inferred: Multihead should wait until feedback and priority signals exist.
- Unknown: best exact name and storage path for outcome records.

### Frontier State

The next frontier is narrow: promote retrospective learning into an operational, reusable protocol that every inquiry/materialization/branch can feed.
