# Critique: Next Load-Bearing Self-Improvement Loop

## User Input

Evaluate competing next capabilities for Homegrown after navigation and materialization: multihead, isolated comprehend/state manager, outcome review, full evaluation harness, and a minimal builder-loop assembly.

## Phase 0 - Dimension Construction

Dimensions extracted from sensemaking:

| Dimension | Weight | Success Criteria |
| --- | --- | --- |
| Load-bearing value | Critical | Makes future building easier, safer, or more accurate across many later tasks. |
| Feedback quality | Critical | Converts real use or observed outcomes into reusable evidence. |
| Complexity control | Critical | Reduces or contains cognitive/operational complexity rather than multiplying it. |
| Incremental feasibility | High | Can be built and used in a small v1 without large automation. |
| Integration fit | High | Connects cleanly to existing MVL/MVL+, navigation, branch, materialization, reflect, and loop diagnose. |
| Autonomy readiness | Medium | Prepares future autonomy without pretending autonomy already exists. |
| Empirical grounding | High | Improves the system's ability to compare claims against observed behavior. |

Dimension validation:

- These dimensions match the user's stated fear of complexity, desire for gradual load-bearing construction, and long-term need for self-improvement.
- Missing-dimension check: no separate "novelty" dimension is needed; novelty without load-bearing value is not useful here.

## Phase 1 - Fitness Landscape

### Viable Region

The viable region is a capability that:

- creates outcome evidence;
- can start small;
- feeds navigation/branch/materialization;
- reduces human memory burden;
- does not require full automation before it produces value.

### Dead Region

Dead or premature region:

- increases output volume before digestion;
- creates a controller without calibration;
- adds taxonomy without use evidence;
- requires major infrastructure before any feedback is gained.

### Boundary Region

Boundary candidates:

- useful but not first;
- valuable as consumers of outcome records;
- valuable for domain-specific evaluation but not the general self-improvement substrate.

### Unexplored Region

- Exact schema and path design remain unexplored implementation details.
- How much review can be skipped for trivial artifacts remains a policy detail.

## Phase 2 and 3 - Candidate Verdicts

## Candidate A - Build Multihead / Branch-Set Execution Next

### Prosecution

The killer objection is sequencing. Multihead increases branch count, output count, merge burden, and coordination complexity. The system currently has branch creation and navigation, but weak outcome comparison, weak merge policy, and no mature selector. Multihead would multiply the very complexity the user fears.

### Defense

Multihead is strategically important. The branch protocol was designed with future branch sets in mind, and multiple heads can explore thinking space faster than one sequential loop.

### Collision

The defense is real but future-facing. It fails as the immediate next priority because breadth without digestion is not self-improvement. It becomes strong only after each branch can produce comparable outcome traces and the system has a way to rank/merge them.

### Verdict

REFINE / DEFER.

Constructive output:

- Keep multihead as a future scaling layer.
- Prerequisite: outcome review records and branch result comparison/merge policy.
- Do not build as the next load-bearing piece.

## Candidate B - Build Isolated Comprehend as State Manager Next

### Prosecution

The killer objection is grounding. A state manager that reads findings and navigation maps but lacks outcome evidence can become a polished summary of claims rather than a calibrated model of what worked. "Comprehend as me" is too broad and could recreate the user's cognitive load inside another vague layer.

### Defense

The user does need an externalized project-state model. Comprehension can reduce the need to hold all Homegrown structure in mind and can help navigation operate at the right resolution.

### Collision

The defense survives as a dependent capability. Comprehension is valuable, but it should read outcome logs, maintenance candidates, branch trees, and materialization traces. It should not be the first controller.

### Verdict

REFINE.

Constructive output:

- Build later as "state comprehension over evidence-backed records."
- Do not frame v1 as autonomous manager.
- The first useful version can summarize current system state, open maintenance candidates, and branch tree health after outcome review exists.

## Candidate C - Build Outcome Review / Retrospective Calibration Protocol Next

### Prosecution

The main objection is that it may become another Markdown ritual. If reviews are not used by navigation, branch, materialization, or maintenance decisions, they become archive noise. Another risk: attribution may overclaim which discipline failed when evidence is weak.

### Defense

This directly addresses the core need: the system learns from its own use. It is small, can start manually, integrates with existing protocols, and creates the evidence that future comprehension, navigation selection, and multihead comparison require. It also matches already-established materialization phase 8 and Baldwin-cycle protocol theory.

### Collision

The prosecution does not kill the candidate; it defines design constraints. The protocol must produce routeable maintenance candidates, allow uncertain attribution, and distinguish "record only" from "action needed."

### Verdict

SURVIVE.

Constructive output:

- Build as a protocol, not a discipline.
- Include explicit record fields and routing actions.
- Require evidence-backed or uncertainty-marked attribution.
- Feed a discoverable maintenance/Baldwin log.

## Candidate D - Build Full Evaluation Harness Next

### Prosecution

The killer objection is scope mismatch. A full evaluation harness is necessary for ARC-like empirical work, but Homegrown's builder loop needs a broader feedback substrate across protocol edits, inquiry outcomes, materialized artifacts, user corrections, and branch results. A harness may be too domain-specific or too heavy as the immediate next general capability.

### Defense

The scientific summary is correct: validation is a weakness. A harness would ground the system empirically and prevent self-confirming protocol expansion.

### Collision

The defense survives for task-specific domains, especially ARC. But the general system still needs outcome review to decide what to validate, record what validation missed, and convert failed predictions into maintenance candidates.

### Verdict

REFINE.

Constructive output:

- Build evaluation harnesses where the domain has exact scoring, such as ARC.
- Do not substitute harness work for cross-run outcome review.
- Let outcome review record validation results and downstream behavior so harnesses become part of the broader learning loop.

## Candidate E - Build Minimal Builder Loop Assembly Next

Assembly:

```text
outcome_review.md
  + maintenance/Baldwin log
  + materialization follow-up gate
  + navigation/branch input convention
```

### Prosecution

The strongest objection is scope creep. If this assembly tries to solve selection, branch merge, comprehension, and automation all at once, it becomes the exact oversized system the user fears.

### Defense

The assembly can be minimal. It creates one protocol and one log/index convention, then uses existing navigation and branch machinery without automating them. It is the smallest closed loop from use back to future improvement.

### Collision

The defense wins if v1 is strict:

- record outcomes;
- extract candidates;
- append to backlog/log;
- expose routeable next actions;
- defer autonomous selection and multihead execution.

### Verdict

SURVIVE - ranked first.

Constructive output:

- Build minimal assembly next.
- Name the protocol concretely: `outcome_review.md` is clearer than `retrospective_calibration.md` for v1.
- Use "Baldwin loop" as the architecture concept, not necessarily the file name.

## Phase 3.5 - Assembly Check

Best assembly:

```text
Materialized artifact or inquiry result is used
  -> outcome review records expected vs actual behavior
  -> maintenance candidate is created or no-op is recorded
  -> maintenance/Baldwin log indexes candidates
  -> navigation reads the log when mapping next directions
  -> branch protocol creates child inquiries for selected candidates
  -> materialization implements selected fixes
  -> later outcome review checks whether the fix worked
```

This assembly survives because it is the first full feedback circuit. It is also the best bridge between the user's intuition and system memory: user observations become durable evidence rather than disappearing into chat.
