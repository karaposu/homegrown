# Sensemaking: Target Fields And MVL Essence

## User Input

`devdocs/inquiries/2026-05-07_07-06__target_fields_mvl_essence/_branch.md`

## SV1 - Baseline Understanding

Initial interpretation:

```text
The previous finding treated the five fields too much like a record that can be written near the start. The user's objection is that MVL exists because those exact roles may not be known yet.
```

The question is not merely naming. It is about whether a target-alignment mechanism should live before the loop, inside the loop, or after the loop.

## Phase 1 - Cognitive Anchor Extraction

### Constraints

- MVL/MVL+ must still start from a question, goal, and scope check.
- MVL/MVL+ must not require complete understanding before its own disciplines run.
- The five fields must not become a rigid form-fill that replaces Exploration or Sensemaking.
- The final answer still needs target-fit validation.
- The fields must be useful for preventing drift without pretending drift is fully knowable upfront.

### Key Insights

- The five fields are not equal in when they become knowable.
- `evidence_domain` is often partly known before the loop but can be corrected by Exploration.
- `diagnosis_target` is often exactly what Sensemaking discovers.
- `implementation_target` is often premature before Innovation and Critique.
- `out_of_scope_target` often appears only after Decomposition exposes boundaries and adjacent temptations.
- The fields are not the whole essence of MVL; they are a lens on MVL's target alignment.

### Structural Points

There are three phases:

1. **Initial orientation**
   - what the loop thinks it is looking at;
   - may contain unknowns and hypotheses.

2. **Loop clarification**
   - disciplines revise the target roles;
   - the record matures through evidence and reasoning.

3. **Final target-fit validation**
   - the finding is compared against the stabilized target roles;
   - mismatch triggers repair, another loop, or non-conclusion.

### Foundational Principles

- A cognitive loop cannot require as input the understanding it is supposed to produce.
- A useful record can begin incomplete if its incompleteness is explicit.
- Alignment checks should compare against the best stabilized understanding, not against stale first guesses.
- A gate is justified only when mismatch has operational consequences.

### Meaning-Nodes

- evolving state;
- provisional target;
- stabilized target;
- target-fit validation;
- discipline discovery;
- framing versus reasoning;
- essence of MVL.

## SV2 - Anchor-Informed Understanding

The five fields are best understood as **evolving target-role state**.

They can exist at the start, but only as orientation:

```text
known / provisional / unknown
```

They are then clarified by the loop and used at the end for target-fit validation.

## Phase 2 - Perspective Checking

### Technical / Logical

If `diagnosis_target` had to be final before Sensemaking, Sensemaking would lose much of its purpose. The loop often starts because the failure or object being explained is ambiguous.

New anchor: fields should have maturity states, not only values.

### Human / User

The user is reacting correctly to the danger of bureaucratic prefill. If an agent asks for all five fields before thinking, it will feel like the system is demanding clarity the human came to MVL to obtain.

New anchor: the record should be readable as "what we currently think," not "what is already settled."

### Strategic / Long-Term

If the fields are treated as evolving state, Homegrown gains durable alignment memory without freezing inquiry too early. If they are treated as mandatory final inputs, Homegrown risks making the loop shallower.

New anchor: v1 should preserve MVL's discovery role.

### Risk / Failure

Two opposite risks exist:

- Too early and fixed: the fields become stale anchors that bias the loop.
- Too late or absent: the loop may drift and no one notices until the finding is wrong.

New anchor: the correct design needs both early provisional visibility and late stabilized validation.

### Resource / Feasibility

The easiest viable model is not to rewrite every discipline. It is to let `_branch.md` or `_state.md` hold a small evolving target record when needed, while each discipline contributes updates naturally.

New anchor: an update protocol may be enough; a new discipline is not needed.

### Definitional / Consistency

The prior finding said `Target Alignment Record` belongs near branch creation. That remains valid only if "near branch creation" means "record the starting orientation and unknowns," not "lock final truth."

New anchor: the previous finding needs a lifecycle refinement, not a reversal.

## SV3 - Multi-Perspective Understanding

The clean model is:

```text
The fields are not pre-loop final inputs.
They are not only post-loop outputs either.
They are evolving target roles whose confidence and maturity change as MVL runs.
```

This preserves both needs:

- start with enough orientation to avoid random thinking;
- let the disciplines discover what was not knowable upfront.

## Phase 3 - Ambiguity Collapse

### Ambiguity: Are the five fields pre-loop inputs?

**Strongest counter-interpretation:**
Yes. A Target Alignment Record should be written before the loop so the loop knows what it is doing.

**Why the counter-interpretation fails (structural grounds):**
Some fields are often unknowable before the loop. `diagnosis_target` may be the output of Sensemaking. `implementation_target` may be the output of Innovation and Critique. Requiring final values before those disciplines run asks the loop to provide its conclusion as input.

**Confidence:** HIGH

**Resolution:**
The fields can be pre-loop orientation, but not final pre-loop inputs.

**What is now fixed?**
Initial field values must allow unknown and provisional states.

**What is no longer allowed?**
Treating the initial Target Alignment Record as settled truth.

**What now depends on this choice?**
Any future doc edit must phrase the record as evolving, not as a fixed questionnaire.

**What changed in the conceptual model?**
The record becomes live state across the loop.

### Ambiguity: Are the five fields outputs of MVL?

**Strongest counter-interpretation:**
Yes. Since MVL discovers them, the fields should only be written at the end.

**Why the counter-interpretation fails (structural grounds):**
If the fields appear only at the end, they cannot prevent drift during the loop. The prior failure happened because the evidence topic became the answer topic. A final-only record would document the problem after the fact but would not guide or check the reasoning while it is happening.

**Confidence:** HIGH

**Resolution:**
The fields are not only outputs. They start as orientation, mature during the loop, and stabilize before conclusion.

**What is now fixed?**
The lifecycle has at least three stages: initial, evolving, final.

**What is no longer allowed?**
Deferring all target-role clarity until `finding.md`.

**What now depends on this choice?**
The final check should compare the finding to stabilized fields, while prior disciplines can update them.

**What changed in the conceptual model?**
The fields become stateful alignment memory, not just final summary.

### Ambiguity: Are the fields the essence of MVL?

**Strongest counter-interpretation:**
Yes. MVL's real job is to identify evidence, target, implementation, and non-targets, then produce the answer.

**Why the counter-interpretation fails (structural grounds):**
MVL also reduces ambiguity, decomposes complexity, generates candidates, adversarially tests them, detects convergence, and loops when no answer survives. The five fields do not describe candidate generation or critique by themselves.

**Confidence:** HIGH

**Resolution:**
The fields are the essence of MVL's target-alignment layer, not the essence of the whole MVL loop.

**What is now fixed?**
Target Alignment should be described as a cross-loop alignment lens.

**What is no longer allowed?**
Replacing MVL's disciplines with the five fields.

**What now depends on this choice?**
Future materialization should integrate with the loop, not become a new main loop.

**What changed in the conceptual model?**
MVL remains the thinking engine; Target Alignment is the loop's target-state instrumentation.

### Ambiguity: Should every discipline alter its output to update these fields?

**Strongest counter-interpretation:**
Yes. If fields evolve through the loop, every discipline should explicitly write its update.

**Why the counter-interpretation fails (structural grounds):**
Every discipline can contribute evidence, but making every discipline own the same record creates duplicate authority and stale overwrite risk. The better mechanism is a shared evolving record with explicit update points when a discipline actually changes target-role understanding.

**Confidence:** MEDIUM-HIGH

**Resolution:**
Disciplines should not all be forced to restate the full record. They should be allowed or required to update it only when their output changes a target role.

**What is now fixed?**
The record's evolution should be event-driven, not ritual-driven.

**What is no longer allowed?**
Mechanical per-discipline restatement with no changed understanding.

**What now depends on this choice?**
Implementation design should focus on update semantics, not on output template duplication.

**What changed in the conceptual model?**
The record is shared loop state, not a section copied into every file.

## SV4 - Clarified Understanding

The five fields are **evolving target-role state**.

They can begin before Exploration, but only as:

```text
known
provisional
unknown
```

They are clarified by the disciplines:

- Exploration improves evidence domain.
- Sensemaking improves diagnosis target and maintenance target.
- Decomposition improves boundary and out-of-scope target.
- Innovation proposes implementation targets.
- Critique tests whether the answer fits the stabilized target roles.

They are then stabilized before CONCLUDE, where the final answer is checked against them.

## Phase 4 - Degrees-of-Freedom Reduction

### Fixed Variables

- The fields are not fixed final pre-loop inputs.
- The fields are not merely final outputs.
- The fields are evolving target-role state.
- The fields are not the whole essence of MVL.
- They are central to MVL's target alignment.

### Eliminated Options

- Require all five fields to be complete before MVL starts.
- Write the five fields only in `finding.md`.
- Add the full field block mechanically to every discipline.
- Treat the fields as a replacement for the MVL/MVL+ pipeline.

### Viable Paths

1. **Target Alignment Record as evolving state**
   - start with known/provisional/unknown values;
   - revise when disciplines discover changes;
   - stabilize before conclusion.

2. **Target Fit Check as final validation**
   - compare the finding against stabilized target roles;
   - if mismatch exists, repair or loop again.

3. **Target Alignment Gate only on failure**
   - use only when the mismatch blocks conclusion or forces branch repair.

## SV5 - Constrained Understanding

The stable lifecycle is:

```text
Before loop:
  Target Alignment Record v0
  values may be known, provisional, or unknown

During loop:
  Exploration/Sensemaking/Decomposition/Innovation/Critique may revise target roles
  revisions should be recorded only when understanding changes

Before CONCLUDE:
  Target Fit Check compares the intended answer against stabilized target roles

If mismatch:
  Target Alignment Gate behavior: repair branch, narrow answer, mark deferred, or loop again
```

## Phase 5 - Conceptual Stabilization

The user's concern is correct. A Target Alignment Record must not be treated as a fully known precondition for MVL.

The earlier finding should be refined this way:

```text
Target Alignment Record is not a form to complete before thinking.
It is a live target-state record that MVL initializes, revises, and stabilizes.
```

This keeps the value of explicitness while preserving the purpose of the cognitive disciplines.

## SV6 - Stabilized Model

Final model:

```text
The five fields are evolving target-role state.
They are initial orientation, discipline-discovered state, and final validation target.
They are central to MVL's target alignment, but not the whole essence of MVL.
MVL still does mapping, understanding, decomposition, generation, critique, and iteration.
```

The change from SV1 is important. SV1 framed the previous answer as too static. SV6 gives the corrected structure: not static fields, not final-only fields, but state that matures through the loop.
