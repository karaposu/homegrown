# Branch: Extended MVL Architecture

## Question
Should the MVL loop be extended beyond pure SIC — with Exploration and Decomposition running as parallel multi-head siblings of Sensemaking in the first phase, with intra-loop chaining of same discipline (e.g., III for expanded innovation surface) permitted when problems demand it, and with orchestration reframed as the enhanced MVL itself rather than a separate capability that grows alongside it?

## Goal
A revised architectural decision that addresses the counter-arguments against discipline_architecture/finding.md:
1. **Parity of E/D with S**: If we always run Innovation without knowing whether it'll produce unique value, why don't we always run Exploration and Decomposition? The "cost" argument that killed ESIC applies identically to I being included in SIC — so either reject both or accept both.
2. **Multi-head parallel first phase**: Instead of SIC, run E + D + S in parallel on the same input, then feed all three outputs into Innovation, then Critique. A richer first phase that covers widening + deepening + structural simultaneously.
3. **Intra-loop same-discipline chaining**: When Innovation runs 3 times sequentially (III), it may naturally expand the innovation surface in ways one Innovation can't. For hard problems, SIIIISEEISC-style sequences may be valuable. The previous "chains between cycles, not within" rule may be wrong.
4. **Orchestration as enhanced MVL**: If orchestration is a capability that grows along the autonomy trajectory, maybe it IS the enhanced MVL itself — not a separate thing. This would confirm we're on the right track (the meta-work we do IS the architecture).
5. **Better analogy**: Unix-for-thinking treats disciplines as tools, which misrepresents them. Find a better analogy for communication.

The user should walk away with: (a) whether the MVL loop structure itself should change (EDS parallel + I + C, or similar), (b) whether intra-loop chaining is allowed, (c) a clear understanding of whether orchestration IS the MVL evolving, (d) a better analogy for the architecture.

## Scope Check
Question covers goal: YES — all five goal points (parity, multi-head, chaining, orchestration=MVL, analogy) are addressed in the question.
