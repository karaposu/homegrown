# Sensemaking: Navigation Depth And Answer Production

## User Input

`devdocs/inquiries/2026-04-28_09-19__navigation_depth_and_answer_production/_branch.md`

## SV1 - Baseline Understanding

The user is right to challenge the prior finding. Saying "MVL+ is answer-producing" made Navigation sound like it does not produce answers, when Navigation clearly answers a real question: what should be considered next?

The initial correction is: Navigation should not be treated lightly. It should not be a shallow option list. It needs enough reasoning depth to understand the current position, blockers, non-obvious routes, weak routes, and risks.

## Phase 1 - Cognitive Anchor Extraction

### Constraints

- MVL+ must remain a bounded loop that can close with a `finding.md`.
- Navigation should not become mandatory inside every MVL+ inquiry.
- Navigation should not be reduced to a lightweight list of next steps.
- Navigation must preserve its output contract: a Navigation Map.
- Selection authority remains separate unless explicitly designed later.

### Key Insights

- The prior distinction "MVL+ produces answers, Navigation maps movement" is too weak because a map can itself be an answer.
- The stronger distinction is not answer versus non-answer. It is question type, artifact type, and authority boundary.
- A discipline can be internally complex. "Separate discipline" does not imply "single shallow operation."
- MVL-like rigor can exist inside Navigation without turning Navigation into the MVL+ command.

### Structural Points

- MVL+ answers inquiry questions and outputs findings.
- Navigation answers movement questions and outputs maps.
- A selector answers choice questions and outputs decisions.
- A meta-loop answers traversal-control questions and outputs sequences of inquiries, branches, merges, or stops.
- These can share cognitive moves without becoming the same component.

### Foundational Principles

- Output contract matters. Different artifacts create different downstream uses.
- Authority boundary matters. Seeing possible moves is not the same as choosing one.
- Depth is a quality requirement, not an identity test.
- The same primitive reasoning pattern can support multiple disciplines with different purposes.

### Meaning-Nodes

- Answer production
- Navigation Map
- Discipline identity
- Internal reasoning depth
- Boundary invocation
- Selector authority
- MVL+ atomicity

## SV2 - Anchor-Informed Understanding

The issue is not whether Navigation answers. It does.

The issue is what kind of answer Navigation is allowed to produce. Navigation should answer the movement-space question: what moves exist, why they matter, what blocks them, how to pursue them, and what confidence each deserves.

Navigation should not answer the commitment question: which move must be executed next. That is selection.

## Phase 2 - Perspective Checking

### Technical / Logical Perspective

If two operations both answer questions, that does not make them the same operation. A compiler and a test runner both produce answers, but their output contracts and downstream authority differ.

Navigation can be a separate discipline if it has:

- a distinct input shape;
- a distinct internal process;
- a distinct output artifact;
- a distinct authority boundary;
- a distinct failure mode profile.

Navigation currently has these. Its spec expects completed cycle outputs and produces a Navigation Map.

**New anchor:** Artifact and authority separate operations more reliably than the broad phrase "answer-producing."

### Human / User Perspective

The user is not asking for mere terminology. The user is warning that if Navigation becomes "just a map," it will be underpowered.

The human need is not only "show me options." It is "help me understand the real next-move terrain well enough that I do not choose a bad path by accident."

**New anchor:** A useful Navigation Map must be earned through serious analysis, not generated by taxonomy assignment alone.

### Strategic / Long-Term Perspective

For Homegrown's larger meta-loop idea, Navigation is strategically important. It can become the perceptual layer that lets the system traverse inquiry space.

If Navigation is shallow, the meta-loop will navigate poorly. If Navigation becomes a selector too early, it will hide values and choose paths without enough evidence.

**New anchor:** Navigation should become deeper before it becomes more autonomous.

### Risk / Failure Perspective

The main risk in the prior finding is demotion by language. Describing MVL+ as answer-producing and Navigation as movement-mapping may cause future agents to treat Navigation as a secondary formatting step.

The opposite risk is over-collapse: treating deep Navigation as proof that Navigation must literally be MVL+. That would erase the Navigation Map contract and make next-move work look like ordinary inquiry work.

**New anchor:** The correction must avoid both shallow Navigation and collapsed Navigation.

### Resource / Feasibility Perspective

The practical next step does not need a full new runner. The Navigation spec can be strengthened first.

A minimal upgrade could add internal Navigation phases:

- position read;
- option-space exploration;
- blocker and dependency sensemaking;
- direction decomposition;
- non-obvious direction generation;
- map critique.

This is cheaper than implementing Navigation as recursive MVL+.

**New anchor:** Strengthen the Navigation discipline before building sub-runners.

### Definitional / Consistency Perspective

The MVL skill says every cognitive task is a SIC loop applied to a different question. This supports the user's point: serious Navigation should contain sensemaking, innovation, and critique pressure.

But it does not force every named discipline to disappear into `/MVL+`. Disciplines can be specialized applications of cognitive primitives with different output contracts.

**New anchor:** Navigation can be a specialized SIC-shaped discipline rather than a generic MVL+ invocation.

## SV3 - Multi-Perspective Understanding

The stabilized meaning is now clearer:

Navigation is answer-producing, but it produces a different class of answer than MVL+.

Navigation should be deep, but depth does not require identity collapse into MVL+.

The correct architecture is a separate Navigation discipline with an explicit internal rigor model. That model may borrow the logic of exploration, sensemaking, decomposition, innovation, and critique, but it ends in a Navigation Map rather than a finding.

## Phase 3 - Ambiguity Collapse

### Ambiguity: Does "answer-producing" distinguish MVL+ from Navigation?

**Strongest counter-interpretation:**

Yes. MVL+ answers substantive inquiry questions, while Navigation only maps options and does not produce a final answer.

**Why the counter-interpretation fails (structural grounds):**

Navigation answers a well-formed question: "What moves are available from here, with what reasons, blockers, guidelines, and confidence?" A Navigation Map can be the final answer to that question. The fact that it does not select one option does not mean it fails to answer.

**Confidence:** HIGH

**Resolution:**

Do not use answer-producing as the distinguishing criterion.

**What is now fixed?**

Navigation is answer-producing in the movement-space domain.

**What is no longer allowed?**

Future findings should not imply that Navigation is merely non-answering support work.

**What now depends on this choice?**

The prior finding must be corrected in language and architecture framing.

**What changed in the conceptual model?**

The main separation shifts from answer/non-answer to question domain, artifact, and authority.

### Ambiguity: If Navigation needs exploration, sensemaking, decomposition, innovation, and critique, is it just MVL+?

**Strongest counter-interpretation:**

Yes. If Navigation uses all MVL+ cognitive moves, then it is simply MVL+ with a Navigation prompt.

**Why the counter-interpretation fails (structural grounds):**

An operation's internal reasoning pattern is not identical to its system role. Navigation can use exploration-like and critique-like substeps while still having a Navigation-specific input, output, telemetry, and failure modes. The output is a Navigation Map, not a finding. The authority is enumeration and evaluation of move-space, not general inquiry closure.

**Confidence:** HIGH

**Resolution:**

Navigation can be internally MVL-like without being the MVL+ command.

**What is now fixed?**

Internal depth does not erase discipline identity.

**What is no longer allowed?**

Do not collapse all deep disciplines into generic MVL+ solely because they use similar reasoning moves.

**What now depends on this choice?**

Navigation can be upgraded in its own spec rather than replaced by a protocol wrapper.

**What changed in the conceptual model?**

Discipline identity is defined by role, output, and authority, while cognitive primitives can be shared.

### Ambiguity: Should Navigation recommend one direction or only enumerate?

**Strongest counter-interpretation:**

If Navigation is answering "what to do next," it should recommend the best next move.

**Why the counter-interpretation fails (structural grounds):**

Choosing a move requires a value function: priorities, risk tolerance, branch budget, user preference, time horizon, and evidence from prior outcomes. Navigation's current artifact contains confidence and guidelines, but it does not contain an explicit value function. Without that, selection would be hidden inside map generation.

**Confidence:** HIGH

**Resolution:**

Navigation answers what moves exist and how they look. A selector or human answers which move to execute.

**What is now fixed?**

Navigation may rank confidence, but it should not silently choose.

**What is no longer allowed?**

No hidden auto-launching or disguised selection inside Navigation.

**What now depends on this choice?**

Meta-loop design must include a separate selector layer.

**What changed in the conceptual model?**

Navigation becomes serious perception, not decision authority.

### Ambiguity: What does "discipline" mean here?

**Strongest counter-interpretation:**

A discipline is a simple step in a pipeline, so if Navigation is complex it should be a runner or protocol instead.

**Why the counter-interpretation fails (structural grounds):**

Existing disciplines are already complex. Exploration has scan-signal-probe cycles. Sensemaking has multiple phases. Critique has structured adversarial evaluation. A discipline is not defined by simplicity; it is defined by a stable cognitive operation and output contract.

**Confidence:** HIGH

**Resolution:**

Navigation can be a complex discipline.

**What is now fixed?**

Complexity alone does not move Navigation into meta-loop or protocol status.

**What is no longer allowed?**

Do not treat "discipline" as synonymous with "lightweight phase."

**What now depends on this choice?**

Navigation can receive internal quality improvements without changing its architectural category.

**What changed in the conceptual model?**

The right question becomes how to deepen Navigation, not whether depth disqualifies it as a discipline.

## SV4 - Clarified Understanding

The prior finding should be corrected.

Navigation is not non-answer-producing. It produces an answer to a next-move question. The valid distinction is that Navigation's answer is a structured possibility map, not a general inquiry finding or an execution decision.

Navigation should stay separate, but it should be made deeper. A serious Navigation discipline should internally perform enough exploration, sensemaking, decomposition, innovation, and critique to avoid shallow enumeration.

## Phase 4 - Degrees-of-Freedom Reduction

### Fixed Variables

- Navigation is answer-producing in the movement-space domain.
- Navigation's output should remain `navigation.md` or an equivalent Navigation Map.
- Navigation should not become a required MVL+ stage.
- Navigation should not silently select or auto-launch the next inquiry.
- Navigation needs internal depth requirements.

### Eliminated Options

- Navigation as a shallow taxonomy assignment step.
- Navigation as "not answer-producing."
- Navigation as a generic MVL+ protocol replacement.
- Navigation as hidden selector.

### Viable Paths

1. Keep Navigation as a separate discipline and update its spec to include deeper internal reasoning.
2. Add a Navigation invocation protocol later if agents forget how to call it.
3. Allow rare complex Navigation inquiries to run MVL+ for subquestions, while preserving the final Navigation Map output.
4. Feed Navigation Maps into future meta-loop selection.

## SV5 - Constrained Understanding

The best current model is:

```text
MVL+ answers an inquiry question -> finding.md
Navigation answers a movement-space question -> navigation.md
Selector/human answers a commitment question -> next chosen move
Meta-loop answers traversal-control questions -> sequence, branch, merge, or stop
```

Navigation should be deep because movement-space can be complex. It should be a disciplined reasoning process, not a formatting step.

The system should update its language: Navigation is not merely "movement-mapping" as a contrast to answer production. It is answer-producing movement-space mapping.

## Phase 5 - Conceptual Stabilization

The user point is correct and important. The prior finding got the broad architecture mostly right, but one explanation was misleading.

The corrected model is:

- Navigation is a separate boundary discipline.
- Navigation is answer-producing.
- Its answer is a Navigation Map, not a finding.
- Its domain is possible movement from the current state, not the original inquiry question.
- Its authority is to enumerate, analyze, confidence-rank, and guideline possible moves.
- It should not select or execute a move unless a separate selector layer is added.
- Its internal process should be strengthened so it can explore, sensemake, decompose, innovate, and critique the next-move space.

## SV6 - Stabilized Model

Navigation should be understood as a deep, answer-producing boundary discipline.

It answers: "Given where this inquiry or project now stands, what moves are possible, what blocks them, why might each matter, and how should each be pursued?"

It does not answer: "Which move will the system now execute?"

This differs from SV1 by replacing the weak answer/non-answer distinction with a stronger four-part distinction:

1. **Question domain:** inquiry answer vs movement-space answer.
2. **Artifact:** `finding.md` vs Navigation Map.
3. **Authority:** conclusion vs enumeration/evaluation vs selection.
4. **Internal rigor:** Navigation should have MVL-like reasoning pressure without becoming the MVL+ command.
