# Branch: Route Memory Review Trigger Boundary

## Question

Is it natural for Route Memory Review to run only for "big" or project-level Navigation, or should Navigation always include a route-memory check regardless of whether the input is bounded or broad?

## Goal

A good answer should determine the correct trigger boundary for Route Memory Review, explain whether bounded/project-level is the wrong distinction, define when a full `route_memory_review.md` is generated, and clarify what every Navigation run should record about route memory.

## Scope Check

Question covers goal. The question asks whether the current trigger model is structurally natural and requires a replacement trigger model if it is not.
