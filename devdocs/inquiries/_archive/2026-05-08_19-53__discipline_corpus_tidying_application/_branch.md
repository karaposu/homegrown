# Branch: Discipline-corpus tidying application

## Question

What is the complete per-discipline tidying work needed to apply the prior inquiry's emerging-pattern verdicts (the **Step Refinement** primitive now spec'd at `enes/step_refinement.md`; the **Discipline Output Contract** proposal) across the 5 MVL+ relevant disciplines (`explore`, `sense-making`, `decompose`, `innovate`, `td-critique`) — including (a) scattered instances the prior inquiry's audit might have missed, and (b) any reframing of the patterns themselves that application reveals?

## Goal

Produce per-discipline tidying plans (5 plans, one per MVL+ discipline) with concrete edits the user can execute as routine maintenance. Plus surface any pattern-level reframings that application reveals — particularly whether the prior inquiry's "descriptive only; existing rules align organically when next touched" reading of phase-fit was correct, or slid too far conservative relative to the user's "brushing teeth" framing (where the deliverable IS the cleaning, not just description of what to clean).

The user should be able to read each per-discipline plan and decide: *"yes, do this now"* or *"yes but later"* or *"no, the tidying doesn't earn its keep here."* Each plan should distinguish surgical edits (visual marker on already-clean refinements) from lifting work (rules that have the shape but are positioned ambiguously or miss an element) from leave-alone (things that look like Step Refinement candidates but aren't, or where application would damage discipline-specific structure).

## Scope Check

Question covers goal. The question asks for the tidying work; the goal asks for tidying plans. Both at the per-discipline level + a pattern-level reframing surface.

**Specific-vs-pattern:** mixed by design. Per-discipline tidying is the specific application work; pattern-level reframing surfaces what application reveals about the patterns themselves. Both are intentional because the prior inquiry's gap was that it stopped at the pattern level without testing application.

**What this inquiry is NOT doing:** it is not executing the tidy edits (those are user-driven follow-ups under `homegrown/protocols/artifact_materialization.md`). It is producing the plans + the reframing.
