# Finding: inquiry_folder_organization

## Question

**The branch question:** How should `devdocs/inquiries/` be organized so the user (and any future agent reading the folder) can tell which inquiries are recent and which are stale at a glance — and is a compact datetime-prefix + archive-folder convention the best approach, or does a better alternative exist?

**The branch goal:** A concrete, applicable organizational convention with: (a) at-a-glance recency, (b) at-a-glance staleness, (c) smallest naming overhead, (d) concrete archival criterion, (e) migration plan for the 55 existing inquiries, (f) honest comparison of datetime-prefix against alternatives. The user wants to act on the answer; the output is a buildable spec, not a discussion.

**Project context the reader needs to know up front:**

- This project (root: `/Users/ns/Desktop/projects/native/`) is a homegrown system of "thinking disciplines" — small specialized skills (`/sense-making`, `/innovate`, `/td-critique`, `/explore`, `/decompose`, `/comprehend`, `/reflect`, `/navigation`) plus loop runners `/MVL` (Sensemaking → Innovation → Critique) and `/MVL+` (Exploration → Sensemaking → Decomposition → Innovation → Critique).
- Each "inquiry" is a folder under `devdocs/inquiries/` containing a `_branch.md` (the question), a `_state.md` (progress + status), discipline outputs (`exploration.md`, `sensemaking.md`, etc.), and — when the inquiry is concluded — a `finding.md` produced by the CONCLUDE protocol (`homegrown/protocols/conclude.md`).
- Today (2026-04-27): 55 inquiries exist in the folder. Mtimes span Apr 14 – Apr 27. Statuses (per `_state.md` `## Status` field) include `ACTIVE`, `COMPLETE`, `SUPERSEDED`. Six folders have no Status field at all.
- The user observed the folder is hard to navigate visually — there's no convention surfacing "what's recent and what's relevant?" — and proposed two specific candidates (datetime prefix on folder names + an archive subfolder), then explicitly asked whether something better exists.

## Finding Summary

- **Don't rename folders.** Folder names are load-bearing identifiers — `finding.md` files across the project reference each other by folder name (`devdocs/inquiries/wayfinding_navigation_unification_check/finding.md`, etc.). Renaming 55 folders would require updating cross-references across ~30+ files; missing one breaks traceability silently. The cost is hidden in the user's original framing.

- **Surface the existing data instead.** The information the user wants visible is *already encoded*: `_state.md` `## Status` field gives staleness, filesystem mtime gives recency, `finding.md` presence gives completion. The visibility problem is *surface*, not *substance*. The recommendation is a small Bash script (`tools/inq`) that renders this data, plus an auto-generated `devdocs/inquiries/README.md` for ambient editor/GitHub visibility.

- **One renderer, two outputs.** `tools/inq` has a single rendering function that consumes a shared data model (8 fields per inquiry: `name`, `status`, `mtime`, `has_finding`, `flow`, `created`, `next_disc`, `relationships`). Default invocation prints terminal-plain columns; `--regen` writes the same data as Markdown to `devdocs/inquiries/README.md`. No duplicate rendering logic; the file IS the helper's output piped to disk.

- **Archive is a STATUS, not a LOCATION.** Add `ARCHIVED` to the Status enum (alongside `ACTIVE | COMPLETE | SUPERSEDED`). Transition is **never-auto** — user manually marks an inquiry ARCHIVED when they decide it's no longer reference material. The helper's `--archive-candidates` flag lists COMPLETE inquiries older than 30 days for review; the user-decides principle is preserved. Folders never move; cross-references stay stable.

- **Format is hybrid sectioned with collapsibles.** `README.md` pins ACTIVE inquiries to the top, shows Recently Completed (last 14 days) below, and collapses Older Completed / Superseded / Archived behind `<details>` blocks. This satisfies the user's primary need ("what's recent and active?") at a glance without losing access to historical inquiries.

- **One-time hygiene pass: mark all 6 missing-Status folders SUPERSEDED.** Per-folder reasons committed below. No `ACTIVE` assignments warranted — all 6 are partial scaffolds, abandoned mid-pipeline, or work-done-outside-the-formal-inquiry. Total cleanup cost: ~15 minutes.

- **Total implementation effort: ~70-90 minutes.** Slightly over the sensemaking estimate of 45-60 minutes due to the 8-field schema and hybrid sectioned format. A leaner Assembly 4 alternative (5 fields + single table) cuts to ~30-40 minutes if the user prefers strict minimum. Both architectures are forward-compatible; switching renderers later is cheap (~15 minutes).

## Finding

### 1. The architecture (one paragraph)

Build a single Bash script at `tools/inq` that walks every folder under `devdocs/inquiries/`, reads `_state.md` (Status, History, Pipeline, Next Discipline) plus filesystem mtime plus `finding.md` presence, and produces a structured view of all inquiries. By default it prints terminal-aligned plain-text columns (no Markdown noise) sorted by status group then mtime descending, ARCHIVED filtered out. With the `--regen` flag, the same renderer wraps its output in Markdown table syntax with hybrid sectioning (Active pinned at top; Recently Completed visible; Older Completed / Superseded / Archived collapsed under `<details>` blocks) and writes the result to `devdocs/inquiries/README.md`. A header line in the generated file declares "Auto-generated; do not edit manually." Status is an enum (`ACTIVE | COMPLETE | SUPERSEDED | ARCHIVED`); the new `ARCHIVED` value is set manually only — the helper's `--archive-candidates` flag lists archive-eligible inquiries (COMPLETE older than 30 days) but does not transition them. No folders are ever renamed; no folders are moved into subfolder containers; cross-references between findings remain valid.

### 2. The data model (8 fields)

Each inquiry is represented by:

| Field | Source | Example |
|---|---|---|
| `name` | folder name | `inquiry_folder_organization` |
| `status` | `## Status` field in `_state.md` | `ACTIVE` |
| `mtime` | filesystem mtime of the folder | `04-27` |
| `has_finding` | `finding.md` exists in the folder | `Y` / `-` |
| `flow` | `## Flow-type` field in `_state.md` | `extended` / `classic` |
| `created` | first History entry's date in `_state.md` | `04-27` |
| `next_disc` | `## Next Discipline` field | `Innovation` / `—` |
| `relationships` | first 1-2 entries from `## Relationships` section, compactly | `REFINES X; RELATED Y` |

If a field is absent from `_state.md`, the helper renders it as `(none)` and emits a one-line warning at the top of output ("WARNING: 1 inquiry has no `## Status` field — run hygiene pass") so missing data is surfaced, not silently dropped.

### 3. The renderer (one function, two output modes)

```bash
# tools/inq — pseudocode

function render(inquiries, output_mode):
    rows = sort_by(status_group, mtime desc)
    rows = filter(default: exclude ARCHIVED)

    if output_mode == "terminal":
        print column-aligned rows; no Markdown pipes; one section
    elif output_mode == "markdown":
        emit "# Inquiries Index" header + "Auto-generated" notice
        emit "## Active (N)" table — ACTIVE rows
        emit "## Recently Completed (last 14 days, N)" table — COMPLETE rows mtime ≥ 14d
        emit "## Older Completed (collapsed)" — <details>...COMPLETE rows mtime < 14d</details>
        emit "## Superseded (collapsed)" — <details>...SUPERSEDED rows</details>
        if --all flag: emit "## Archived (collapsed)" — <details>...ARCHIVED rows</details>
```

Default invocation calls `render(*, terminal)`. The `--regen` flag calls `render(*, markdown)` and writes to `devdocs/inquiries/README.md`.

### 5. Freshness-check warning

On every invocation (terminal or `--regen`), the helper compares `README.md`'s mtime to the most recent inquiry-folder mtime. If `README.md` is older, prints a warning at the top:

```
WARNING: README.md may be stale (last regenerated 04-27; latest inquiry activity 04-29). Run `tools/inq --regen` to refresh.
```

This is the trigger mitigation — the helper is the freshness check; no separate cron / hook needed. If the user wants automatic regen, they can install a git pre-commit hook that runs `tools/inq --regen` and stages `README.md` if it changed; this is user-opt-in, not default.

### 6. The Status enum (extended)

Current convention: `ACTIVE | COMPLETE | SUPERSEDED`.
After this finding: `ACTIVE | COMPLETE | SUPERSEDED | ARCHIVED`.

**Semantics:**

- **ACTIVE** — pipeline in progress; `_state.md` `## Next Discipline` is non-empty.
- **COMPLETE** — pipeline finished; `finding.md` exists.
- **SUPERSEDED** — pipeline was started but rendered moot by a later finding; the supersedor is named in the `_state.md` `## Status` block (per the convention established in this session).
- **ARCHIVED** — pipeline was COMPLETE; user has decided this finding is no longer active reference material. Distinguished from COMPLETE only by user judgment. Manually transitioned only.

**Transition mechanics:** the user edits the `## Status` field in `_state.md`. No automation transitions a finding's status. The `--archive-candidates` flag *suggests* archive-eligibility; the user *decides*.

## Reasoning

### Why this finding over alternatives

**The user's stated proposal — datetime prefix on folder names + archive subfolder — was rejected primarily because folder-name renames break cross-references densely throughout the project.** The cross-references include `refines:` frontmatter pointers, body-text references (e.g., "supersedor: `devdocs/inquiries/X/finding.md`"), and Relationships sections in `_state.md` files. Renaming 55 folders requires updating ~30+ files; missing one breaks traceability silently. The user's framing didn't price this cost. The alternative (auto-INDEX + helper) achieves the same visibility outcome at lower cost without touching folder identity.

**The helper-only alternative (no persistent INDEX.md) was killed because sensemaking SV6 explicitly resolved in favor of persistent visibility.** Sensemaking's argument was that "open it in any editor, on GitHub, or in a terminal pager — visibility from anywhere" requires a file, not a tool invocation. Critique sustained that argument; re-litigating it was post-hoc. The "always live, no freshness problem" intuition was preserved by adding the helper's freshness-check warning instead.

**Auto-archive at 30 days was killed because it creates surprise transitions.** Sensemaking's principle ("status changes should be deliberate") is structural; any automation that transitions an inquiry's status without user input violates it. The neglect concern (manual archive will be forgotten) was addressed by the `--archive-candidates` flag, which surfaces archive-eligible inquiries without transitioning them.

**The CONCLUDE-time hook was killed because it bloats CONCLUDE.** CONCLUDE's responsibility is compiling a finding from one inquiry's SIC outputs; regenerating an index of all inquiries is a different scope. Coupling the two violates the single-responsibility principle and creates side-effects that future CONCLUDE edits would need to remember. The freshness-check warning + optional git pre-commit hook are cleaner alternatives.

**The 5-field minimal data model was set aside in favor of the 8-field richer model.** Marginal cost (still one line per inquiry) is small; diagnostic value (what's the next discipline? what's blocked?) is high. If implementation cost matters more than diagnostic richness, the user can fall back to Assembly 4 (5 fields + single table) at ~30-40 minutes total.

**The "single table" format was set aside in favor of "hybrid sectioned with collapsibles."** The user's primary need is "what's active?" — pinning ACTIVE to the top satisfies it directly; sectioned format with collapsibles preserves access to historical inquiries without scrolling noise. The complexity delta (~10-15 minutes more implementation) is justified by the readability win.

**The `INDEX.md` and `_index.md` filenames were set aside in favor of `README.md`.** GitHub auto-renders README.md on folder pages; the alternatives don't get this rendering. The "is this safe to edit?" concern is mitigated by an in-file header warning.

### Significant kills

- **Group A (datetime-prefix rename):** killed by cross-reference cost.
- **Group B (status-subfolder reorganization):** killed by recurring rename cost on every status change.
- **Assembly 2 (helper-only, no INDEX.md):** killed by sensemaking SV6's persistent-visibility resolution.
- **Auto-archive at 30 days:** killed by deliberate-transition principle.
- **CONCLUDE-time index regeneration:** killed by single-responsibility violation.
- **Cosmetic-INDEX.md filename:** killed in favor of README.md (GitHub rendering).

### Significant SURVIVE

- **Persistent INDEX file (as `README.md`):** sustained because ambient editor/GitHub visibility requires a file, not a tool.
- **Single renderer with two output modes (terminal-plain, markdown-table):** sustained as the structural-cleanest unification of the helper and the file generator.
- **Status-not-location archive:** sustained because location-based archive breaks cross-references.
- **Hybrid sectioned format with collapsibles:** sustained because it serves the user's primary need (Active pinned) directly.
- **Per-folder hygiene assignments (all 6 → SUPERSEDED):** sustained because each has a specific structural reason; no `ACTIVE` assignments warranted on inspection.
