## Phase 2 — Perspective Checking

### Technical / Logical

- Multi-head architecture: each head consumes the enumeration and picks its own direction. There is no global "pick ONE." Single-direction selection at iteration boundary is a single-head artifact.
- BUT: cross-head integration (e.g., Head 2's findings invalidate kill conditions in Head 1's branch) IS a real operation. /wayfinding's RECONSIDER sub-actions handle this. It's not selection-at-iteration-boundary; it's verdict-integration-across-heads. This substance needs to live somewhere.
- The Three-Layer Awareness Model integrates temporal awareness:
  - **Present:** position, heading, reachability — maps to /navigation's existing scope check + confidence levels + a new reachability/gates input.
  - **Trend:** velocity, acceleration, goal distance — partially in /navigation's telemetry-routing (RE-RUN DEEPER for thin telemetry) and scope-check (WIDEN for goal mismatch). Some absorption needed.
  - **Memory:** kill conditions, survival conditions, near-misses, dependencies — partially in /navigation's REVISIT type but without sub-actions, threshold, or dependency-tracking. Significant absorption needed.
- Reachability/Gates: at multi-head, gates determine which heads can dispatch where. Operationally important. Belongs in /navigation as part of pre-enumeration assessment.

### Human / User

- The user wrote /wayfinding (or extended the older meta-search.md). They wrote /navigation.md later. Both contain their thinking; the more-recent /navigation.md represents the more-current architectural position.
- The user's empirical observation: /wayfinding is "not used so much now."
- The user's challenge phrases as a question ("wayfinding can be deleted?") not a directive — open to the answer. Their intuition is sound on the central direction.
- The user has NOT yet thought through the migration scope. Sensemaking should surface this as the next decision point.

### Strategic / Long-term

- Multi-head: enumeration is the primitive. Heads dispatch from enumeration. Selection-as-pick-one is a single-head artifact and gets superseded.
- Year-long execution plans: state across iterations matters; RECONSIDER (memory layer) becomes operationally central. The substance is multi-iteration-aligned, not just multi-head-aligned.
- Autonomy-ladder Level 2-3+: system selects autonomously from enumerations; needs to know when to RECONSIDER past verdicts. /navigation's autonomy-ladder integration (auto-derivable vs human-judgment split) is the natural extension point.
- Baldwin cycles: predictions calibrated against outcomes. The calibration is a memory-layer operation (track predictions, observe outcomes, learn the delta). RECONSIDER's threshold logic is structurally similar.

Strategically: /wayfinding's substance (memory layer, threshold, layer priority) becomes operationally CENTRAL at higher autonomy levels. Multi-head doesn't make it obsolete — it makes it more important. But the locus of the substance can change. It can live in /navigation (one bigger discipline) or in a separate discipline. The cleaner architectural endpoint is unification (one discipline owns iteration-boundary cognition).

### Risk / Failure

- **If we delete /wayfinding without migration:** lose architectural-prep slots for higher-autonomy and multi-head scenarios. Hard to recover when needed. NOT acceptable.
- **If we keep both:** contradiction in shipped text persists. Future agents/users hit ambiguity. The two specs drift over time. Maintenance debt grows. NOT acceptable as final state.
- **If we delete with migration (Path A):** /navigation grows by ~150-250 lines. Pure-enumeration framing of /navigation is muddied (now contains some selection-related substance like reachability and threshold logic). Acceptable cost; clean endpoint.
- **If we refactor to thin wrapper (Path C):** /wayfinding becomes ~80-150 lines, focused on selection-from-enumeration. Maintenance of two coupled specs. Unclear when (if ever) the wrapper would be deleted. Risk of permanent in-between state.

Risks favor Path A (migrate-then-delete). The "pure enumeration" purity loss in /navigation is the cost; it's acceptable because pure enumeration without temporal awareness is incomplete at scale anyway.

### Resource / Feasibility

- **Path A.1 (full migration):** ~150-250 lines added to /navigation; ~5 cross-references updated; /wayfinding deleted (~381 lines). Net: ~130-230 lines fewer in the project. ~1-2 hours of focused editing. Coordination with prior audit's punch list (one variant change, one reversal).

Other features (Three-Layer Awareness explicit framing, threshold self-adjustment, layer priority) deferred or accepted as loss. ~30-45 minutes editing.
- **Path C (thin wrapper):** /navigation unchanged or +20 lines; /wayfinding refactored to ~80-150 lines; cross-references mostly unchanged. ~1 hour. But leaves two coupled specs to maintain.

A.2 is the cheapest path that honors the structural verdict. A.1 is the cleanest endpoint. C is a fallback if user wants gradual transition.
