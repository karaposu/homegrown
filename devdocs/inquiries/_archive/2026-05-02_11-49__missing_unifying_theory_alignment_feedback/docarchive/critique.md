## Phase 1 - Fitness Landscape

### Viable Region

The viable answer should identify a missing theory layer that:

- sits above `alignment_control.md` but below future runners;
- explains dynamics over time;
- treats existing operations as local lifecycle mechanisms;
- supports later operationalization without requiring it now.

### Dead Regions

- Another sibling feedback protocol as the answer.
- Expanding `alignment_control.md` immediately into a broad theory document.
- Building a generic route selector now.
- Treating meaningful traversal as the whole missing layer.
- Waiting for records without naming the kind of thing to observe.

### Boundary Regions

- `signal_lifecycle.md` as narrower subcomponent.
- `self_maintenance.md` as future parent program.
- `selector.md` as later implementation candidate.

## Phase 2 - Candidate Evaluation

### Candidate A: `alignment_dynamics.md`

**Prosecution:** The term may be abstract. It might duplicate `alignment_control.md` unless carefully scoped. It could become a theory sink if it tries to explain everything.

**Defense:** It directly names the missing dimension: how alignment changes over time. It preserves existing operations and explains their positions in a lifecycle. It can remain a theory note, not a contract or runner.

**Collision:** Defense wins if the artifact is explicitly scoped to state, signals, evidence maturity, control actions, outcomes, and calibration.

**Verdict:** SURVIVE

**Constructive output:** Create `enes/alignment_dynamics.md` as a theory note before patching contracts or protocols.

### Candidate B: `cognitive_control.md`

**Prosecution:** This is probably the more general scientific name, but it may expand too far into attention, valuation, autonomy, controller design, and consciousness.

**Defense:** It accurately captures the missing control problem: sensing state, choosing action, observing outcome, and learning.

**Collision:** The concept survives as framing, but the file name and first artifact should be narrower.

**Verdict:** REFINE

**Constructive output:** Use "cognitive control" inside `alignment_dynamics.md`, not as the first top-level artifact.

### Candidate C: `signal_lifecycle.md`

**Prosecution:** Too narrow. It explains signal birth/maturity/decay, but not alignment state, control action, outcome calibration, or existing operation interfaces.

**Defense:** Signal lifecycle is a concrete missing sub-theory and would immediately clarify route confusion.

**Collision:** Defense wins as a section, not as the parent.

**Verdict:** REFINE

**Constructive output:** Put signal lifecycle inside `alignment_dynamics.md`.

### Candidate D: `self_maintenance.md`

**Prosecution:** Too broad for the current question. It would pull in recovery, materialization, regression checks, memory, autonomy policy, and consciousness. High risk of building more than the system can carry.

**Defense:** The user's deeper goal is a self-maintaining cognitive system. This is the true long-term parent.

**Collision:** Defense is true long-term, but prosecution wins for now. The immediate missing theory is a sublayer of self-maintenance.

**Verdict:** DEFER

**Constructive output:** Treat self-maintenance as the broader program that will consume alignment dynamics later.

### Candidate E: `selector.md`

**Prosecution:** Navigation explicitly does not choose, and selection is a real gap. But selector design needs signal maturity and control policy first. Otherwise it becomes arbitrary priority logic.

**Defense:** The absence of selection is a major reason feedback logic feels manual.

**Collision:** Selector is real but downstream. It needs alignment dynamics as input.

**Verdict:** DEFER

**Constructive output:** Revisit after at least one alignment-dynamics theory note and several route records.

### Candidate F: Expand `alignment_control.md` Now

**Prosecution:** `alignment_control.md` intentionally avoids becoming a universal runner or broad theory. Expanding it now would overload a useful thin contract.

**Defense:** Keeping everything in one contract might reduce scattered concepts.

**Collision:** Prosecution wins. Thin contracts should not become theory documents.

**Verdict:** KILL

**Seed extracted:** Let a theory note mature first, then import only stable field changes into `alignment_control.md`.

### Candidate G: Wait For Records Before Naming Anything

**Prosecution:** The user explicitly asks what kind of thing might be missing. Waiting produces no conceptual target and leaves future records unguided.

**Defense:** It avoids premature theory from too little evidence.

**Collision:** Defense is valid as a guardrail but incomplete. We can name the missing kind while deferring automation and hard schema changes.

**Verdict:** REFINE

**Constructive output:** Write a light theory note, then test it against real records.
