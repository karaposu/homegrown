## Finding

### 1. Why the branch.md surface has higher leverage than Sensemaking

The cascade in the prior `2026-05-08_06-30` loop ran:

```
A (branch framing pre-encoded narrow scope at Diagnostic Constraints line 31)
  → C (Exploration Region A characterized failure shape narrowly; inherited from A)
  → B (Sensemaking Phase 1 Insight #3 stabilized narrow scope; Ambiguity Collapse #5
        used it as structural argument WITHOUT M6 testing) ← LOAD-BEARING + PROXIMATE
  → propagation through Decomposition / Innovation / Critique
  → D (Critique 9 dimensions missing scope-adequacy; detective gap)
  → finding.md narrow scope
```

The prior `2026-05-08_08-15` LOOP_DIAGNOSE finding identified W1 (Sensemaking M6 extension) as PRIMARY ACTIONABLE because B is the load-bearing proximate cause. But B is the proximate cause precisely because Sensemaking is the discipline whose job is to test interpretive choices — and Sensemaking has to do this mid-cascade, after frame-momentum has already accumulated from upstream Exploration + solve-mode momentum from goal.

The user's question raises a structurally important alternative: prevent the wrong frame from forming at the source. Branch.md is the source. Three structural reasons branch.md beats Sensemaking on leverage:

**Reason 1 — Upstream of all 5 disciplines.** Edit at frame-formation prevents the cascade root. By the time Sensemaking runs, Exploration has already collapsed types under one shape (the prior 06-30 Exploration Region A line 16). Catching at branch.md happens BEFORE any discipline runs.

**Reason 2 — Single canonical home.** The `_branch.md` template is defined in `homegrown/MVL+/SKILL.md` (lines 76-87). One edit at this single canonical home propagates to all future MVL+ root inquiries. By contrast, the M6 cross-cutting rule lives at `sensemaking.md` and applies discipline-by-discipline; a per-discipline-execution intervention.

**Reason 3 — Mechanical template fill-in.** The runner has to fill in the Scope Check section every time `_branch.md` is created. The act of filling in surfaces the question structurally — the runner cannot skip it without leaving a blank section. By contrast, M6 is a judgment-dependent rule: it fires when the runner recognizes a load-bearing concept. The prior 06-30 failure was exactly recognition-cue failure (Insight #3 was load-bearing but the runner didn't recognize it for M6 testing).

Mechanical surfaces are more reliable than judgment-based ones. The user's instinct that branch.md is the cleaner intervention point is structurally correct.

### 2. Why composition (Edit 1 + Edit 2) beats replacement

The user's stronger framing — *"instead of frame-level change at Sensemaking"* — suggested replacing W1 with a branch.md fix. This is partially incorrect.

The two surfaces catch different cases:

- **Edit 1 (branch.md) catches:** when the Question or Goal text explicitly points at specific examples. Examples: *"the 10 observed failures from inquiry X"*, *"the corpus chains 1-7"*, *"these specific instances"*. The recognition cue is at the runner-template-fill-in step.

- **Edit 2 (Sensemaking M6) catches:** when Sensemaking pins down a key concept at Phase 1 anchor extraction built from specific examples — even when the branch was framed broadly. Example: branch broadly framed ("fix the X discipline"), but Sensemaking's Phase 1 / Key Insights generates an insight ("Y is the load-bearing case") built from whatever specific examples happen to be on the table.

These cases are correlated but non-identical. Edit 1 prevents most cascades from forming; Edit 2 covers residual cases where a correctly-framed branch still gets narrowed at Sensemaking.

Composition gives defense-in-depth at marginal cost. The aggregate is ~10-15 lines across 2 files — smaller than the original Component B's ~25-30 lines + per-output scan overhead. Same composition discipline the user has consistently endorsed across earlier chains (Component A + W1 in the original `2026-05-07_22-10` finding's pattern; Component A + Edit 2 in the prior `2026-05-08_07-15` finding).

### 3. Edit 1 — concrete wording

**Where:** `homegrown/MVL+/SKILL.md`, Scope Check section of the `_branch.md` template (lines 84-87). The current template:

```markdown
## Scope Check
[compare the question's scope to the goal's requirements. Does the question, if answered perfectly, cover everything the goal asks for? If YES: "Question covers goal." If NO: "Question covers goal: NO — goal includes [X, Y] but question only addresses [Z]. Consider widening to: [proposed wider question]."]
```

**Edit:** append the following inline to the existing Scope Check bracket (one paragraph; ~6 lines added):

```markdown
Specific-vs-pattern check: if the Question or Goal points at specific examples (e.g., "the 10 observed failures from inquiry X", "these 7 chains", "these specific instances"), explicitly state whether the inquiry should address JUST THOSE EXAMPLES or the BROADER PATTERN those examples illustrate. Default: address the broader pattern unless the user has explicitly scoped to the specific examples. If both readings are plausible, present both to the user before proceeding (per the existing scope-widening flow above).
```

**Mechanism:** the runner reads the template when writing `_branch.md`. The Scope Check section now has two clauses: (a) sufficiency check (Q covers G?); (b) specific-vs-pattern check (does G target the specific examples or the broader pattern?). The runner has to address both clauses when filling in the section. If the goal references specific examples, the runner explicitly states which target. If both readings are plausible, the existing user-presentation flow at MVL+/SKILL.md line 87 fires — the runner presents alternatives to the user, who decides.

**Over-firing mitigation:** the existing scope-widening flow at line 87 says: *"If the scope check flags a gap, present the proposed wider question to the user before proceeding. The user decides whether to widen or keep the original scope."* This bounds the over-firing risk structurally. When the specific-vs-pattern sub-check flags ambiguity, the runner doesn't unilaterally widen — they ask the user. Legitimate user-narrow-scope requests (e.g., "fix THIS specific instance") are preserved by user confirmation.
