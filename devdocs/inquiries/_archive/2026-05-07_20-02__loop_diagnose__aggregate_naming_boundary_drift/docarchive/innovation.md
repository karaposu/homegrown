## Phase 2 — Generate (7 Mechanisms × 3 Variations Each)

### Mechanism 1: Combination (Generator)

**Generic variation:** Combine **U1's Phase-5 Sensemaking validation rule** + **TP3's prosecution-depth rule** to produce a unified vocabulary-validation framework that fires at both Sensemaking and Critique stages.

**Focused variation:** Combine **U1's Phase-5 user-language validation predicate** + **chain 6's T1 discoverability check predicate** to produce a *paired Sensemaking sub-rule* covering BOTH discoverability AND user-language validation at the same Phase 5 surface. This makes T1 + U1 explicit sister rules under a single Phase 5 vocabulary-justification umbrella.

**Contrarian variation:** Combine **the chain-1 "Terms Used Here" pattern** + **TP3's prosecution-depth rule** to produce a CONCLUDE-stage mandate that EVERY finding must include a "Terms Used Here" section listing each load-bearing concept name + (a) user input verbatim/paraphrase OR (b) explicit justification. This bypasses Sensemaking and operates at CONCLUDE only.

Surviving combination output: **Focused variation** (T1 + U1 paired Sensemaking sub-rule). Pairs cleanly with already-promoted T1; no consolidation trigger; HIGH composability with Phase 5.

### Mechanism 2: Absence Recognition (Generator)

**Generic variation:** What's missing from the corpus is a "vocabulary justification artifact" — a structured Markdown record showing each load-bearing concept's name + alternatives considered + user-language match score. Currently no such artifact exists.

**Focused variation:** What's missing from the Sensemaking discipline is a **Phase 5 / Conceptual Stabilization output marker for vocabulary commitment** — currently SV6 commits vocabulary as a side effect of model commitment, without a dedicated step. The redesign-from-scratch question: if Sensemaking were designed today, vocabulary would be a first-class Phase 5 commitment, not an afterthought.

**Contrarian variation:** What's missing across the corpus is NOT vocabulary validation — it's the user's vocabulary corpus. The loop has access to user input but treats it as conversational, not as a vocabulary corpus to validate against. The missing artifact is a **user-vocabulary corpus extracted from user input across the inquiry** — a list of terms the user has actually used.

Surviving absence output: **Focused variation** (Phase 5 vocabulary commitment marker as redesign). Reinforces U1's Phase-5 placement and explains why the gap exists structurally.

### Mechanism 3: Domain Transfer (Generator)

**Generic variation:** Software API design uses "naming conventions" as a first-class artifact (e.g., "Designing Data-Intensive Applications" naming chapter). Transfer: Sensemaking should treat naming conventions as a first-class commitment, with documented rationale per chosen name.

**Focused variation:** Linguistics has the concept of "speaker-language alignment" — when communicating, you use the listener's vocabulary, not your own. Transfer: when Sensemaking commits a vocabulary, the load-bearing question is "does this match the user's speaker-language?" not "is this technically correct?"

**Contrarian variation:** Pharmaceutical naming requires regulatory approval (the name itself must pass FDA review separately from the drug's efficacy). Transfer: vocabulary commitment in Sensemaking should be its own approval gate — the name passes/fails independently of the model's correctness.

Surviving domain-transfer output: **Focused variation** (speaker-language alignment). Clean conceptual fit with U1's user-language validation predicate; provides an explanatory frame for why the rule matters.

### Mechanism 4: Extrapolation (Generator)

**Generic variation:** If the corpus continues to coin names autonomously without validation, the rename event count will continue to grow at ~1 per chain. After 10 chains, the corpus will have 10+ rename events with downstream cleanup cost.

**Focused variation:** If U1 is promoted at chain 7, the next 3 chains' Sensemaking outputs become the test bed. If U1 fires correctly at all 3 chains, the rename event count drops from ~1/chain to ~0/chain post-chain-7. If U1 misses at any of the 3, U1 needs refinement (similar to M6's refinement track).

**Contrarian variation:** If user-language validation becomes a formal step, the loop loses some of its capacity for novel concept introduction — coining `PastNavigationMemoryFile` was a step beyond user vocabulary that captured something the user didn't yet have a word for. Validation may suppress useful loop-coined neologisms.

Surviving extrapolation output: **Focused variation** (U1 chain-7+3 effectiveness window) provides the evaluation gate concrete shape; **Contrarian variation** (suppression risk) becomes a caveat for U1's recognition predicate (must allow loop-coined names with explicit justification, not blanket-prohibit).

### Mechanism 5: Lens Shifting (Framer)

**Generic variation:** Under current conditions (Sensemaking optimized for model stability), vocabulary validation feels like extra burden. Under conditions where the corpus has matured (chain 7+ rename events) the burden shifts: rename cleanup costs more than upfront validation. Lens shift: from "validation is expensive" to "rename cleanup is more expensive."

**Focused variation:** Under conditions of "the loop is already mature enough to coin names autonomously" (chain 3+ corpus), vocabulary validation makes sense. Earlier, when names were largely user-supplied, validation would have been redundant. Lens shift: vocabulary validation is a chain-3+ readiness commitment.

**Contrarian variation:** Under conditions where user feedback is the primary correction signal, vocabulary validation BY THE LOOP is unnecessary — the user IS the validation. The lens shift: instead of internalizing validation, formalize the user's role as the vocabulary critic. This makes validation a process artifact (e.g., a `vocab_check.md` review request).

Surviving lens-shifting output: **Focused variation** (corpus-maturity readiness frame). Reinforces U1's chain-7 promotion as appropriate timing.

### Mechanism 6: Constraint Manipulation (Framer)

**Generic variation:** Add the constraint "every load-bearing concept name must appear in user input." This kills loop-coined neologisms entirely. Too restrictive.

**Focused variation:** Add the constraint "every load-bearing concept name must EITHER appear in user input OR be explicitly justified in SV6 with named alternatives." This is U1's existing rule shape — the constraint enables vocabulary validation without prohibiting loop coining.

**Contrarian variation:** Remove the constraint "names must be stable across an inquiry." Allow names to drift within a single inquiry — Sensemaking commits a working name with the explicit acknowledgment that it may rename at Phase 5 / Conceptual Stabilization. This treats vocabulary as iteratively refinable, not committed at coining.

Surviving constraint output: **Focused variation** (the constraint that IS U1). Pre-existing in U1's rule shape; this mechanism confirms U1 is the right constraint.

### Mechanism 7: Inversion (Framer)

**Generic variation, Level 1 (component):** "Sensemaking commits vocabulary at Phase 5" → invert to "Sensemaking commits vocabulary at Phase 1 (Anchor Extraction)." Earlier commitment = earlier opportunity to reject. But this conflicts with anchor extraction's purpose (capturing initial concepts, not refining them).

**Focused variation, Level 2 (system):** "Sensemaking commits vocabulary at Phase 5" → invert to "Sensemaking commits vocabulary as the LAST step before Conceptual Stabilization, and the commitment requires user-language validation." This is U1's existing rule shape elevated to system-level: vocabulary is the system's interface to the user; validation is the system's commitment criterion.

**Contrarian variation, Level 3 (root-cause):** "Sensemaking is the right stage for vocabulary commitment" → invert to "Sensemaking is the WRONG stage; vocabulary should be committed by a separate Vocabulary discipline that runs between Sensemaking and Decomposition." This is structural over-reach for chain 7's evidence — adds a new SIC-shaped discipline.

Surviving inversion output: **Focused variation** (Level 2 system inversion). Re-confirms U1's rule shape from a different angle. Level 3 is killed by Step 5 + Step 6 guardrails (no protocol-level changes from chain 7 alone; new disciplines need 5-10 findings of evidence).
