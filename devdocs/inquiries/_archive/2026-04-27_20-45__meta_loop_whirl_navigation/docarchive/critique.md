## B. Fitness Landscape

### Viable Region

Phrases that describe the meta-loop as stateful, artifact-native, traversal-oriented, Navigation-assisted, selection-explicit, and bounded by meaningful traversal.

### Dead Region

Phrases that reduce it to repeated MVL+, make Navigation the whole agent, or claim exhaustive discovery of thinking space.

### Boundary Region

Phrases that are useful from one angle but insufficient alone: purely poetic metaphors, purely technical graph models, or v1-only implementation descriptions.

## C. Candidate Verdicts

### Candidate A: Minimal Engineering Phrase

> A meta-loop is a runner that repeatedly invokes MVL+ and Navigation until a stop condition is reached.

**Prosecution:** This misses the user's core intuition. It describes a sequential implementation, not the concept. It does not naturally express backward movement, branch merge, or artifact terrain.

**Defense:** It is clear and buildable. It is a good v1 implementation shape.

**Collision:** Useful for v1, weak as conceptual definition.

**Verdict: REFINE.**

Constructive output: keep this as implementation language only.

### Candidate B: Stateful Traversal Engine

> A meta-loop is a stateful traversal engine for thinking space...

**Prosecution:** It may sound abstract. "Traversal engine" is technical and needs examples to feel alive.

**Defense:** It is precise, bounded, and captures the actual architecture: seed, terrain, Navigation, selection, MVL+, artifacts, meta-state, and stopping.

**Collision:** The prosecution is stylistic, not structural. The phrase survives if paired with the whirl metaphor.

**Verdict: SURVIVE.**

Constructive output: use this as the primary operational definition.

### Candidate C: Controlled Whirl

> The meta-loop is a controlled whirl through the project's thinking space...

**Prosecution:** The metaphor can overstate agency and make the system sound more autonomous than it is.

**Defense:** It preserves the user's actual insight: movement is not linear; the loop consumes artifacts, changes the terrain, and can move in multiple directions.

**Collision:** The metaphor survives only if "controlled" is load-bearing and the operational roles are named immediately.

**Verdict: REFINE / SURVIVE AS METAPHOR.**

Constructive output: use as the conceptual image, not as the full definition.

### Candidate D: Artifact Graph Walker

> A meta-loop is a graph walker over inquiry artifacts...

**Prosecution:** Too dry for the user's goal. It loses the sense that the process changes the space as it moves.

**Defense:** Technically excellent. It makes backward edges, branch edges, merge edges, and node creation easy to model.

**Collision:** Strong for implementation, insufficient for explanation by itself.

**Verdict: REFINE.**

Constructive output: use as the technical model under the operational definition.

### Candidate E: Cognitive Runtime

> A meta-loop is the cognitive runtime above MVL+...

**Prosecution:** It risks sounding more mature than the system is. "Runtime" implies executable enforcement that does not exist yet.

**Defense:** It correctly locates the meta-loop above disciplines and above MVL+.

**Collision:** Good long-term architecture label, too strong for current state.

**Verdict: REFINE.**

Constructive output: call it a future cognitive runtime, not the current one.

### Candidate F: Exhaustive Thinking-Space Discoverer

> A meta-loop is a process that discovers the full thinking space around a problem.

**Prosecution:** This overclaims. Thinking space is not fixed; new artifacts create new regions. Full discovery has no stop condition.

**Defense:** It captures ambition and explains why the meta-loop matters.

**Collision:** Ambition does not rescue an impossible literal claim.

**Verdict: KILL.**

Constructive seed: replace "fully discovers" with "traverses a bounded region until traversal is meaningful enough."

### Candidate G: First Implementation Phrase

> Meta-loop v1 is a sequential human-selected runner...

**Prosecution:** Too narrow as the concept. It hides backward motion and graph traversal.

**Defense:** Excellent v1. It avoids overbuilding and creates evidence for selector and state design.

**Collision:** It survives as the implementation entry point, not as the general concept.

**Verdict: SURVIVE AS V1.**

Constructive output: keep this as the first buildable expression.

## D. Assembly Check

This assembly is stronger than any single phrase. It lets the concept stay fuzzy and alive while also making it precise enough to build.
