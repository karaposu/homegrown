### Mechanism 3: Domain Transfer (Generator)

### Mechanism 4: Extrapolation (Generator)

**Generic:** As the corpus accumulates, Rule A and Rule B will catch dimension blindness and prosecution-depth shortcomings before they become recurrent failure patterns requiring corpus-level promotion.

**Focused:** If Rule A is encoded, future Critique runs evaluating candidates with project artifacts/operations/state will systematically check for project-specific risk axes. If Rule B is encoded, future Critique runs will systematically construct prosecution at relevant depth-axes. **Contrarian:** Encoding may codify too narrowly — the corpus's 4 specific dimensions and 3 specific depth-axes might not generalize to all future Critique contexts.

**Surviving:** focused extrapolation provides concrete evaluation gates. Contrarian "codify too narrowly" is mitigated by the rules' meta-pattern wording — they require A risk axis / depth-axis, not THE specific ones the corpus surfaced.

### Mechanism 5: Lens Shifting (Framer)

**Generic:** Under the current frame (Phase 0 says "extract dimensions"; Phase 2 says "construct prosecution"), the rules look like additions. Under a different frame (Phase 0 says "extract dimensions WITH project-specific risk axis coverage check"; Phase 2 says "construct prosecution WITH depth-axis enumeration"), the rules are clarifications of always-implicit checks.

**Focused:** Under "discipline-spec purity" lens, both rules are minimum viable additions — no embedded caveats, no protocol-specific concerns, no new failure modes. Aggregate ~10–15 lines is the smallest cost among sister analyses.

**Contrarian:** Under a "Critique should be intuitive, not overly structured" lens, depth-axis enumeration formalizes what should be runner judgment.

**Surviving:** focused lens shift confirms the rules' minimum-viable shape. Contrarian "intuition" objection is mitigated by the rules giving INTUITIVE judgments structured vocabulary; runners can still apply judgment within the depth-axes.

### Mechanism 6: Constraint Manipulation (Framer)

**Generic:** Add the constraint that both rules together must add ≤ 15 lines.

**Focused:** Add the constraint that each rule's wording must use only existing td-critique.md vocabulary (dimension list, validate dimensions, prosecution, depth, candidate, Phase 0, Phase 2, failure mode). No new abstractions.

**Contrarian:** Remove the constraint that rules must be at td-critique.md surfaces. Allow rules at LOOP_DIAGNOSE protocol level instead.

**Surviving:** focused constraint set shapes the wording. Contrarian relaxation rejected per Sensemaking — td-critique.md is generic; LOOP_DIAGNOSE-specific rules belong elsewhere.

### Mechanism 7: Inversion (Framer)

**Generic level 1 (component):** "Phase 0 extracts dimensions" → invert to "Phase 0 extracts dimensions AND verifies coverage of risk-axis categories." Rule A as inversion-derived.

**Focused level 2 (system):** "Prosecution constructs the strongest objection" → invert to "Prosecution constructs strong objections at multiple depth-axes." Rule B as inversion-derived. Re-frames prosecution as multi-axis rather than single-best-objection.

**Contrarian level 3 (root):** "Critique evaluates candidates against dimensions" → invert to "Critique evaluates candidates against dimensions AND constructs prosecution that the dimensions don't directly cover (depth-axis prosecution)." This is the key insight: dimension-level evaluation is necessary but insufficient; depth-axis prosecution covers what dimensions don't.

**Surviving:** focused inversion (level 2) confirms Rule B's multi-axis shape; level 3 reinforces the dimensions-vs-depth-axes distinction.

## Phase 3 — Test (5 criteria per surviving candidate)

### Candidate A-1 — Rule A Wording + Cross-Reference

**Concrete wording for Rule A** (canonical home: `homegrown/td-critique/references/td-critique.md`, Phase 0 / Dimension Construction, integrated into the validate-dimensions sub-step):

> **Project-specific risk dimension check.** When the candidate set being evaluated involves project artifacts, operations, or state, the dimension list must include at least one project-specific risk dimension that captures the project's documented risk axes. The default dimensions (Correctness, Coherence, Feasibility, Completeness, Robustness, Elegance) are content-oriented; project-specific risk dimensions are mechanism-oriented (e.g., across recent inquiries: duplicate-derivable-state, explicit-culture-fit, operation-parsimony, phase-fit have each been the load-bearing axis for specific candidate types). A dimension list that omits project-specific risk axes when the candidate set involves project artifacts/operations/state is incomplete; the validate-dimensions sub-step must explicitly check this and flag any missing axes for inclusion.

**Concrete cross-reference text** (at Dimension Blindness failure mode #4, appended to the existing "How to prevent" line):

> For project-specific risk axis coverage when the candidate set involves project artifacts/operations/state, see Phase 0 / Dimension Construction → Project-specific risk dimension check.

**Tests:**

- **Novelty:** new check; no equivalent in current spec. NOVEL.
- **Scrutiny survival:** strongest objection — *"identifying 'project artifacts/operations/state' is judgment-dependent."* Defense — the trigger is observable from candidate descriptions (do they propose creating/modifying artifacts? do they propose operations? do they touch durable state?); examples in the wording (duplicate-derivable-state, etc.) ground the abstraction without locking it. SURVIVES.
- **Fertility:** opens validate-dimensions sub-step as a structured surface for project-specific axis coverage; future axis types accumulate as evidence. FERTILE.
- **Actionability:** runner reads the rule → checks if candidate set involves artifacts/operations/state → if yes, verifies dimension list has ≥1 project-specific risk dimension → flags missing. Deterministic. ACTIONABLE.
- **Mechanism independence:** 5 mechanisms converge (Combination focused, Absence Recognition focused, Domain Transfer focused, Extrapolation focused, Constraint Manipulation focused). HIGH.

**VERDICT: SURVIVES.**

### Candidate B-1 — Rule B Wording + Cross-Reference

**Concrete wording for Rule B** (canonical home: `homegrown/td-critique/references/td-critique.md`, Phase 2 / Adversarial Evaluation, Prosecution sub-section, as a follow-on to the existing 4 prosecution prompts):

> **Multi-axis prosecution depth check.** In addition to dimension-level objections (what dimensions does this candidate fail?), construct prosecution at the appropriate depth-axes when applicable:
>
> - **User-perspective objection** — when the inquiry's source input includes user-stated concerns (e.g., a `_branch.md` Source Input section), construct at least one objection that addresses one of those concerns at the level the user expressed. Dimension-level objections without a corresponding user-perspective objection are shallow when user concerns are observable.
> - **Specific failure-case scenario** — for candidates with conditional or trigger-based behavior, construct at least one concrete edge case where the candidate's logic might fail to prevent the issue it's meant to prevent. Abstract "could fail" prosecution is weaker than concrete failure-case construction.
> - **Specification-gap probe** — for candidates whose runtime behavior depends on load-bearing concepts (e.g., "skip if X exists"), probe whether the candidate specifies HOW the load-bearing concept's runtime state is determined. Candidates that presuppose the determination without specifying the mechanism have an operational gap.
>
> The runner picks the depth-axes most relevant to the candidate's risk surface; not every axis applies to every candidate. Prosecution that constructs only dimension-level objections without considering relevant depth-axes is shallow.

**Concrete cross-reference text** (at Rubber-Stamping failure mode #2, appended to the existing "How to prevent" line):

> For multi-axis prosecution depth (user-perspective, failure-case scenario, specification-gap probe), see Phase 2 / Adversarial Evaluation → Prosecution → Multi-axis prosecution depth check.

**Tests:**

- **Novelty:** new structured prosecution check; the current spec gives 4 generic prompts but no depth-axis enumeration. NOVEL.
- **Scrutiny survival:** strongest objection — *"depth-axis enumeration may over-formalize what should be runner judgment."* Defense — the rule says "the runner picks the depth-axes most relevant"; runner judgment is preserved at the axis-selection layer. The depth-axes are observable (source input availability for user-perspective; conditional behavior for failure-case; runtime-determined behavior for specification-gap). SURVIVES.
- **Fertility:** opens Prosecution as a structured surface for depth-axis enumeration; future depth-axes could be added if corpus accumulates. FERTILE.
- **Actionability:** runner constructs prosecution → checks if user concerns are observable → if yes, constructs user-perspective objection → checks if conditional behavior → if yes, constructs failure-case scenario → checks if runtime-determined behavior → if yes, constructs specification-gap probe. Deterministic. ACTIONABLE.
- **Mechanism independence:** 6 mechanisms converge (Combination focused, Absence Recognition focused, Domain Transfer focused, Extrapolation focused, Lens Shifting focused, Inversion focused level 2 + level 3). HIGH.

**VERDICT: SURVIVES.**
