## Cycle 2 — Signal detection

### Signal S1 — The 13-30 Sensemaking perspective set is missing a Cross-Candidate Unity check

The 8 perspectives applied in 13-30 Sensemaking are:
- Technical/Logical: tests internal logic of THE DIAGNOSIS
- Human/User: tests user-perspective on THE DIAGNOSIS
- Strategic/Long-term: tests long-term value of THE DIAGNOSIS
- Risk/Failure: tests risks of THE DIAGNOSIS
- Resource/Feasibility: tests cost of THE DIAGNOSIS
- Internal Consistency: tests internal consistency of THE DIAGNOSIS
- FEC meta-application: tests whether the inquiry's FRAME excludes project-wide referents
- Phase/Calibration-State: tests phase-dependence

**Observation:** all 8 perspectives test PROPERTIES of the diagnosis. NONE test whether the CANDIDATES being diagnosed are at the right abstraction level — i.e., whether two "separate" candidates might be one underlying thing. The Cross-Candidate Unity check would be: "Take any two candidates in the candidate set. Is there a level of abstraction at which they are the same? If yes, the diagnosis treating them as separate misses the unity."

**Confidence:** Confirmed (artifact-direct; the 13-30 sensemaking.md does not contain this question).

### Signal S2 — The 13-30 Innovation Domain Transfer Generator was applied with a distinction-finding bias

Per 13-30 innovation.md line 38-39:
- "Biology / taxonomy: species **splitting** (when one species is recognized as two distinct populations)."
- "Philosophy / ontology: concept **distinction** (when one term covers two referents, name them separately)."

Both examples are about SEPARATING what looks unified. The opposite-direction examples (biology: convergent evolution producing similar species; philosophy: showing two terms refer to the same concept) were not surfaced.

**Observation:** Domain Transfer in 13-30 was applied to support the dual-diagnosis verdict that was already converging. It didn't test the OPPOSITE direction (find domains where surface-different things turn out to be one thing).

**Confidence:** Confirmed (artifact-direct).

### Signal S3 — The 13-30 Innovation Combination Generator was applied at the wrong level

The 13-30 Combination Generator combined "R2/R3 distinction" with "project conventions" to derive operational paths. **It did NOT combine R2 + R3 + Explore as candidates of a single operation.**

The Combination Generator's documented power per innovate.md: "Connects previously unrelated concepts to produce something neither contained alone." Applied at the candidate-set level, it would have asked: "What if R2, R3, and Explore are all combined into one operation? What does that operation look like?"

**Confidence:** Confirmed.

### Signal S4 — 13-30 Decomposition's piece-separation assumption

The 13-30 Decomposition had a piece P3 called "R2/R3 characterization" that lists R2 and R3 as separate sub-entries. The DECOMPOSITION rule is: partition into pieces with low cross-coupling. **There's no rule for: "before partitioning, check if any candidates are actually instances of the same piece at a deeper abstraction level."**

The Decomposition spec's "Determination-mechanism piece check" (recently added) is about runtime determination — different from abstraction-level deduplication.

**Confidence:** Confirmed.

### Signal S5 — 13-30 Critique's dimensions tested distinctness, not unity

The 13-30 Critique had 12 dimensions. None of them asked: "Could these candidates be instances of the same underlying operation at a different abstraction level?" Critique tested:
- D1 Diagnostic accuracy: assumed the dual diagnosis as the candidate to test
- D2 R2/R3 distinction clarity: presupposed the distinction was real
- D3 Path coverage; D4 User-vision alignment; D5 Recommendation soundness; etc.

**Observation:** Critique's dimensions presupposed the diagnostic frame (dual diagnosis). No dimension tested whether the frame itself was the right abstraction level.

**Confidence:** Confirmed.

### Signal S6 — Even 13-45 (the corrected loop) didn't autonomously find the convergence

The 13-45 Sensemaking's K1 says: "the unification holds at the OPERATION LEVEL but the IMPLEMENTATIONS differ." But this insight came AFTER the user's pushback (correction #3). The Sensemaking phase TESTED the user-supplied claim; it didn't INDEPENDENTLY derive it.

**Implication:** the missing capability isn't just "in 13-30"; it's a project-wide capability gap. Even with the just-adopted Comprehending+Stabilizing operations and FEC perspective, the corrected loop relied on user-supplied seed for the unification claim.

**Confidence:** Confirmed (artifact-direct review of 13-45 sensemaking.md).

### Signal S7 — The failure mode is structurally distinct from existing project failure modes

Existing project failure modes (across the discipline specs):
- Frame-scope capture (Memory; warming): frame EXCLUDES something that's in the project
- Clean Resolution Trap: counter-argument not tested
- Perspective Blindness: didn't check uncomfortable perspectives
- Premature Evaluation in Possibility Mode (just added to explore.md): Exploration applies dimensional evaluation
- Status Quo Bias; Anchor Dominance; Premature Stabilization; Self-Reference Blindness; etc.

**None** of these is "treating surface-different candidates as separate when they're instances of one underlying operation." Convergence-recognition failure is a new family.

**Confidence:** Confirmed.

### Signal S8 — The failure happens at the CANDIDATE-SET level, not within any single candidate's analysis

This is a key structural observation. The 13-30 loop analyzed each candidate (R2, R3, Explore) thoroughly. The miss was not within any single analysis. The miss was at the CANDIDATE-SET LEVEL: "given this set of candidates, is the set itself at the right abstraction level, or are some candidates instances of the same underlying thing?"

This points to where the fix should live: at a meta-level above individual candidate analysis. Either:
- A new perspective in Sensemaking that operates on the candidate set
- A new Generator/Framer application in Innovation that combines candidates
- A new check in Decomposition that asks "could these pieces be one piece at a higher abstraction?"
- A new dimension in Critique that tests the candidate-set's abstraction level
- A cross-discipline protocol element

**Confidence:** Confirmed (structural observation).

### Signal S9 — User has explicitly hypothesized the discipline locations

User said: "imo at least our innovation or sensemaking should have recognise that both are converging." The user's hypothesis: Innovation OR Sensemaking. They also mentioned Decomposition as a possibility. The user is asking ACROSS-DISCIPLINE; not pinpointing one.

The diagnosis must handle the cross-discipline framing — the fix might live in multiple disciplines, or in a cross-discipline protocol element.

**Confidence:** Scanned (user signal).

### Signal S10 — Jump scan: is there a different failure-mode framing the user missed?

**Confidence:** Scanned (jump scan).

Add a 9th perspective to Sensemaking Phase 2: when the analysis has multiple candidates, test whether any subset of candidates is instances of the same underlying operation. Conditional gating: fires when ≥2 candidates are being analyzed for their relationships.

**Evidence supporting:** Sensemaking's perspectives already test multiple angles; adding a candidate-set-level perspective is a natural extension. The user named Sensemaking as a likely location.

**Evidence neutral:** the perspective would be cross-cutting (operates on candidates, not on the analysis itself). May not fit the existing perspective shape (each existing perspective is content-axis, e.g., Technical, Risk).

**Confidence:** Scanned with positive support.

### Candidate Fix B — Innovation adds Combination-at-candidate-set-level mode

Innovation's Combination Generator already says: "Connects previously unrelated concepts to produce something neither contained alone." Extend to apply at the candidate-set level: "When the analysis has multiple candidates, try combining them — does combining them yield an underlying operation neither describes alone?"

**Evidence supporting:** Combination Generator's documented power is exactly this kind of synthesis. Applying it at the candidate-set level is a refinement, not a new mechanism.

**Evidence neutral:** the user named Innovation as a likely location.

**Confidence:** Scanned with positive support.

### Candidate Fix C — Decomposition adds piece-deduplication-at-abstraction-level check

Add to Decomposition's Step 3 (Validate Boundaries) or Step 7 (Self-Evaluate): "Check whether any pieces are instances of the same piece at a higher abstraction level. If yes, the pieces should be merged or the abstraction level reconsidered."

**Evidence supporting:** Decomposition's coupling-perception step already looks at relationships; adding an abstraction-level check is adjacent. The user mentioned Decomposition.

**Evidence neutral:** Decomposition operates on PIECES, not on CANDIDATES per se. The candidate-set framing may not map cleanly to Decomposition's piece framing.

**Confidence:** Scanned.

### Candidate Fix D — Cross-discipline protocol element

A new check that runs across multiple disciplines: "Cross-candidate operation-level unity check" applied between Sensemaking and Decomposition (or as a CONCLUDE-level check before finding compilation).

**Evidence supporting:** the failure spans multiple disciplines; a cross-discipline check might be more efficient than adding the same logic to each discipline.

**Evidence neutral:** new protocol elements are heavier than discipline-spec edits; would set a new precedent.

**Confidence:** Scanned.

### Candidate Fix E — Promote convergence-recognition to a named failure mode (without specifying which discipline owns it yet)

## Cycle 5 — Jump scan

**Jump-scan question:** is the failure really about "convergence-recognition," or is it about something subtler — e.g., the inherited frame from the user's question?

**Probe:** the user's 13-30 `_branch.md` framed the question as "is navigation = explore + assess, OR specialization, OR schema-overlay?" — these candidates assume SEPARATENESS. The user's framing PRE-BIASED toward dual-diagnosis. The disciplines worked within the inherited frame.

**Observation:** the failure might be partly attributable to FRAME INHERITANCE — the disciplines accepted the user's frame ("they're different things; how are they different?") without questioning whether the frame itself was right.

This is related to but distinct from frame-scope capture (which is about exclusions). It's more like FRAME-PREMISE CAPTURE: the disciplines didn't test whether the question's premise was right.

**Candidate Fix F (NEW from jump scan):** add a "Question Premise Check" at the start of Sensemaking — test whether the question's framing pre-biases the candidate set. E.g., a question like "how are X and Y different?" pre-biases toward separateness; a question like "are X and Y the same?" pre-biases toward unity. The check would flag: "the question is framed in a way that biases the candidate set; consider testing the inverse framing."

**Confidence:** Scanned (jump scan; new candidate).

**Implication:** the diagnosis might point to MULTIPLE fixes — Candidate A (Sensemaking unity perspective) + Candidate F (question-premise check). Or one comprehensive fix that includes both.
