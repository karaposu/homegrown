## Phase 2 - Candidate Evaluation

### Candidate P1 - Operation Output Contract Gate

Rule: before a loop treats an artifact as inline-only, print-only, optional, adaptive, or no-file, ask whether the artifact is canonical output of a meaningful future-affecting operation.

**Prosecution:**
This could become a generic checkbox that every loop mechanically answers "yes" to, causing artifact growth. It also may be too abstract unless connected to concrete examples like Route Memory Review.

**Defense:**
It targets the exact prior error. The prior loop did not need more enthusiasm for files; it needed the missing distinction between canonical output and storage convenience. The gate also preserves the possibility that some outputs are truly optional storage.

**Collision:**
The prosecution warns about vague enforcement, but the defense wins if the rule includes a pass condition: name the operation, explain future effect, and identify canonical output. That makes the gate discriminating instead of ceremonial.

**Position:** Viable.

**Verdict:** SURVIVE.

Constructive output:

- Keep the gate.
- Require concrete answers: operation name, future effect, canonical output, and why no-file is valid if chosen.

### Candidate P2 - Bloat Control Layer Gate

Rule: when artifact bloat is cited, test operation-trigger control before making canonical output optional.

**Prosecution:**
This could be overused to justify many mandatory outputs as long as a trigger exists on paper. Weak triggers can still create bloat.

**Defense:**
It fixes the precise level error in the prior inquiry. The corrected model does not deny bloat; it moves bloat control to "should Route Memory Review run?" The candidate forces that distinction.

**Collision:**
The defense survives if the gate requires the trigger to be explicit and falsifiable. A trigger like "when useful" is too vague; a trigger like "when old Navigation memory may materially affect the new map" is viable.

**Position:** Viable with wording constraint.

**Verdict:** SURVIVE.

Constructive output:

- Keep the gate.
- Add requirement: trigger must state skip conditions, not only run conditions.

### Candidate P3 - Operational Memory Critique Guard

**Prosecution:**

**Defense:**

The candidate is scoped to operational memory: old route maps, prior findings, protocol state, outcome records, and similar load-bearing memory. In that scope, the dimensions are not optional detail; they are the risk model.

**Position:** Viable with scope constraint.

**Verdict:** SURVIVE.

Constructive output:

- Scope it to operational-memory and future-interpretation artifacts.
- Pair it with P2 so explicitness never suppresses anti-bloat trigger discipline.

### Candidate P4 - Artifact-Policy Context Pairing

**Prosecution:**
More context intake alone would not have fixed this. The active local Navigation policy itself matched the weak prior answer. This candidate risks diagnosing the failure as "not enough reading," which the evidence does not support.

**Defense:**
It still addresses a real contributor. The prior loop appears to have over-weighted local anti-bloat context and under-weighted broader explicit-artifact patterns. Context pairing makes that imbalance less likely.

**Collision:**
The candidate survives only as a secondary support rule. It is not the main prevention layer and should not be represented as the fix.

**Position:** Boundary.

**Verdict:** REFINE.

Constructive output:

- Word it as: "When the local protocol is being evaluated, treat it as evidence and as the suspect policy; compare it with broader artifact culture."
- Do not make it the primary diagnosis or primary fix.
