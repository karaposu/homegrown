# Decomposition: Navigation Naming Boundary Drift Loop Diagnose

## Input

`_branch.md`, `exploration.md`, and `sensemaking.md` from this inquiry.

## Step 1 - Perceive Coupling Topology

The whole problem is:

```text
How should we diagnose and prevent the repeated Navigation route-memory pattern where unclear durable names made mechanisms feel unnatural or wrong?
```

### Elements

- Primary prior inquiry evidence.
- Corrected inquiry evidence.
- Supporting vocabulary evidence.
- Name type: status, operation, artifact, discovery, authority.
- User-language fit.
- Homegrown explicitness.
- Discipline failure location.
- Maintenance candidate.
- Confidence level.

### Coupling Gradients

Strong coupling:

- Name type and operation boundary. If the type is wrong, the operation implied by the name is wrong.
- Primary evidence and confidence. Confidence must follow evidence strength.
- User-language fit and mechanism confusion. The user's discomfort is the signal that the name implied the wrong mechanism.
- Explicitness form and artifact/index choices. Homegrown explicitness can be satisfied by files, telemetry, or repeatable procedures depending on operation type.

Moderate coupling:

- Discipline failure location and maintenance candidate. Where the failure appears affects where the guard should be added, but the guard can be cross-loop.
- Supporting vocabulary evidence and aggregate hypotheses. Supporting evidence can shape but not prove the primary diagnosis.
- Legacy/provisional naming and conclusion wording. Conclusion can harden weak terms if it does not mark them as legacy.

Weak coupling:

- Exact final replacement name and the diagnostic structure. The diagnosis can proceed without choosing a new name for `PastNavigationMemoryFile`.
- Index/search policy details and the primary `prior_map_overlay` / `Route-Memory Preflight` diagnosis.
- Formatting of future artifacts and the core failure model.

## Step 2 - Detect Boundaries Top-Down

Natural boundaries:

1. **Evidence Boundary**
   - What the artifacts prove, what they only suggest, and what must stay low confidence.

2. **Name-Type Boundary**
   - Whether a term names a status, operation, artifact, discovery procedure, or authority.

3. **Failure-Location Boundary**
   - Where the loop likely let the bad boundary through: framing, sensemaking, innovation, critique, or conclude.

4. **Explicitness-Form Boundary**
   - Whether explicitness should produce a file, telemetry/status line, procedure, index, or no new artifact.

5. **Maintenance Boundary**
   - What prevention mechanism should be added without changing core MVL/MVL+ machinery too broadly.

These boundaries cut at low coupling. Each can be answered independently and then reassembled into a diagnosis and prevention recommendation.

## Step 3 - Validate Boundaries Bottom-Up

### Atoms

- `prior_map_overlay` remained in reasoning after being identified as weak.
- `Route-Memory Preflight` sounded like a routine but should be status classification.
- `route_memory_review.md` is mandatory when full review runs.
- "source" is too generic for user-facing explanation.
- `PastNavigationMemoryFile` is dense and needs a gloss, but not proven wrong.
- `PastNavigationMemoryFile Index` turned discovery into a maintained object.
- `PastNavigationMemoryFile Discovery Search` made discovery procedural and scoped.
- User corrections repeatedly targeted what the mechanism sounded like.
- Primary evidence is stronger than supporting evidence.

### Validation

The atoms group naturally into the top-down boundaries:

- evidence atoms -> Evidence Boundary;
- status/operation/artifact/search/index atoms -> Name-Type Boundary;
- prior loop trace across disciplines -> Failure-Location Boundary;
- file/status/procedure/index atoms -> Explicitness-Form Boundary;
- recurring prevention need -> Maintenance Boundary.

Confidence: high. No atom is forced into an artificial category. The main crossing interface is deliberate: Name-Type Boundary feeds Failure-Location and Maintenance.

## Step 4 - Question Tree

### Q1. What exactly did the primary evidence prove?

Verification criteria:

- [ ] Separates primary evidence from supporting vocabulary evidence.
- [ ] Identifies high-confidence misses for `prior_map_overlay` and `Route-Memory Preflight`.
- [ ] Avoids treating later terminology as ground truth.
- [ ] States lower confidence for `source`, `PastNavigationMemoryFile`, index, and search aggregate claims.

### Q2. What type confusion did each problematic name create?

Verification criteria:

- [ ] Classifies each durable term as status, operation, artifact, discovery procedure, standing object, or authority-adjacent.
- [ ] Identifies where the name implied the wrong type.
- [ ] Explains why the wrong type caused mechanism confusion.
- [ ] Distinguishes dense-but-possibly-valid names from actually boundary-breaking names.

### Q3. Where did the loop likely miss the issue?

Verification criteria:

- [ ] Checks framing/branch question effects.
- [ ] Checks sensemaking vocabulary and boundary construction.
- [ ] Checks innovation candidate generation and alternative testing.
- [ ] Checks critique weighting and missing dimensions.
- [ ] Checks conclude/title/next-action hardening.
- [ ] Avoids blaming only one discipline when evidence is cross-loop.

### Q4. How should Homegrown explicitness be interpreted here?

Verification criteria:

- [ ] Preserves the rule that meaningful operations should create explicit durable artifacts.
- [ ] Avoids equating explicitness with maintained Markdown object by default.
- [ ] Distinguishes explicit procedure, status telemetry, operation artifact, run-local report, and global index.
- [ ] Explains why `route_memory_review.md` and the index/search decision differ.

### Q5. What prevention mechanism follows?

Verification criteria:

- [ ] Produces a lightweight invariant, not a new heavy sub-loop.
- [ ] Applies only to durable protocol terms and artifacts.
- [ ] Includes user-language/plain-gloss validation.
- [ ] Includes name-type validation.
- [ ] Includes legacy/provisional/canonical marking.
- [ ] Includes explicitness-form testing.
- [ ] Fits without changing core MVL/MVL+ disciplines wholesale.

## Step 5 - Interface Map

| Source Piece | Target Piece | What Flows | Direction |
| --- | --- | --- | --- |
| Q1 Evidence Boundary | Q2 Name-Type Boundary | Confidence-weighted facts about each term | one-way |
| Q2 Name-Type Boundary | Q3 Failure Location | Which layer/type mismatch each discipline should have caught | one-way |
| Q1 Evidence Boundary | Q3 Failure Location | Proof limits for discipline attribution | one-way |
| Q2 Name-Type Boundary | Q4 Explicitness Form | Whether explicitness should be file, status, procedure, report, or index | one-way |
| Q4 Explicitness Form | Q5 Prevention | Explicitness-form test requirements | one-way |
| Q3 Failure Location | Q5 Prevention | Placement of guardrails in critique/conclude/sensemaking | one-way |
| Q1 Evidence Boundary | Q5 Prevention | Confidence labels and avoidance of overfitted fixes | one-way |

Hidden coupling to avoid:

- Letting the supporting index/search correction decide the whole diagnosis.
- Letting "Homegrown likes Markdown" collapse into "always create a file."
- Letting "bad name" hide the deeper operation-boundary issue.
- Letting the prevention mechanism become a full new routine for every small term.

## Step 6 - Dependency Order

1. Answer Q1 first: evidence and confidence.
2. Answer Q2 second: name-type confusion.
3. Answer Q4 in parallel with Q2 after Q1 is stable: explicitness forms.
4. Answer Q3 after Q2 and Q4: where the loop failed.
5. Answer Q5 last: prevention mechanism.

Parallel work possible:

- Q2 name-type analysis and Q4 explicitness-form analysis can proceed together once Q1 evidence limits are fixed.

Must wait:

- Do not propose a prevention mechanism before deciding whether the issue is naming-only, boundary-name, or explicitness-form selection.
- Do not attribute discipline failure before separating primary proof from supporting pattern evidence.
