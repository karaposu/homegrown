The initial interpretation is:

```text
The prior loops kept choosing names that were internally meaningful to the model but externally confusing to the user.
```

That is true but incomplete. The evidence does not point to vocabulary alone. It points to names being stabilized before the operation boundary was tested from the user's perspective.

```text

```

## Phase 1 - Cognitive Anchor Extraction

### Constraints

- Primary diagnosis must come from the prior and corrected folders, not from later wording treated as ground truth.
- Supporting context can only strengthen vocabulary-pattern hypotheses.
- Confidence must stay lower for aggregate naming claims.
- The diagnosis must identify likely loop misses, not merely restate the corrected finding.
- Homegrown values explicit Markdown artifacts for meaningful operations.
- Navigation should remain one workflow; old route memory handling should not create unnatural run types.
- Route Memory Review must not become Navigation.
- Past Navigation memory files are historical evidence, not current route truth.

### Key Insights

- The prior loop saw that `prior_map_overlay` was a weak name, but did not fully retire the overlay frame.
- The corrected loop improved when it separated a cheap status classification from full review, not when it invented a stronger label.
- "Route-Memory Preflight" failed because it named a field-like classification as if it were a routine.
- "Source" failed because it named an abstract input role instead of the concrete file object the user needed to reason about.
- `PastNavigationMemoryFile Index` failed later because it converted a discovery need into a maintained standing object before search had been validated.
- The repeated pattern is not "names should be prettier." It is "names must preserve the correct layer: status, operation, artifact, discovery, or authority."

### Structural Points

The chain contains five layers that kept getting blended:

1. Navigation map generation.
2. Context freshness and route-memory status classification.
3. Full Route Memory Review.
4. Review artifact generation.
5. Past-memory discovery.

Names became confusing when they crossed these layers:

- `prior_map_overlay` blended review operation and storage artifact.
- `Route-Memory Preflight` blended status classification and protocol routine.
- `source` blended concrete file, evidence role, and discovery candidate.
- `PastNavigationMemoryFile Index` blended discovery convenience and persistent registry.

### Foundational Principles

- A name is not neutral. It teaches the next runner what kind of thing exists.
- A protocol name should name the operation, not its storage mode.
- A status field should not sound like a separate operation.
- A discovery mechanism should not sound like current authority.
- Explicitness is not synonymous with "make a maintained Markdown object." It can also mean an explicit repeatable procedure.
- In loop diagnosis, user discomfort is evidence when it points to a mechanism mismatch, not just a naming preference.

### Meaning-Nodes

- Boundary-name fit.
- Provisional term hardening.
- User-language validation.
- Artifact explicitness.
- Operation proliferation.
- Discovery versus authority.
- Internal precision versus understandable policy.

## SV2 - Anchor-Informed Understanding

## Phase 2 - Perspective Checking

### Technical / Logical

From a technical perspective, each term should have one type:

- status field;
- operation;
- artifact;
- discovery procedure;
- historical evidence object.

The confusing terms had type leaks. `Route-Memory Preflight` sounded like an operation but meant a status check. `prior_map_overlay.md` was an artifact path but kept shaping the operation model. `PastNavigationMemoryFile Index` was intended as a discovery aid but risked becoming an authoritative absence signal.

New anchor: type discipline is the technical core of naming hygiene.

### Human / User

The user repeatedly objected when a term did not match how they expected the workflow to feel:

- Navigation should be Navigation, not split by "big" versus "bounded."
- "Source" was too vague.
- `prior_map_overlay` and "preflight" sounded like wrong or extra mechanisms.
- The index sounded easier only until search was considered.

New anchor: the user's discomfort was not cosmetic. It was a signal that the proposed name implied the wrong workflow.

### Strategic / Long-Term

If names encode the wrong layer, future protocol patches will accumulate standing objects and side routines:

- overlay files;
- preflight routines;
- maintained indexes;
- candidate reports;
- status fields.

Some are valid, but only if each exists for a distinct operation boundary. Otherwise Homegrown gets explicit-looking artifacts that increase route-memory confusion.

New anchor: explicitness needs operation discipline or it becomes artifact multiplication.

### Risk / Failure

Main failure risks:

- stale route resurrection because old files are treated as current truth;
- route amnesia because past memory discovery is skipped;
- operation proliferation because a field is named like a routine;
- duplicate mutable state because a discovery search is turned into a registry;
- false confidence because a name sounds official before its boundary is proven.

New anchor: unclear names create behavioral risk because future agents follow names as contracts.

### Resource / Feasibility

Full user-language validation for every internal term would be heavy. But the failure pattern occurs around durable protocol terms, artifact names, and operation names. A small validation gate can focus there.

New anchor: the fix should target names that create durable workflow contracts, not every local variable or temporary phrase.

### Definitional / Consistency

Navigation is current route enumeration. Route Memory Review is current interpretation of past route memory. Discovery Search finds candidate files. A review artifact records review decisions.

When a term crosses one of these definitions, it creates a consistency problem. The corrected loop's success came from restoring these definitions:

```text
classification is not review
review is not Navigation
review artifact is not optional storage
discovery is not interpretation
```

New anchor: established definitions are useful only when their names keep the layers separate.

### Self-Reference Check

This diagnosis uses MVL+ and sensemaking vocabulary to inspect MVL+ outputs. That creates self-reference risk: the loop could overdiagnose "sensemaking boundary failure" because the diagnostic tool itself thinks in boundaries.

External anchors reduce that risk:

- user corrections repeatedly objected to names and boundaries;
- corrected artifacts explicitly identify "Route-Memory Preflight" as a named-routine risk;
- prior artifacts explicitly identify `prior_map_overlay` as weaker while retaining overlay framing;
- index-versus-search correction explicitly rejects a maintained object in favor of a protocol search.

New anchor: the boundary-name diagnosis is not only framework self-confirmation; it is grounded in user corrections and artifact transitions.

## Phase 3 - Ambiguity Collapse

### Ambiguity: Is this mainly a naming failure?

**Strongest counter-interpretation:**
Yes. The user was objecting to awkward or unclear words. Better names would have avoided the confusion.

**Why the counter-interpretation fails (structural grounds):**
The corrected inquiry did not simply replace "Route-Memory Preflight" with a better phrase. It changed the mechanism boundary: the cheap part became `route_memory_status` inside existing intake, while full review remained the artifact-producing operation. The index-versus-search correction also changed the mechanism from maintained registry to discovery search. These are operation-structure changes, not just wording changes.

**Confidence:** HIGH

**Resolution:**
This is a boundary-name failure. Names were symptoms and carriers of the failure, but the underlying issue was unstable operation layering.

**What is now fixed?**
Diagnosis must evaluate names together with the layer they imply.

**What is no longer allowed?**
Treating the maintenance fix as "rename things better" without testing operation boundaries.

**What now depends on this choice?**
Decomposition must split vocabulary selection, operation typing, artifact policy, and user-language validation.

**What changed in the conceptual model?**
The core failure moves from word choice to workflow-contract formation.

### Ambiguity: Was the prior loop careless about naming?

**Strongest counter-interpretation:**
No. The prior loop explicitly said `prior_map_overlay` was less clear and proposed `Route Memory Review`. It was already doing naming critique.

**Why the counter-interpretation fails (structural grounds):**
The prior loop noticed the weak name but kept the overlay frame in discipline titles, assemblies, and artifact policy. It did not make "legacy/transitional name must not govern the operation" an irreversible constraint. The result still treated saved overlay output as an adaptive storage mode, which the later user correction rejected.

**Confidence:** HIGH

**Resolution:**
The prior loop was not careless; it performed incomplete name migration. It identified the weak name but did not quarantine it from subsequent reasoning.

**What is now fixed?**
The likely failure is not lack of any naming awareness. It is lack of a commit step that retires or downgrades misleading names.

**What is no longer allowed?**

**What now depends on this choice?**
Innovation should consider a "provisional/legacy name quarantine" or "name migration check" rather than a generic naming reminder.

**What changed in the conceptual model?**

## Phase 4 - Degrees-of-Freedom Reduction

### Fixed Variables

- Names that become protocol terms, artifact names, routine names, or indexes require boundary validation.
- The validation must ask what type of thing the name implies: status, operation, artifact, discovery procedure, or authority.
- Legacy/transitional names must be explicitly quarantined if they are known to be misleading.
- User-facing policy names need a plain-language gloss.
- Explicitness must be satisfied by the right form, not automatically by a maintained file.

### Eliminated Options

- Pure rename-only fix.
- Pure "write more docs" fix.
- Pure "always ask the user about names" fix for every term.
- Treating `PastNavigationMemoryFile` as proven wrong.
- Treating the index correction as proof that explicit artifacts should be avoided.
- Adding a heavy naming protocol before every small loop.

### Viable Paths

1. Add a durable-term validation gate before CONCLUDE.
2. Add a Critique dimension: name-implied behavior / operation-boundary fit.
3. Add a Sensemaking check for status-vs-operation-vs-artifact-vs-discovery naming.
4. Add Conclusion wording that marks legacy names as legacy and prevents them from governing future reasoning.
5. Require plain-language gloss for new durable protocol terms.
6. Require explicitness-form testing: procedure vs file vs index vs telemetry.

### Stable Diagnostic Model

```text
Framing chooses a mechanism-shaped name.
Sensemaking finds partial conceptual clarity.
Innovation turns the name into candidate machinery.
Critique tests authority/artifact risks but not enough user-language/name-implied behavior.
Conclude makes the name durable.
User detects that the named mechanism feels unnatural.
```

### Stable Maintenance Direction

The fix should target durable names at the point where they become contracts:

```text
Before CONCLUDE, validate each new durable term:
  What type of thing is it?
  What might the user infer it does?
  Does it name operation, artifact, status, discovery, or authority?
  Is there a plainer name or gloss?
  Is this term provisional, legacy, or canonical?
  Does explicitness require a file, a status line, a procedure, or an index?
```

```text

type, user inference, operation/artifact/status/discovery/authority separation,
plain-language gloss, legacy/provisional/canonical status, and explicitness form.
```
