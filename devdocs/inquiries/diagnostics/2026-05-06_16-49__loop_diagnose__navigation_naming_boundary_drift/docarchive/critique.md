## Phase 2 - Candidate Evaluation

### Candidate A - Boundary-Name Validation Gate

**Prosecution:**
This could become another checklist that slows MVL+ and encourages performative compliance. If applied to every phrase, it will create noise and make the loop more bureaucratic.

**Defense:**
The gate is explicitly scoped to durable protocol terms, artifact names, status fields, discovery mechanisms, route names, and indexes. Those are exactly the places where prior failures occurred. The check happens before CONCLUDE, where terms become durable, so it catches the failure at the right time.

**Collision:**
Defense survives if the scope is strict. The candidate fails if expanded to all wording.

**Verdict: SURVIVE with scope constraint**

Constructive output: use only when a term can affect future behavior or durable artifacts.

### Candidate B - Critique Dimension: Name-Implied Behavior

**Prosecution:**
Critique already has many dimensions. Adding another dimension may dilute focus, and a weak critic may rubber-stamp the name anyway.

**Defense:**
The missing dimension is visible in the chain. `Route-Memory Preflight` should have been prosecuted as operation proliferation. `PastNavigationMemoryFile Index` should have been prosecuted as duplicate state and standing-object bias earlier. A named dimension makes this risk visible.

**Collision:**
Defense survives, but the dimension is not sufficient alone. It must feed the Boundary-Name Validation Gate.

**Verdict: SURVIVE as component**

Constructive output: add as a conditional critique axis when durable terms are introduced or preserved.

### Candidate C - Durable Terms Block In Findings

**Prosecution:**
A table can become boilerplate. It may create the appearance of clarity without actually testing whether the name is good.

**Defense:**
The table makes the decision inspectable. It records type, plain meaning, exclusions, status, and explicitness form. Future readers can see whether a term is legacy, provisional, or canonical.

**Collision:**
Defense survives only if the table is an output of the validation, not a substitute for it.

**Verdict: SURVIVE as output shape**

Constructive output: require the block only when the finding creates, renames, retires, or materially relies on durable terms.

### Candidate D - Legacy / Provisional Name Quarantine

**Prosecution:**
This can be awkward because old names still appear in filenames and historical findings. Banning them too hard may make docs harder to search.

**Defense:**
The rule does not ban references. It says weak/transitional names may be cited as legacy aliases but must not govern current reasoning. That directly addresses `prior_map_overlay`, where the weak name was detected but stayed active.

**Collision:**
Defense wins. Searchability and historical reference are preserved, but authority is removed.

**Verdict: SURVIVE**

Constructive output: include "legacy alias" handling in durable-term validation.

### Candidate E - Explicitness Form Matrix

**Prosecution:**
This may be used to dodge the user's explicitness culture by choosing "procedure" or "status record" instead of creating files.

**Defense:**
The matrix does not weaken operation artifacts. It makes the distinction explicit:

- full Route Memory Review -> operation artifact;
- route-memory status classification -> status record;
- discovery by known patterns -> repeatable procedure;
- broad disputed search -> run-local report;
- cross-run browseability or search failure -> global index.

This is exactly the distinction the correction chain needed.

**Collision:**
Defense wins if the matrix includes "no full meaningful operation without artifact."

**Verdict: SURVIVE**

Constructive output: pair the matrix with a rule that meaningful review operations still write Markdown artifacts.

### Candidate F - Plain Gloss Requirement

**Prosecution:**
Plain glosses can be superficial. A term can have a nice explanation and still imply the wrong workflow.

**Defense:**
The gloss is not the whole solution. It catches the "source????" problem and forces formal terms like `PastNavigationMemoryFile` to be explained in concrete language.

**Collision:**
Defense survives as support, not as the core mechanism.

**Verdict: SURVIVE as support**

Constructive output: require "plain meaning" and "not this" fields in the durable term block.

### Candidate G - Full Naming Review Discipline

**Prosecution:**
This creates another named operation to prevent over-named operations. It is structurally suspicious in the exact way the user objected to "Route-Memory Preflight."

**Defense:**
If durable naming failures continue, a full review could provide stronger coverage.

**Collision:**
Prosecution wins for v1. The defense is a future escalation, not a current default.

**Verdict: KILL as v1; DEFER as escalation**

Seed from failure: if the lightweight gate fails repeatedly, revive a focused naming review only for high-impact protocol renames.

### Candidate H - Rename `PastNavigationMemoryFile`

**Prosecution:**
This overfits. The evidence does not prove the term itself is wrong. Renaming now may create churn and repeat the same failure without improving boundary validation.

**Defense:**
A clearer term could reduce user confusion.

**Collision:**
Prosecution wins as a prevention mechanism. The defense remains possible after a term-specific review.

**Verdict: KILL as current fix; DEFER term-specific rename**

Seed from failure: test a replacement name only after durable-term validation shows the current term's plain gloss is still insufficient.

### Candidate I - No Protocol Change

**Prosecution:**
The pattern already repeated across several inquiries. Relying on user correction wastes time and lets wrong terms harden into docs.

**Defense:**
No added process.

**Collision:**
Defense is too weak. Low process cost does not solve the observed failure.

**Verdict: KILL**

Seed from failure: keep the prevention lightweight, but do not leave it to chance.
