## Seed

The seed is a correction:

```text
The previous prioritization loop answered the Navigation-shaped evidence,
not the MVL self-maintenance question.
```

The innovation target is therefore not a Navigation patch. It is a set of MVL maintenance mechanisms that prevent similar loop failures.

## Mechanism 1 - Lens Shifting

### Generic

Evaluate fixes as **loop-quality controls**, not topic repairs.

Output: Navigation examples become tests for MVL gates.

### Focused

Evaluate each diagnostic as **stage evidence**.

Output: a fix must name whether it repairs framing, Sensemaking, Innovation, Critique, Decomposition, CONCLUDE, or diagnostic workflow.

### Contrarian

Evaluate the previous Navigation-heavy answer as useful, but in the wrong layer.

Output: keep some concepts as examples inside MVL checks, but do not rank them as primary self-maintenance fixes.

## Mechanism 2 - Combination

### Generic

Combine latest failure with repeated corpus failure.

Output:

```text
Target-Layer Alignment Gate
  + Scoped Critique Decisive-Dimension Pack
```

### Focused

Combine quick checks that prevent low-cost repeated errors.

Output:

```text
LOOP_DIAGNOSE archive-use
  + correction-chain framing
  + answer-target validation
```

### Contrarian

Combine every diagnostic candidate into one "MVL self-repair mode."

Output: too broad. It risks producing the same operation-proliferation failure the diagnostics warned about.

## Mechanism 3 - Inversion

### Generic

Instead of asking "what failed most often?", ask "what must not fail in the next self-maintenance loop?"

Output: target-layer alignment wins urgency because a wrong target invalidates the whole answer.

### Focused

Instead of assuming Critique is downstream, treat Critique as the system's broad reliability brake.

Output: Critique pack wins reliability and breadth.

### Contrarian

Instead of adding guards, remove MVL autonomy and ask the user before every priority interpretation.

Output: rejected. It would slow the loop and contradict the working style. Better: record assumptions and ask only when target layers cannot be inferred.

## Mechanism 4 - Constraint Manipulation

### Generic

Add constraint: no fix may target Navigation unless it directly changes MVL self-maintenance behavior.

Output: Navigation intake package is no longer a top winner.

### Focused

Add constraint: broad fixes must be problem-triggered, not always-on.

Output: Critique pack becomes a selectable dimension set, not a global checklist.

### Contrarian

Remove constraint: allow broad MVL rewrite now.

Output: rejected by source evidence. Many diagnostics explicitly defer broad core rewrites until repeated scoped fixes fail.

## Mechanism 6 - Domain Transfer

### Generic

Transfer from incident response: distinguish incident symptom, root cause, corrective action, and preventive action.

Output: evidence domain and maintenance target must not collapse.

### Focused

Transfer from software testing: a regression test must reproduce the bug, not test an adjacent feature.

Output: the latest regression test is "given Navigation-heavy diagnostics, does the loop still rank MVL maintenance fixes?"

### Contrarian

Transfer from compliance: add mandatory sign-offs for each discipline.

Output: rejected. The diagnostics favor scoped checks, not routine ceremony.

## Mechanism 7 - Extrapolation

### Generic

If MVL keeps doing self-maintenance, evidence topics will often be salient and tempting.

Output: target-layer alignment becomes increasingly important as diagnostic corpus grows.

### Focused

If Critique keeps missing decisive dimensions, future findings will continue to harden plausible wrong answers.

Output: Critique pack is the highest-leverage reliability investment.

### Contrarian

If every correction produces a new gate, MVL becomes overloaded.

Output: each proposed gate must have activation conditions and evaluation gates.

## Candidate Set

### Candidate A - Target-Layer Alignment Gate

For self-maintenance, diagnostic, and meta inquiries, record:

```yaml
evidence_domain:
diagnosis_target:
maintenance_target:
implementation_target:
out_of_scope_target:
```

Core rule:

```text
The final answer must repair the maintenance_target.
The evidence_domain may supply examples, but it is not the target unless explicitly named.
```

### Candidate B - Scoped Critique Decisive-Dimension Pack

Critique selects relevant dimensions by problem type:

- target fit and branch-goal coverage;
- real dependency versus observable proxy;
- premise/layer validity;
- canonical versus provenance layer;
- base-loop burden and artifact reuse;
- fallback as possible primary mechanism;
- operation boundary and operation proliferation;
- phase, calibration, and breakthrough preservation;
- name-implied behavior;
- load-bearing interface defined or deferred;
- strongest alternative visible before candidate death.

### Candidate C - `LOOP_DIAGNOSE` Archive-Use And Correction-Chain Framing Check

When diagnosing a correction chain:

- use `homegrown/protocols/loop_diagnose.md`;
- read `_branch.md`, `_state.md`, root outputs, and `docarchive/*` when the diagnostic asks for stage attribution;
- separate routing, diagnosis, evidence storage, and maintenance candidates;
- inspect existing artifacts before proposing new marks or registries.

### Candidate D - Answer-Target Finalization Check

Before Critique terminates or CONCLUDE writes a finding:

```text
Does the survivor answer the branch question and advance the branch goal?
Does it target the requested system, not merely the most salient evidence domain?
```

This is the lightweight final form of Candidate A.

### Candidate E - Sensemaking Boundary And Assumption Pack

Sensemaking must extract and test anchors for:

- canonical/provenance layer;
- explicitness form;
- operation output versus storage convenience;
- status classification versus review;
- phase and cost/risk priority;
- user-felt messiness;
- durable term plain meaning and "not this" boundary.

### Candidate F - Innovation Strongest Alternative Pack

Innovation must strengthen alternatives before killing them:

- search/no-index -> protocolized deterministic search plus optional report;
- embedded marks -> archive-derived or wrapper-protocol alternatives;
- robust expensive option -> temporary mode with exit telemetry;
- named routine -> status field or section inside existing operation;
- inherited hard constraint -> contrarian challenge when user correction points against it.

### Candidate G - Decomposition Load-Bearing Interface Check

Decomposition must split visible load-bearing dependencies:

- if a trigger depends on X existing, decompose discovery method, status, confidence, fallback, and ownership;
- separate classification, operation, artifact, handoff, and authority;
- make secondary interfaces first-class when a finding proposes operational patch guidance.

### Candidate H - CONCLUDE Handoff Integrity Backstops

CONCLUDE checks before hardening:

- secondary operational claim defined or deferred;
- load-bearing interface defined or deferred;
- examples labeled as examples when not formal triggers;
- overstatement check for caveated survivors;
- Durable Terms block when terms become future workflow contracts.

### Candidate I - Broad MVL Core Rewrite

Rewrite MVL/MVL+ and all discipline specs with all recurring gates.

### Candidate J - Navigation Implementation Patch As Top Fix

Patch Navigation route-memory intake, phase policy, and discovery as the main answer.
