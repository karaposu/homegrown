# Innovation: Loop Diagnose - Project-Level Proxy Route Memory

## User Input

`devdocs/inquiries/2026-05-06_14-48__loop_diagnose__project_level_proxy_route_memory/_branch.md`

## Seed

The prior loop got the file contract right but let scale-shaped examples become proxy trigger guidance. The innovation need is a prevention bundle that preserves the artifact rule while making route-memory dependency explicit across bounded and broad Navigation inputs.

## Mechanism 1 - Lens Shifting

### Generic

Lens: treat the problem as trigger taxonomy, not artifact policy.

Output: require every route-memory rule to name the source type and dependency condition separately from artifact output.

### Focused

Lens: evaluate bounded inputs by semantic role, not path size.

Output: add a bounded route-memory artifact test: a single file or folder can still be `PastNavigationMemoryFile`.

### Contrarian

Lens: treat project-level as a useful probability hint.

Output: allow scale examples only if labeled as likelihood hints and followed by counterexamples.

## Mechanism 2 - Combination

### Generic

Combine universal status with conditional artifact output.

Output:

```text
every Navigation run -> route_memory_status
review_needed -> route_memory_review.md
```

### Focused

Combine source taxonomy with skip reasons.

Output: `none_detected`, `not_relevant`, `review_consumed`, and `thin_skipped` must each include a reason that does not use "bounded" or "project-level" as a sufficient explanation.

### Contrarian

Combine all fix into CONCLUDE only.

Output: final findings must label examples as non-rules. This helps but is downstream and cannot replace upstream source taxonomy.

## Mechanism 3 - Inversion

### Generic

Invert "bounded local is usually safe."

Output: any bounded input must first be asked: "is this file or folder itself route memory?"

### Focused

Invert "project-level likely needs review."

Output: any project-level input must still prove a route-memory object exists and is relevant.

### Contrarian

Invert "avoid full review every run."

Output: run full review every time to avoid missing bounded route memory. This fails because no-source runs have no review object; status recording is the better universal behavior.

## Mechanism 4 - Constraint Manipulation

### Generic

Add constraint: examples cannot carry trigger authority.

Output: if a finding uses examples like bounded/project-level, it must also state the structural trigger and at least one counterexample.

### Focused

Add constraint: every proposed Navigation routing patch must include a source-dependency test case matrix.

Output:

```text
bounded + route-memory source -> review status may be review_needed
bounded + no route-memory source -> no full review
project-level + route-memory source -> review status may be review_needed
project-level + no relevant source -> no full review
```

### Contrarian

Remove constraint: allow intuitive examples without formal trigger taxonomy.

Output: faster findings, but exactly the proxy leak can recur.

## Mechanism 5 - Absence Recognition

### Generic

Missing thing: a named "proxy trigger risk" check.

Output: policy findings should ask whether a convenient observable is standing in for the real dependency.

### Focused

Missing thing: `PastNavigationMemoryFile` detection as a source taxonomy.

Output: define examples of route-memory sources before deciding review trigger.

### Contrarian

Missing thing: no-op review artifacts for skipped cases.

Output: generate `route_memory_review.md` for `none_detected` and `not_relevant` to make every skip explicit. This overcorrects; status telemetry is enough unless audits prove otherwise.

## Mechanism 6 - Domain Transfer

### Generic

Transfer from type systems: do not infer behavior from file size or call site; inspect the type.

Output: classify input source type before choosing the route-memory operation.

### Focused

Transfer from routing tables: route by destination/dependency, not packet size.

Output: Navigation context intake routes by `route_memory_status`, not by bounded/project-level category.

### Contrarian

Transfer from triage: quick heuristics are useful first-pass filters.

Output: project-level/bounded can remain search hints, but not decision rules.

## Mechanism 7 - Extrapolation

### Generic

If scale proxies remain in docs, future Navigation may repeatedly miss local route memory stored in old inquiry folders.

Output: add bounded-source regression examples to route-memory policy.

### Focused

If every correction adds more fields, Navigation output may get noisy.

Output: keep universal route-memory status compact; reserve full review file for `review_needed`.

### Contrarian

If broad MVL+ rules are added for every proxy failure, the loop becomes heavy.

Output: patch Navigation route-memory docs first; defer global MVL+ proxy-trigger guard until repeated diagnoses show it across domains.

### Candidate B - Universal Route-Memory Status Record

### Candidate C - Bounded Route-Memory Regression Matrix

### Candidate D - Example Discipline Guard

### Candidate E - Proxy Trigger Critique Dimension

### Candidate F - Secondary Operational Claim Guard

### Candidate G - No-Op Review Artifact For Every Skip

Generate `route_memory_review.md` for `none_detected` and `not_relevant` cases so explicitness always creates a file.

### Candidate H - Broad MVL+ Proxy-Trigger Invariant

## Recommended Innovation

Adopt a scoped trigger-proxy prevention bundle:

```text
For Navigation route-memory routing:
  define trigger by PastNavigationMemoryFile source dependency;
  record route_memory_status for every Navigation run;
  run full Route Memory Review only for review_needed;
  keep route_memory_review.md mandatory when full review runs.

For future route-memory findings:
  test bounded route-memory and broad no-source cases;
  label bounded/project-level examples as likelihood hints, not rules;
  critique whether any convenient proxy is replacing the real dependency;
  if the finding proposes routing behavior, define the trigger or mark it deferred.
```
