# Decomposition: Loop Diagnose - Early Stage Route Memory Default

## Input

`_branch.md`, `exploration.md`, and `sensemaking.md` from this inquiry.

## Step 1 - Perceive Coupling Topology

The whole problem:

```text
Diagnose why the prior Route Memory Review trigger loop produced a mature material-effect policy instead of the later early-stage robust default, while avoiding overconfident root-cause claims.
```

### Elements

- Weak prior inquiry artifacts.
- Human correction signal.
- Corrected inquiry artifacts.
- Prior branch framing.
- Prior discipline behavior: Exploration, Sensemaking, Decomposition, Innovation, Critique, CONCLUDE.
- Corrected discipline behavior.
- Failure hypotheses.
- Confidence levels.
- Affected stages.
- Maintenance candidates.
- Evaluation gates.
- Guardrail against treating corrected inquiry as ground truth.

### Coupling Gradients

Strong coupling:

- Correction-chain delta and failure hypotheses: the hypotheses must explain the specific change from prior to corrected inquiry.
- Failure hypotheses and affected-stage attribution: attribution depends on what each hypothesis says was missed.
- Maintenance candidates and evidence strength: source edits require stronger evidence than monitoring candidates.
- Maintenance candidates and evaluation gates: every candidate needs a way to test whether it prevents recurrence.

Moderate coupling:

- Prior branch framing and discipline attribution: branch framing shapes all later outputs, but does not fully excuse discipline misses.
- Corrected inquiry and diagnostic verdict: corrected inquiry provides comparative evidence, but cannot be treated as final truth.
- Active-doc surface and maintenance scope: active docs lacking searched terms suggests likely patch areas, but does not prove exact source edits.

Weak coupling:

- File naming details and the core failure: the diagnosis does not depend on exact future route-memory filename choices.
- Structural checker availability and diagnosis quality: checker absence affects process verification, not the correction-chain substance.

## Step 2 - Detect Boundaries Top-Down

Natural pieces:

1. **Correction Chain Delta**
   - What changed from prior result to corrected result?

2. **Failure Hypothesis Set**
   - What likely did the prior loop miss?

3. **Stage Attribution**
   - Where in the loop or orchestration did the miss likely enter and propagate?

4. **Maintenance Candidate Design**
   - What should change to prevent recurrence?

5. **Evaluation And Risk Gates**
   - How should each candidate be tested and bounded?

6. **Diagnostic Verdict**
   - Is the diagnosis actionable, partial, or inconclusive?

These boundaries are natural because each has a distinct output contract. The delta is descriptive, hypotheses are explanatory, attribution is localization, maintenance is intervention, gates are validation, and verdict is decision.

## Step 4 - Question Tree

### Q1. What changed from prior inquiry to corrected inquiry?

Verification criteria:

- [ ] States prior path and corrected path.
- [ ] Quotes or faithfully summarizes the human correction.
- [ ] Distinguishes preserved rules from changed rules.
- [ ] Identifies the main policy shift: mature material-effect trigger -> early-stage source-gated robust default.
- [ ] Notes that corrected inquiry is comparative evidence, not ground truth.

### Q2. What failure hypotheses explain the change?

Verification criteria:

- [ ] Includes phase blindness / phase-insensitive optimization.
- [ ] Includes calibration underweighting.
- [ ] Includes breakthrough preservation underweighting.
- [ ] Includes user priority or context elicitation gap.
- [ ] Avoids unsupported claims that artifact explicitness was the main miss.
- [ ] Assigns confidence and "why not stronger" to each hypothesis.

### Q3. Which stage or orchestration layer likely failed?

Verification criteria:

- [ ] Allows mixed attribution.
- [ ] Separates branch framing/context elicitation from discipline outputs.
- [ ] Identifies Sensemaking's missing phase anchor.
- [ ] Identifies Innovation's underdeveloped temporary robust-mode candidate.
- [ ] Identifies Critique's missing dimensions.
- [ ] Does not claim exact single-stage root cause unless evidence isolates it.

### Q4. What maintenance candidates follow?

Verification criteria:

- [ ] Proposes a phase/calibration guardrail for Route Memory Review trigger selection.
- [ ] Proposes robustness/breakthrough/calibration dimensions for relevant critiques or policy inquiries.
- [ ] Proposes a Navigation policy patch only if framed as narrow and evidence-gated.
- [ ] Distinguishes source edits from monitoring candidates.
- [ ] Avoids broad MVL+ rewrite from one correction chain.

### Q5. What evaluation gates bound the maintenance candidates?

Verification criteria:

- [ ] Defines how to test phase guardrails.
- [ ] Defines telemetry needed to return from robust mode to mature trigger.
- [ ] Defines recurrence checks for future LOOP_DIAGNOSE findings.
- [ ] Includes over-review risks: old-map authority drift, cost burden, no-op review rate.
- [ ] Includes under-review risks: stale route resurrection, route amnesia, missed blockers.

### Q6. What final diagnostic verdict is justified?

Verification criteria:

- [ ] Chooses ACTIONABLE, PARTIAL, or INCONCLUSIVE.
- [ ] Names the best-supported diagnosis.
- [ ] Names the strongest maintenance candidate.
- [ ] Names the main uncertainty.
- [ ] Recommends a next step without overreach.

## Step 5 - Interface Map

| Source Piece | Target Piece | What Flows | Direction |
| --- | --- | --- | --- |
| Q1 Correction Chain Delta | Q2 Failure Hypothesis Set | Evidence of what changed and what stayed stable | one-way |
| Q1 Correction Chain Delta | Q3 Stage Attribution | Prior/corrected discipline contrasts | one-way |
| Q2 Failure Hypothesis Set | Q3 Stage Attribution | Candidate misses that need stage localization | one-way |
| Q2 Failure Hypothesis Set | Q4 Maintenance Candidate Design | What each candidate must prevent | one-way |
| Q3 Stage Attribution | Q4 Maintenance Candidate Design | Correct intervention layer | one-way |
| Q4 Maintenance Candidate Design | Q5 Evaluation And Risk Gates | Candidate claims that need tests | one-way |
| Q5 Evaluation And Risk Gates | Q4 Maintenance Candidate Design | Gate failure may narrow or defer a candidate | feedback |
| Q1-Q5 | Q6 Diagnostic Verdict | Evidence strength, uncertainty, and actionability | many-to-one |

Hidden coupling to avoid:

- Letting the corrected inquiry decide the verdict without independent confidence assessment.
- Letting maintenance design skip evaluation gates.
- Treating active-doc term absence as proof of exact patch text.
- Collapsing branch framing and discipline failure into a single blame point.

## Step 6 - Dependency Order

1. Establish correction-chain delta.
2. Generate failure hypotheses from the delta.
3. Attribute hypotheses to stages or orchestration layers.
4. Design maintenance candidates from the strongest attributions.
5. Attach evaluation and risk gates.
6. Issue diagnostic verdict.

Parallel work possible after step 2:

- stage attribution and maintenance brainstorming can proceed in parallel if they share the same hypothesis list;
- Navigation-specific candidates and MVL+/diagnostic candidates can be considered separately.

Must wait:

- Do not propose source edits before evidence strength and stage attribution are clear.
- Do not issue an ACTIONABLE verdict before evaluation gates exist.
