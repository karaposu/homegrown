# Innovation: Approximating Regression Measurement

## Seed

Numeric telemetry doesn't capture meaning-quality. But regression must be detectable. The human is part of the system. The measurement co-evolves with what it measures.

---

## Surviving Innovations

### 1. Regression Symptom Checklist

Not a measurement — a diagnostic pattern. Check after each run:

| Symptom | Signal |
|---|---|
| **Thin frontier** | Few or no frontier questions produced |
| **No surprise** | Human wasn't surprised or taught something new |
| **Downstream rejection** | Next discipline in pipeline struggles with the input |
| **Rapid completion** | Unusually fast, no adversarial challenge |
| **Prior-confirmation** | Output perfectly matches prior expectations (potential assimilation error) |

0-1 symptoms: probably fine.
2 symptoms: check more carefully.
3+ symptoms: likely regression.

### 2. Downstream Usability Test

The SIC loop IS the regression test. If innovation struggles with sensemaking output → sensemaking may have regressed. The consuming discipline's experience with the input is the quality signal. Happens naturally in the loop.

### 3. Canary Problem

Save one known-good run per discipline as a reference. Periodically re-run the same problem. Human compares qualitatively: "is this as rich as the reference?" Catches slow drift.

---

## Key Insight

Regression is detectable through SYMPTOMS, not through measurement. No single symptom is conclusive. The pattern of symptoms is diagnostic. This uses existing outputs (frontier questions) + one human signal (surprise y/n) + the pipeline itself (downstream usability). Zero infrastructure.

## Co-evolution

The measurement improves as the system improves:
- Early: human notices symptoms manually
- Later: system learns which symptom patterns the human flags
- Eventually: system predicts "the human would flag this" without asking
